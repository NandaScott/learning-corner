Content Security Policy В· Lesson 7

Strict CSP: nonces + 'strict-dynamic'

L6 showed the host-allowlist model is broken. This is the answer the same researchers proposed and Google now recommends: stop trusting origins and start trusting the specific scripts you emitted. A nonce marks your scripts, 'strict-dynamic' lets them pull in their dependencies, and the domain allowlist is thrown away, which is exactly why every L6 vector stops working.

The concept

A strict CSP inverts the question. Instead of "which origins may serve script?" (unanswerable safely, per L6), it asks "which script elements did this server put on this page?", and answers with a per-response nonce. The whole recommended policy is short:1

Content-Security-Policy:
  script-src 'nonce-{RANDOM}' 'strict-dynamic';
  object-src 'none';
  base-uri  'none';

Three directives. The nonce authorizes your scripts, 'strict-dynamic' lets them load more, and object-src/base-uri close the two flanks a script policy can't. No host allowlist anywhere, so there's nothing for a JSONP endpoint or open redirect to hide behind.

The steps

Build the policy piece by piece, then make it survive old browsers.

Step 1 вҖ” The nonce: trust what you emitted

Recapping L2: the server generates a fresh, unguessable value each response, puts it in the policy ('nonce-r4nd0m') and on every script it means to run (<script nonce="r4nd0m">). Injected markup can't guess this response's nonce, so injected <script>, inline or src, never runs. That's the foundation; the rest of the policy just extends this trust safely.

Step 2 вҖ” 'strict-dynamic': propagate trust, discard the allowlist

A nonce alone is painful: every third-party script your app loads (analytics, tag managers, widgets) would each need the nonce threaded through, and many load further scripts you never see. 'strict-dynamic' solves it with one rule: a script that was already trusted (via nonce or hash) may create more scripts, and those run too. The critical distinction:1

'strict-dynamic' draws the line exactly where the danger is: script a trusted script chose to add (safe) vs script that showed up in the parsed HTML (the injection). The key consequence: when 'strict-dynamic' is present, host-source, scheme-source, 'self' and 'unsafe-inline' in that directive are all ignored.2

Why this kills every L6 vector With the host allowlist ignored, there's no trusted domain to abuse. A JSONP endpoint on apis.example.com? Not allowlisted anymore: the injected <script src> to it is parser-inserted and unnonced, so it's blocked. An open redirect? Same. A hosted AngularJS gadget? The injected markup that would drive it has no nonce. Strict CSP doesn't patch the L6 bypasses: it removes the allowlist they all depended on.

Step 3 вҖ” The flanks: object-src 'none' and base-uri 'none'

'strict-dynamic' governs script, but two non-script holes (L4) would still sink you, and neither inherits from a script directive. object-src 'none' kills plugin execution (<object>/<embed>, legacy Flash-style vectors). base-uri 'none' forbids an injected <base> from rewriting where your relative script URLs resolve. These two lines are non-negotiable in a strict policy precisely because a great script-src doesn't cover them.1

Step 4 вҖ” Make it backwards-compatible

Not every browser speaks CSP3. The deployable form layers fallbacks so each engine takes the strictest rule it understands:1

script-src 'nonce-{RANDOM}' 'strict-dynamic' https: 'unsafe-inline';

Step 5 вҖ” The costs you take on

Strict CSP forbids two things teams lean on. Inline event handlers (<span onclick="вҖҰ">) and javascript: URLs are blocked: refactor them to addEventListener() inside a nonced script. And eval(): a strict policy blocks it; if you truly need it you add 'unsafe-eval' (weakening the policy). Better to refactor to JSON.parse(). Whatever DOM-XSS remains inside your trusted scripts is beyond script-src entirely: that's the residue Trusted Types (L4) targets.1

Why this matters

1. It's the model L1вҖ“L6 were building toward. The evaluation algorithm, the source grammar, the directive map, the level history, the allowlist failure, all of it converges here: trust specific emitted code, not origins. If you can derive this policy from L6's problem, you understand CSP rather than memorize it.

2. The programmatic-vs-parser-inserted line is the interview core. Anyone can recite "nonce plus strict-dynamic." The depth signal is why it's safe: it trusts scripts a trusted script deliberately created, and blocks scripts that arrived in the parsed HTML, which is the injection. And that 'strict-dynamic' ignores the host allowlist is the sentence that connects it back to L6.

3. Strict CSP is a mitigation ceiling. It neutralizes reflected and stored injection of script tags; it does not fix 'unsafe-eval' if you keep it, DOM-XSS inside trusted code (вҶ’ Trusted Types), or non-script issues. Naming those limits is the honest, senior answer.

Key line "Strict CSP stops trusting origins and trusts the scripts you emitted: a per-response nonce authorizes your <script>s, and 'strict-dynamic' lets an already-trusted script create more, while ignoring the host allowlist, 'self', and 'unsafe-inline'. It allows programmatically-created script and blocks parser-inserted script, the injection. Add object-src 'none' and base-uri 'none' for the non-script flanks, and layer https: 'unsafe-inline' as fallbacks for old browsers that ignore strict-dynamic."
The trap Shallow: "strict-dynamic lets trusted scripts load other scripts." Half-right, and it misses the security-defining half: it allows programmatically created scripts while blocking parser-inserted ones, and it makes the host allowlist inert. Someone who can't state what strict-dynamic blocks, or that it discards the allowlist, doesn't yet see why it's safe.

show answer

No. It's parser-inserted and carries no nonce, so it's blocked, exactly the injection 'strict-dynamic' is designed to stop.

в‘ў Why are object-src 'none' and base-uri 'none' in the strict policy if script-src is already tight?

show answer

Because neither is covered by a script directive and neither falls back to default-src (L3/L4). object-src 'none' kills plugin vectors; base-uri 'none' stops a <base> injection rewriting relative script URLs.

show answer
