Content Security Policy · Lesson 2

The source-list grammar

Lesson 1 ended at "the browser matches the request against the directive's source list." This lesson is that match, in full. A source list is a handful of expression kinds, each with its own matching rule — and the rules have sharp edges (wildcards that don't cover the apex, paths that stop mattering after a redirect, an 'unsafe-inline' that silently disables itself) that separate a deep answer from a shallow one.

The concept

Most directives take a source list as their value: a space-separated set of source expressions. A request is allowed if it matches any one expression (OR within the directive, as we saw). So the whole game is: what are the kinds of expression, and exactly what does each one match?1

script-src 'self' https://cdn.example.com 'nonce-Xk9f2' 'sha256-abc…'
           └──┬──┘ └────────┬──────────┘ └────┬────┘ └────┬────┘
          keyword       host-source        nonce      hash

Four different expression kinds in one directive. A script is allowed if it satisfies any of them. There are five kinds in total — keyword, scheme, host, nonce, hash.

The steps

Walk the five kinds, then the matching edge cases that live inside them.

KIND            EXAMPLE                     MATCHES
1. keyword      'self'  'unsafe-inline'     fixed tokens with special meaning
2. scheme       https:   data:             any URL with that scheme
3. host         *.cdn.example.com:443/js/   an origin (+ optional path)
4. nonce        'nonce-Xk9f2p…'             an element carrying nonce="Xk9f2p…"
5. hash         'sha256-abc123…'            an inline block whose digest equals this

Kinds 1–3 are about where a resource comes from. Kinds 4–5 are about which specific inline/marked element is trusted — the CSP2 additions that made inline code governable without 'unsafe-inline'.

Step 1 — Keyword sources

Fixed, quoted tokens with hard-coded meaning. The ones that matter:

  • 'self' — the page's own origin (scheme + host + port). Not subdomains, not "inline." CSP3 also lets 'self' match the https:/wss: variants of the origin even on an http page.1
  • 'none' — matches nothing. Must be the only token in the list; it is the empty list. Blocks the whole category.
  • 'unsafe-inline' — allows inline scripts/styles and on*= handlers. The hole CSP exists to close.
  • 'unsafe-eval' — allows eval/Function/string setTimeout.
  • 'strict-dynamic', 'unsafe-hashes', 'report-sample', 'wasm-unsafe-eval' — CSP3 additions (covered in L5/L7). The keyword set is open-ended; browsers ignore ones they don't know.

Step 2 — Scheme sources

A bare scheme plus colon — https:, data:, blob:, mediastream:, filesystem:. Matches any URL with that scheme, regardless of host. img-src https: = "images from any HTTPS origin." Powerful and blunt — data: in a script-src, for instance, re-opens injection via data: URLs.

Edge — the secure-upgrade rule (CSP2+) URL matching treats insecure schemes/ports as matching their secure variants. The spec: "the source expression http://example.com:80 will match both http://example.com:80 and https://example.com:443." A scheme-less host-source uses the page's scheme, with the same http→https / ws→wss upgrade allowance. So listing the http form doesn't exclude HTTPS — it includes it.

Step 3 — Host sources (where the edges cluster)

The general shape is [scheme://] host [:port] [path] — e.g. https://*.cdn.example.com:443/assets/. Each part has a rule:

WILDCARD HOST   *.example.com   matches  sub.example.com, a.b.example.com
                                does NOT  example.com        ← the apex is not covered
BARE  *          *              matches  any host on a network scheme (http/https/ws…)
                                does NOT  data:  blob:  filesystem:   ← must be listed explicitly
PORT             example.com     → the scheme's default port only (443 for https)
                 example.com:*   → any port
PATH             /assets/        trailing slash ⇒ PREFIX match  (/assets/app.js ✓, /assets ✗)
                 /assets/app.js  no trailing slash ⇒ EXACT match of that path
                 (query string and #fragment are ignored in path matching)

The two most-missed: *.example.com does not match the bare apex example.com (the *. prefix requires at least one extra label), and a bare * does not cover data:/blob:/filesystem: — those always need an explicit scheme-source.1

Edge — paths stop mattering after a redirect If a request is redirected, the browser matches against the request's original URL and does not enforce the path component against the redirect target — only scheme/host/port. This is deliberate: it stops an attacker from using redirects to probe a path-restricted allowlist (e.g. learning valid paths on an internal host) and stops path info leaking across the redirect. Consequence: a path-restricted allowlist entry like example.com/safe/ gives you no path guarantee once a redirect is in play.2

Step 4 — Nonce sources

'nonce-base64value' (CSP2). The server puts a fresh random value in the policy and as a nonce attribute on the element it wants to trust: <script nonce="Xk9f2p">. The element runs iff the two match. The value must be ≥128 bits, cryptographically random, and regenerated on every response — a static or guessable nonce is worthless, since an injected element could just carry it. Nonces are what let you trust your inline script without opening 'unsafe-inline' to all inline script.

Step 5 — Hash sources

'sha256-…' / 'sha384-…' / 'sha512-…' (CSP2): the base64 digest of an inline block's exact bytes. The block runs iff its computed digest is in the list — so one stray whitespace change breaks the match. Good for static inline snippets you can hash at build time (no per-request token needed). CSP3 extends hashes to match an external script whose integrity attribute lists that same hash.1

Edge — a nonce or hash silently disables 'unsafe-inline' If a directive's source list contains any nonce-source or hash-source, then 'unsafe-inline' in that same directive is ignored. This is a back-compat ramp: a CSP1 browser sees 'unsafe-inline' and runs your inline script; a CSP2+ browser sees the nonce/hash, ignores 'unsafe-inline', and enforces the stricter rule. Deliberately listing both is how you degrade gracefully — not a mistake.1

Why this matters

The grammar is where "I wrote a CSP" and "I wrote a CSP that does what I think" diverge.

1. The edges are where real policies leak. *.example.com that forgot the apex; a data: left in script-src; a path restriction that evaporates on redirect; the http form that quietly admits HTTPS. None of these throw an error — the policy just permits more than its author believed. Reading a source list means reading it against these rules, not at face value.

2. Nonces and hashes changed the model. Before CSP2, governing inline code meant 'unsafe-inline' — all-or-nothing. Nonces/hashes made it surgical: trust this specific block, deny the rest. That shift is the seed of strict CSP (L7), where a nonce plus 'strict-dynamic' replaces the host allowlist entirely.

3. "OR within a directive" cuts both ways. Every expression you add is another way in. A long allowlist isn't "more configured" — it's more surface. The security of a directive is the security of its weakest expression, because a match against any one is a pass.

Key line "A source list is five kinds of expression — keyword, scheme, host, nonce, hash — matched with OR. The traps live in the host rule: *.example.com excludes the apex, bare * excludes data:/blob:/filesystem:, the path is a prefix match only with a trailing slash and is dropped entirely after a redirect, and http: matches https: too. And any nonce or hash in a directive makes 'unsafe-inline' a no-op."
The trap Shallow: "script-src *.example.com locks scripts to example.com." Deep answer catches that it (a) doesn't include example.com itself, (b) admits every subdomain — including an attacker-controlled one if subdomains aren't tightly held, and (c) says nothing about inline script or eval. Naming the redirect-path-drop and the 'unsafe-inline'-ignored rules unprompted is the tell of someone who's actually read the matching algorithm.

Recall — answer out loud before peeking:

① Does img-src *.example.com allow an image from example.com (no subdomain)?

show answer

No. *.example.com matches subdomains only; the *. prefix requires at least one extra label. The apex needs its own entry (example.com).

② When does a source path do a prefix match vs an exact match?

show answer

Trailing slash ⇒ prefix (/assets/ matches /assets/app.js); no trailing slash ⇒ exact (/app.js matches only that path). Query and fragment are ignored either way — and the path is dropped after a redirect.

③ A directive is script-src 'unsafe-inline' 'nonce-abc123'. Does an un-nonced inline script run?

show answer

No. The presence of a nonce (or hash) makes 'unsafe-inline' ignored in CSP2+ browsers. Only the element carrying nonce="abc123" runs. (A CSP1 browser would honour 'unsafe-inline' — that's the graceful-degradation intent.)

④ Does default-src * allow a data: URL script?

show answer

No. Bare * matches network-scheme hosts but excludes data:, blob:, and filesystem: — those must be listed as explicit scheme-sources.
