Content Security Policy · Lesson 1

How the browser evaluates a policy

Before any directive catalogue or the CSP1/2/3 history means anything, you have to see what the browser does with a policy. The browser re-runs a small algorithm on every resource the page tries to load. Once you can run that algorithm in your head, the rest of CSP is filling in a table.

The concept

A Content Security Policy is a set of rules a server ships to a browser, and the browser enforces them on the client, per fetch. That last part is the mental model. When a page tries to load a script, an image, a font, a frame, or to run inline code, or submit a form, the browser pauses, finds the one directive that governs that kind of request, checks the request against that directive's allowlist, and then either lets it through or blocks it and files a report.1

The rules live in a response header:

Content-Security-Policy: default-src 'self'; script-src 'self' https://cdn.example.com; img-src *

One header, semicolon-separated directives. Each directive is a name (script-src) plus a value, usually a source list ('self' https://cdn.example.com). The browser parses this once, then consults it on every fetch.

Two consequences of "the browser enforces it" frame everything that follows. First, CSP is defense-in-depth, not a server guarantee: a client that ignores the header (an old browser, a non-browser HTTP client, a hostile user) is unaffected; CSP shrinks what a compromised page can do to a normal victim's browser, it doesn't protect the server. Second, because it's enforced per fetch, the interesting question is never "does the page have a CSP" but "for this request, which directive wins, and does the request match it?"

Steps 3–4 pick which rule applies. Step 5 applies it. Every subresource, every inline block, every form submission runs this same loop.

Step 1 — Parse the header into directives

When the response arrives, the browser splits the header value on ; into directives, and each directive into a name and a token list. Unknown directive names are ignored (forward-compatibility: a CSP3 directive does no harm in a browser that only knows CSP2). A response may carry more than one policy: several Content-Security-Policy headers, or a header plus a <meta> tag. Hold that thought; it matters at the synthesis.

Step 2 — A request (or behaviour) is about to happen

CSP doesn't only gate network fetches. It gates a category of things: loading a subresource (script, style, image, font, frame, connection…), and executing certain behaviours that don't hit the network at all: running an inline <script>, an onclick= handler, a javascript: URL, or a call to eval(). Each of these maps to a request "type" that the next step keys off. CSP3 defines this by hooking into the browser's Fetch machinery, which is why the modern spec is written "in terms of Fetch."1

Step 3 — Find the one governing directive

People skip this step. The browser does not check the request against the whole policy. It resolves the single directive that governs this request type. A <script> element looks for script-src-elem first; a stylesheet looks for style-src-elem; an image looks for img-src; a framed page's parent is checked against frame-ancestors; and so on. Exactly one directive wins per request.

Flag — one directive, not the whole policy A request that's allowed by img-src * is not also re-checked against script-src. Each request type has its own governing directive; the others are irrelevant to it. "Is this blocked?" always reduces to a single directive's source list.

Step 4 — The fallback chain (and why absence can mean allow)

If the specific directive isn't present in the policy, the browser walks a fallback chain. For a <script> element that's script-src-elem → script-src → default-src. Whichever it finds first governs; default-src is the fallback of last resort.

  <script src>      →  script-src-elem  ?  →  script-src  ?  →  default-src  ?  →  (nothing)
                                                                                    │
  found any of these?  ── yes ──▶  use it as the governing directive                 ▼
                       ── no  ──────────────────────────────────────────────▶  ALLOW (no rule covers this)

If none of the chain is present, the request is allowed rather than blocked. CSP restricts only what a directive actually covers. A policy of just script-src 'self' says nothing about images: images load from anywhere. default-src is how you make the policy default-deny across types you didn't name explicitly.

Step 5 — Match the request against the source list

Now the browser has one directive and its source list, say script-src 'self' https://cdn.example.com. It compares the request against each source expression, and the request is allowed if it matches any of them (logical OR within a directive). Our https://cdn.example.com/app.js matches the host expression https://cdn.example.com → allowed. The exact grammar of those expressions ('self', schemes, hosts, wildcards, nonces, hashes) is Lesson 2. For now: OR across the list, and an empty list (or 'none') matches nothing, so it blocks everything of that type.

Step 6 — Allow, or block and report

Match → the fetch proceeds or the code runs. No match → the browser blocks it (the script never executes, the image never loads) and, in the same breath, generates a violation report: it fires a SecurityPolicyViolationEvent in the page, and if the policy named a reporting endpoint (report-uri or report-to), it POSTs a JSON report there.2 Reporting is how you operate CSP without breaking your own site (Lesson 8).

Why this matters

Three things fall straight out of "the browser runs this loop per fetch," and they set up the rest of the course.

1. A CSP is only as complete as its directives. Because a missing governing directive with no default-src means allow, the dangerous policy is the partial one that looks protective while permitting categories of request it never named. "Does it have a default-src, and which types override it?" is the first question to ask of any policy.

2. Multiple policies intersect. They never loosen. If a response carries two policies, the browser enforces all of them, and a request must satisfy each. One policy saying script-src 'self' and another saying script-src https://cdn.example.com means no script loads: neither origin satisfies both. Adding a policy can only ever restrict further. That's why you can safely layer a tight Report-Only policy over a loose enforced one.

3. It's client-side, so it's a mitigation, not a wall. CSP reduces the blast radius of an injection that already got through: it stops injected script from running or phoning home. It is the second line, behind output encoding and input handling, not a replacement for them. Framing CSP as "XSS protection" without "defense-in-depth" is the shallow tell.

Key line "CSP is a browser-enforced, per-fetch allowlist. For each request the browser finds the one governing directive, walking the fallback chain to default-src, and allows the request if it matches any expression in that directive's source list, otherwise blocks it and reports. Crucially, if no directive covers a request and there's no default-src, it's allowed."
The trap The shallow answer is "CSP whitelists which domains a page can load from." That misses two things a deep answer nails: (a) it's resolved per request type via a fallback chain, so absence of a rule can mean allow, not deny; and (b) it governs behaviours too (inline script, eval), not just network origins. Someone who says "just set script-src and you're safe" hasn't met the partial-policy hole.

Recall — answer out loud before peeking:

① A page loads a font and there is no font-src and no default-src in the policy. Blocked or allowed?

show answer

Allowed. With no governing directive and no default-src fallback, nothing restricts the request. CSP only restricts what a directive actually covers: this is why default-src matters.

② For a <script src>, what's the fallback chain the browser walks?

show answer

script-src-elem → script-src → default-src. First one present wins; if none are present, the fetch is allowed.

③ A request must satisfy a source list how, all expressions, or any one?

show answer

Any one (OR within a directive). But across multiple policies it must satisfy each policy: policies intersect and only ever restrict further.

④ Why is CSP called defense-in-depth rather than an XSS fix?

show answer

It's enforced by the browser on the client. It shrinks what an already-injected page can do (run inline script, exfiltrate) for a compliant browser; it doesn't stop the injection or protect non-compliant clients. Encoding/validation is still the first line.
