Content Security Policy · Lesson 3

Fetch directives & the fallback cascade

L1 said the browser resolves "the one governing directive" by walking a fallback chain. L2 was the value side of a directive. This lesson is the name side: the fetch directives — the largest family — and the exact cascades they fall through. The payoff is being able to answer, for any resource type, "which directive actually decides this?" — and to see why default-src is a weaker backstop than it looks.

The concept

Fetch directives are the "where may this kind of resource load from" rules. They're the bulk of CSP, and they share one behaviour: when the specific directive is absent, the browser walks a fallback list until it finds one that's present — and if it reaches the end with nothing, the fetch is allowed (L1's rule). default-src sits at the bottom of most of those lists. That's its entire job: it is the fallback for fetch directives — not a global "deny everything" switch.1

GROUP                DIRECTIVES                              falls back to
scripts              script-src-elem, script-src-attr   →   script-src   →  default-src
styles               style-src-elem, style-src-attr     →   style-src    →  default-src
simple loaders       img-src, font-src, media-src,      →                   default-src
                     connect-src, object-src, manifest-src
nested contexts      frame-src   →  child-src           →                   default-src
                     worker-src  →  child-src  →  script-src  →             default-src

Every fetch directive terminates at default-src, but the paths differ. The nested-context group has intermediate hops — that's where people mis-predict which rule wins.

The steps

Three cascades to hold: the script/style elem/attr split, the simple loaders, and the nested-context chains.

Step 1 — The simple loaders fall straight to default-src

The easy tier. img-src, font-src, media-src (<audio>/<video>/<track>), connect-src (fetch/XHR/WebSocket/sendBeacon/EventSource), object-src (<object>/<embed>), and manifest-src each govern one resource type and, if absent, fall directly back to default-src. No intermediate hop. If there's no default-src either, that type is unrestricted.

Step 2 — The script/style elem vs attr split (CSP3)

CSP3 split scripts into two governing directives so you can rule on elements and attributes separately:2

  • script-src-elem — <script> elements (inline blocks and src= loads).
  • script-src-attr — inline event-handler attributes (onclick=, onload=, …).

Both fall back to script-src, then default-src. Style mirrors this exactly: style-src-elem (<style> / <link rel=stylesheet>) and style-src-attr (inline style=), both backstopped by style-src.

  <script src> or inline <script>   →  script-src-elem  →  script-src  →  default-src
  onclick="…"  (event handler attr)  →  script-src-attr  →  script-src  →  default-src

  set only script-src  ⇒  it backstops BOTH  (one rule for elements and handlers)
  set script-src-attr 'none' + script-src 'self'
                        ⇒  external/inline scripts allowed, ALL inline handlers blocked

The split's value: block DOM event-handler attributes (a classic injection sink) while still permitting normal <script> loads — impossible with the single script-src.

Step 3 — The nested-context cascade (the confusing one)

Framing and workers are the tier people get wrong, because child-src sits in the middle and the two consumers diverge:1

  <iframe src>              →  frame-src   →  child-src  →  default-src
  new Worker(…)/SharedWorker →  worker-src  →  child-src  →  script-src  →  default-src
                                                             └── note the extra hop ──┘

frame-src and worker-src both prefer child-src when they're absent — but if child-src is also absent, they part ways: framing drops to default-src, whereas a worker drops to script-src first (a worker is executable script), and only then default-src.

History note (full story in L5): CSP1 had only frame-src; CSP2 deprecated it in favour of child-src (which also governed workers); CSP3 un-deprecated frame-src and split workers back out into worker-src. So child-src is now a middle-tier fallback that rarely needs setting directly.

Step 4 — What default-src actually backstops (and doesn't)

Because every fetch directive terminates at default-src, it's tempting to read default-src 'none' as "nothing can happen." It isn't. default-src is the fallback for fetch directives only. A whole set of directives never consult it — they're simply off if you don't set them:

Flag — default-src is not a global deny default-src 'none' does not restrict base-uri, form-action, frame-ancestors, sandbox, or report-to/report-uri. So a page with only default-src 'none' can still be framed by anyone (no frame-ancestors), can still have its <base> hijacked (no base-uri), and can still post a form anywhere (no form-action). Those are Lesson 4's directives — and they're exactly why a "locked down" default-src 'none' policy often isn't.

Why this matters

1. "Which directive decides this?" is now mechanical. Given any resource, you name its specific directive and walk the chain to the first one present. That's the L1 algorithm made concrete across the whole fetch family — and it's the exact reasoning an interviewer probes with "if I only set script-src, are my inline onclicks covered?" (Yes — script-src-attr falls back to it.)

2. The intermediate hops are the trap. Almost everyone knows "falls back to default-src." The depth signal is knowing the middle: worker-src's extra script-src hop, frame-src/worker-src both preferring child-src, the elem/attr split resolving through script-src. Get the middle wrong and you'll mis-predict which rule a real policy applies.

3. Fetch directives are only half of CSP. default-src backstops fetches and nothing else. The directives that stop framing, form exfiltration, and <base> hijacking stand alone and must be set explicitly — which is why the next lesson exists.

Key line "Fetch directives all terminate at default-src, but the paths differ. Scripts and styles resolve through an elem/attr split into script-src/style-src first; simple loaders like img-src go straight to default-src; framing goes frame-src → child-src → default-src, and workers take an extra hop: worker-src → child-src → script-src → default-src. And default-src only backstops fetch directives — not frame-ancestors, base-uri, or form-action."
The trap Shallow: "default-src 'none' locks the page down." It locks down fetches. It does nothing about who can frame the page, where forms can submit, or a <base> tag rewriting relative script URLs. Someone who says "just set default-src 'none' and add back what you need" has covered subresources and missed the entire navigation/document family.

Recall — answer out loud before peeking:

① You set only script-src 'self'. Is an inline onclick= handler allowed?

show answer

No — it's governed, and blocked. script-src-attr is absent, so it falls back to script-src 'self'. An inline attribute isn't same-origin script, so it fails to match and is blocked. (Add 'unsafe-hashes' or restructure to allow it.)

② A worker is created and there's no worker-src and no child-src. Which directive decides?

show answer

script-src — then default-src if that's also absent. Full chain: worker-src → child-src → script-src → default-src. The extra script-src hop is the worker-specific quirk.

③ Does default-src 'none' stop the page from being framed by evil.com?

show answer

No. Framing is governed by frame-ancestors, which does not fall back to default-src. Without an explicit frame-ancestors, anyone can frame the page.

④ Which directive governs a <link rel="stylesheet">, and what's its fallback chain?

show answer

style-src-elem → style-src → default-src. (Inline style= attributes go through style-src-attr instead.)

Primary source — read this one W3C CSP Level 3 — §6.1 Fetch Directives (each directive lists its own fallback), and MDN's fetch-directive table for the fallback chains at a glance. The worker/frame chains are called out verbatim in CSP3 §1.3 "Changes from Level 2."
