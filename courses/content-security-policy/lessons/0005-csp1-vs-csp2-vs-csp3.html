Content Security Policy · Lesson 5

CSP1 vs CSP2 vs CSP3

You now have the whole directive map. This lesson is the story that produced it: three W3C levels, each born because the previous one didn't stop XSS well enough. Read as history it's trivia; read as an argument it's the setup for the rest of the course — the levels trace a single arc, from "allowlist the hosts" toward "drop the host allowlist entirely."

The concept

"CSP" isn't one spec. It's three levels, each a near-superset of the last, published years apart. Knowing which directive arrived in which level is the surface answer to "CSP1 vs CSP2 vs CSP3." The deeper answer is why each level happened — and that why is the same each time: the previous level left a way for injected script to run.1

Origin: proposed as "Content Restrictions" (Hansen, 2004), shipped experimentally behind vendor prefixes in 2011, standardized as 1.0 in 2012. Note CSP 1.0 never became a full Recommendation — it stalled at Candidate Recommendation and CSP2 was the first to reach REC.2

The steps

Walk the three levels: what each shipped, and the hole that motivated the next.

Step 1 — CSP1 (1.0, 2012): allowlist the sources

The founding idea: a page declares, per resource type, which origins it trusts. CSP1 shipped the core fetch directives — default-src, script-src, style-src, img-src, connect-src, font-src, media-src, object-src, frame-src — plus sandbox and report-uri, and the keywords 'self', 'none', 'unsafe-inline', 'unsafe-eval'.3

The hole CSP1 left Inline script was all-or-nothing. To run your own inline <script> you set 'unsafe-inline' — which also permitted injected inline script. So any page with inline code (nearly all of them) had to open the exact door CSP was built to shut. There was also no framing-protection directive — X-Frame-Options lived outside CSP.

Step 2 — CSP2 (2016): trust specific inline code

CSP2 — the first full W3C Recommendation (15 Dec 2016) — answered the inline problem with nonces and hashes: mark this inline block as trusted without trusting inline in general. It also folded in the neighbouring gaps: frame-ancestors (absorbing X-Frame-Options), form-action, base-uri, child-src (deprecating frame-src), and plugin-types. Reports got richer (effective-directive, source file, line/column), and URL matching gained the secure-upgrade and redirect-path rules from L2.1

The hole CSP2 left Nonces/hashes fixed inline, but most policies still leaned on host allowlists for external script (script-src 'self' https://cdn… https://analytics…). Google's research then showed those allowlists were bypassable in the overwhelming majority of real sites — via JSONP endpoints, open redirects, and hosted libraries on allowlisted domains (L6). The allowlist model itself was the weakness.

Step 3 — CSP3 (Working Draft): drop the host allowlist

CSP3 (still a Working Draft, but widely shipped) is a rewrite rebased on the Fetch standard, and its headline is 'strict-dynamic': trust propagates from a nonce/hash'd script to the scripts it loads, and host/scheme entries in that directive are ignored. That lets a policy stop depending on a host allowlist at all (L7). CSP3 also added 'unsafe-hashes', 'report-sample', report-to (Reporting API) superseding report-uri, worker-src and manifest-src, the script-src-elem/-attr and style-src-elem/-attr split, and hashes that can match external scripts via integrity. It un-deprecated frame-src and deprecated plugin-types and block-all-mixed-content.1

History detail — things that came and went "History of the changes" isn't only additions. CSP3 drafts introduced navigate-to (restrict where the page could navigate) and prefetch-src, then removed both over compatibility and privacy concerns. Directive names in a Working Draft are not load-bearing until they ship broadly — a reason to check current browser support, not just the spec.

Why this matters

1. The arc is the whole point. CSP1 = allowlist origins. CSP2 = plus trust specific inline blocks (nonces/hashes). CSP3 = propagate that trust and abandon the host allowlist ('strict-dynamic'). Each level is a response to the previous one failing to stop injected script. Tell that story and the directive list stops being a list.

2. "Which level" has real deployment stakes. Nonces need CSP2+; 'strict-dynamic' and report-to need CSP3-era browsers. This is exactly why real policies layer old and new — 'unsafe-inline' beside a nonce (L2), report-uri beside report-to (L8), X-Frame-Options beside frame-ancestors (L4). Older engines take the fallback; newer ones enforce the stricter rule.

3. The spec status is a caveat, not a footnote. CSP3 is a Working Draft whose contents have shifted (navigate-to, prefetch-src came and went). Teach and deploy against current browser support, not just the draft — the healthy instinct is "verify it ships," which is the sourcing discipline this whole course runs on.

Key line "Three levels, one arc. CSP1 (2012) allowlisted origins but forced 'unsafe-inline' for any inline script. CSP2 (2016, the first W3C REC) added nonces and hashes to trust specific inline blocks, plus frame-ancestors/form-action/base-uri. CSP3 added 'strict-dynamic' — propagate trust from a nonced script and ignore the host allowlist — because Google showed host allowlists were almost always bypassable. Each level exists because the last one still let injected script run."
The trap Shallow: "CSP2 added nonces, CSP3 added strict-dynamic" — a memorized list. Deep: naming the motivation — nonces because 'unsafe-inline' was all-or-nothing; strict-dynamic because host allowlists were empirically bypassable — and knowing CSP1 never reached full Recommendation while CSP2 did. The list is fluency; the causal chain is understanding.

Recall — answer out loud before peeking:

① Which level introduced nonces and hashes, and what problem did they solve?

show answer

CSP2. They solved inline being all-or-nothing: before, running your own inline script meant 'unsafe-inline', which also allowed injected inline script. Nonces/hashes trust specific blocks without trusting inline in general.

② What is 'strict-dynamic' (CSP3) a response to?

show answer

The empirical failure of host allowlists: research showed most real allowlists were bypassable (JSONP, open redirects, hosted libs). 'strict-dynamic' lets a policy propagate trust from a nonced/hashed script and ignore the host allowlist entirely.

③ Which was the first CSP level to reach full W3C Recommendation?

show answer

CSP Level 2 (15 Dec 2016). CSP 1.0 stalled at Candidate Recommendation (2012) and was superseded; CSP3 remains a Working Draft.

④ Name one directive that was proposed in a CSP3 draft and later removed.

show answer

navigate-to (and prefetch-src) — both introduced in CSP3 drafts, then removed. A reminder that Working-Draft names aren't stable until broadly shipped.
