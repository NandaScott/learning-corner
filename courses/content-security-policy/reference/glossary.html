Glossary

Canonical vocabulary for this course. Every lesson uses these words to mean exactly this. Verified against the W3C CSP specs (Level 1 / 2 / 3) and MDN — not folklore.

The delivery

Content Security Policy (CSP)
A set of rules, delivered by a server to a browser, that constrains which resources a page may load and which behaviours it may perform — enforced by the browser, on the client. Its primary job is to blunt cross-site scripting (XSS): even if an attacker injects markup, the policy can stop the injected script from executing or exfiltrating.
Policy
The value of one Content-Security-Policy header: a semicolon-separated list of directives. A response may carry several policies (multiple headers or a header plus a <meta>); all of them are enforced, and each can only further restrict — policies never loosen one another.
Directive
One rule inside a policy: a name plus a value, e.g. script-src 'self' https://cdn.example.com. The name selects what is governed (scripts, images, framing…); the value is usually a source list.
Header vs <meta>
CSP is normally the Content-Security-Policy HTTP response header. It can also ship in a <meta http-equiv="Content-Security-Policy"> tag, but the meta form can't express some directives (e.g. frame-ancestors, report-uri/report-to, sandbox) and only applies to content after the tag — the header is authoritative.

The evaluation model

Governing directive
For a given resource fetch or behaviour, the single directive the browser consults. Found by resource type (a <script> → script-src-elem → script-src → default-src). If no matching directive exists and there's no default-src, the fetch is allowed — CSP only restricts what a directive actually covers.
Fallback chain
The ordered lookup a directive follows when absent. Most fetch directives fall back to default-src. Some have longer chains, e.g. script-src-elem → script-src → default-src, and worker-src → child-src → script-src → default-src. default-src is the fallback of last resort, not a global rule.
Source list
A directive's value: a space-separated set of source expressions. A fetch is allowed if it matches any expression in the list. An empty list (or the token 'none') matches nothing → blocks everything of that type.
Source expression
One entry in a source list. Kinds: keyword-source ('self', 'none', 'unsafe-inline', 'unsafe-eval', 'strict-dynamic', 'unsafe-hashes'), scheme-source (https:, data:), host-source (cdn.example.com, *.example.com, with optional scheme/port/path), nonce-source ('nonce-…'), and hash-source ('sha256-…').

The trust tokens

'self'
Matches the page's own origin (scheme + host + port). Not a wildcard for subdomains and not "inline" — it's the current origin only.
'unsafe-inline'
Allows inline scripts/styles (inline <script>, on*= handlers, style=). The dangerous one: it re-opens the exact hole CSP exists to close. Present in a script-src, it is ignored if a nonce or hash is also present (CSP2+ back-compat rule).
'unsafe-eval'
Allows string-to-code APIs (eval, Function, setTimeout("…")). Independent of 'unsafe-inline'.
Nonce
A per-response random token (≥128 bits, base64) placed both in the policy ('nonce-r4nd0m') and on the element (<script nonce="r4nd0m">). The element runs iff the values match. Introduced in CSP2. Must be unguessable and fresh every response.
Hash
A base64 digest ('sha256-…'/'sha384-…'/'sha512-…') of an inline block's exact contents. The block runs iff its computed digest is in the list. Introduced in CSP2; CSP3 lets a hash also match an external script that carries matching integrity metadata.
'strict-dynamic'
(CSP3) In a script directive, propagates trust: a script already trusted (via nonce/hash) may create further scripts programmatically, and those run too — while host/scheme allowlist entries in the same directive are ignored. The core of a modern strict CSP; makes the policy independent of a domain allowlist.
'unsafe-hashes'
(CSP3) Lets a hash match inline event handlers and style attributes (and javascript: targets), which plain hashes don't cover. Narrower and safer than 'unsafe-inline', but still a loosening.

Enforcement & reporting

Enforce vs Report-Only
The Content-Security-Policy header blocks violations. Content-Security-Policy-Report-Only blocks nothing — it only reports what would have been blocked. The standard rollout is to run Report-Only first, watch the reports, then flip to enforce.
Violation report
A JSON document the browser sends when a directive is violated (and also a SecurityPolicyViolationEvent fired in-page). Names the effective-directive, the blocked-uri, the document, and (CSP2+) source file / line / column.
report-uri vs report-to
report-uri <url> (CSP1) posts reports to a URL — now deprecated. report-to <group> (CSP3) names a group defined via the Reporting API (Reporting-Endpoints header). Because browser support lagged, real deployments often send both.

The levels

CSP1 (1.0, 2012)
The first W3C standardized level. Core fetch directives + sandbox + report-uri, and the 'unsafe-inline'/'unsafe-eval' keywords. No nonces, no hashes, no frame-ancestors.
CSP2 (2016, W3C Recommendation)
Added nonces and hashes, child-src, frame-ancestors, form-action, base-uri, plugin-types, and richer violation reports.
CSP3 (Working Draft)
Rebased the spec on the Fetch standard. Added 'strict-dynamic', 'unsafe-hashes', report-to, worker-src, manifest-src, the script-src-elem/script-src-attr and style-src-elem/style-src-attr split; hashes for external scripts; deprecated plugin-types/block-all-mixed-content.
