Fallback-chain cheat sheet

Every fetch directive, what it governs, and the exact chain the browser walks when it's absent. Plus the directives that never fall back — the ones default-src can't save you on. Designed to print on one page. Verified against W3C CSP3 §6.1 + §1.3.

Fetch directives → fallback chain

When a directive is absent, the browser walks its chain left-to-right and uses the first one present. Reach the end with nothing present ⇒ the fetch is allowed.

DirectiveGovernsFallback chain (when absent)
default-srcthe fallback for all fetch directives below— (bottom of every chain)
Scripts & styles — elem/attr split (CSP3)
script-src-elem<script> — inline & src=script-src → default-src
script-src-attrinline event handlers (onclick=…)script-src → default-src
script-srcall script (backstops both above)default-src
style-src-elem<style>, <link rel=stylesheet>style-src → default-src
style-src-attrinline style= attributesstyle-src → default-src
style-srcall styles (backstops both above)default-src
Simple loaders — straight to default-src
img-srcimages, faviconsdefault-src
font-src@font-face fontsdefault-src
media-src<audio>, <video>, <track>default-src
connect-srcfetch, XHR, WebSocket, sendBeacon, EventSourcedefault-src
object-src<object>, <embed> (plugins)default-src
manifest-srcweb app manifestdefault-src
Nested contexts — intermediate hops (the trap tier)
frame-src<frame>, <iframe>child-src → default-src
worker-srcWorker, SharedWorker, ServiceWorkerchild-src → script-src → default-src
child-srcmiddle-tier fallback for the two abovedefault-src
The one people miss frame-src and worker-src both prefer child-src — but if child-src is also absent they diverge: framing drops to default-src, a worker takes an extra script-src hop first (a worker is executable script). Full worker chain: worker-src → child-src → script-src → default-src.

Directives that never fall back to default-src

These are simply off if you don't set them. Setting default-src — even 'none' — does nothing for them.

DirectiveFamilyWhat's unprotected if you omit it
base-uridocument<base> tag can rewrite relative URLs (e.g. hijack relative script paths)
sandboxdocumentno extra sandboxing applied to the resource
form-actionnavigationforms can submit (exfiltrate) to any origin
frame-ancestorsnavigationany site can frame the page (clickjacking) — supersedes X-Frame-Options
report-to / report-urireportingno violation reports sent (report-uri deprecated)
upgrade-insecure-requestsrider*HTTP subresource URLs not auto-upgraded to HTTPS
trusted-types / require-trusted-types-forrider*no Trusted Types enforcement at DOM injection sinks

* "rider" = defined in a separate spec (Upgrade Insecure Requests; Trusted Types) but delivered on the CSP header. block-all-mixed-content is deprecated — upgrade-insecure-requests covers it.

Reading a policy fast

  1. Is there a default-src? No ⇒ every fetch type you didn't name is allowed. This is the #1 hole.
  2. Is object-src 'none' set? Plugins are a classic bypass; strict CSP always pins it (doesn't rely on default-src).
  3. Is base-uri set? Without it, a <base> injection can redirect relative script loads — default-src won't stop it.
  4. Is frame-ancestors set? No ⇒ clickjackable, regardless of default-src 'none'.
  5. For each script rule, walk elem/attr → script-src → default-src to find what actually governs inline vs external vs handlers.
