Celery + Redis · Lesson 1

A task is a message in a Redis list

Before acks, prefetch, or visibility timeouts make any sense, you have to see what physically lands in Redis when you call .delay(). So we open the box first — every later lesson sits on top of this one fact.

The concept

When you write add.delay(2, 3), it's tempting to picture Celery "sending the task to the worker." Nothing of the sort happens. The web process and the worker never talk to each other. They talk to Redis, and the only thing that moves between them is a string.

The fact underneath the whole course:

Redis is not a queue. It's a key-value store with a few data structures, and a Celery "queue" is literally a Redis LIST. Calling .delay() serializes your task request into a JSON message and LPUSHes that string onto a list. A worker BRPOPs the string back off. That's the entire transport.1

RabbitMQ is a real message broker with native queues and acknowledgements built in. Redis isn't — so kombu (Celery's messaging layer) has to build queue behaviour out of plain Redis commands. This lesson is the foundation it builds on; Lessons 3–4 are where the clever emulation lives.

The steps

Here's the full path of one .delay() call, from your view to the worker's hands. We'll walk each step underneath.

1. .delay(2, 3)        in the web process    build a task-request message
2. serialize           message → JSON         body base64-encoded, headers + properties wrap it
3. pick the list       routing_key "celery"   → Redis key "celery"
4. LPUSH               web proc → Redis        prepend the JSON string onto list "celery"
   ───────────────  ( the web process is now done — returns an AsyncResult ) ───────────────
5. BRPOP               worker  ← Redis         blocking pop from the right of "celery"
6. deserialize         JSON → task name+args   look up the registered task, decode the body
7. run the function    add(2, 3) → 5           execute in a worker child process

Steps 1–4 happen in your web request. Steps 5–7 happen later, in the worker. Redis is the only thing they share.

Step 1 — .delay() builds a request, not a result

add.delay(2, 3) is sugar for add.apply_async(args=(2, 3)). It does not call add. It packages a description — "run the task named myapp.add with args (2, 3)" — and hands you back an AsyncResult (a promise keyed by a task id) almost instantly. The actual addition happens elsewhere, later.

Step 2 — Serialize into a JSON envelope

The request becomes a self-contained message. Redis stores bytes, so everything the worker needs to reconstruct the call has to be inside that message. Its shape (Celery protocol v2):2

{
  "headers": {
    "task": "myapp.add",            ← which function to run
    "id":   "f3c8…-…-…",            ← task id (your AsyncResult tracks this)
    "argsrepr": "(2, 3)", …          ← metadata
  },
  "properties": {
    "delivery_tag": "a91f…",        ← used later for ack tracking (L3)
    "delivery_info": { "exchange": "", "routing_key": "celery" },
    "body_encoding": "base64", …
  },
  "body": "WzIsIDNd…"                ← base64 of the args/kwargs: [[2,3],{},…]
}

The task name travels in headers, the arguments in the base64 body. The function's code is never sent — only its registered name. The worker must already have that task defined.

Step 3 — The routing key picks a list

Celery has an AMQP-flavoured vocabulary — exchanges, routing keys, queues — inherited from RabbitMQ. On Redis most of that machinery collapses: the routing key becomes the Redis list name. Send to the default queue and the routing key is celery, so the target key is the list celery. Route to a queue named emails and it's the list emails. One queue = one list.

Flag — abstraction ≠ implementation Exchanges and bindings are real on RabbitMQ. On the Redis transport they're mostly a thin compatibility shim — direct routing maps straight to a list key. Don't reason about Redis as if it ran AMQP; it's emulating just enough of it.

Step 4 — LPUSH: the task is now a string in Redis

The web process runs one Redis command and is done:

  web process                         Redis
      │   LPUSH celery "{…json…}"        ┌──────────────────────────────┐
      │ ───────────────────────────────▶│ key "celery"  (a LIST)        │
      │                                  │  ┌────┬────┬────┐             │
      │ ◀─── returns AsyncResult ───     │  │ m3 │ m2 │ m1 │ ← older     │
      ▼                                  │  └────┴────┴────┘             │
  (request ends, task not yet run)       └──────────────────────────────┘
                                          new messages pushed on the LEFT

The whole "send" is a single LPUSH. If no worker is running, the message just sits in the list — durable as long as Redis keeps it. Nothing is lost simply because no worker was listening.

Step 5 — BRPOP: the worker pulls from the other end

A worker blocks on BRPOP celery — "pop the rightmost element, and if the list is empty, wait until something arrives (up to a timeout) rather than busy-polling."1 Push on the left, pop on the right ⇒ oldest message comes out first — FIFO.

      LPUSH ──▶  [ m3 ][ m2 ][ m1 ]  ──▶ BRPOP
                   ▲                  ▲
                 newest            oldest comes out first  (FIFO)

Blocking is the point: BRPOP lets an idle worker sleep on the socket instead of hammering Redis with "anything yet?" — near-zero latency when work arrives, near-zero load when it doesn't.

Steps 6–7 — Deserialize and run

The worker parses the JSON, reads headers.task (myapp.add), looks that name up in its own task registry, base64-decodes the body to recover (2, 3), and finally calls the real function in one of its child processes. Only now — well after the web request already returned — does 2 + 3 actually run.

Why this matters

Three consequences fall straight out of "a task is just a string in a list," and they set up the rest of the course.

1. The broker is dumb storage. Redis isn't tracking tasks, retries, or who's working on what. It's holding a list of strings. Every smart behaviour you associate with Celery — acknowledgement, redelivery, prefetch, priorities — is emulated by kombu using extra Redis keys and commands. Knowing the base case is a plain list is what makes the emulation (Lessons 3–4) legible instead of magic.

2. Producer and consumer are fully decoupled. The web process finishes at step 4 regardless of whether any worker exists. That's the entire value proposition — offload slow work, return fast — and also the first failure mode: a message sitting in a list with no worker draining it is invisible until you look at the list length.

3. The message is self-contained, and that has teeth. The worker needs the task defined under the same name, and it decodes whatever args were frozen in at publish time. Deploy a worker that doesn't know myapp.add, or change a task's signature while old messages are still in the list, and the worker chokes on a string it can't honour.

Interview line "A Celery queue on Redis is literally a Redis list. .delay() serializes the task into a JSON message and LPUSHes it; the worker BRPOPs it off the other end — push-left, pop-right, so it's FIFO. Redis is dumb storage; everything else Celery does is emulated on top of that list."
The trap The shallow answer is "Celery sends the task to the worker." It doesn't — producer and worker never communicate; they only share a Redis list, and the only thing that crosses is a serialized message, not a function or an object. Naming that the message carries the task's name (not its code) and its arguments (frozen at publish time) is the tell of someone who's actually looked.

Recall — answer out loud before peeking:

① Physically, what is a Celery "queue" on the Redis broker?

show answer

A Redis LIST, keyed by the queue name (default celery). Not a special queue object — a list.

② Which two Redis commands move a task from producer to worker, and which way does each go?

show answer

LPUSH (producer prepends on the left) and BRPOP (worker blocking-pops from the right). Left-in, right-out ⇒ FIFO.

③ What's inside the message — and what is not?

show answer

Inside: the task name (headers), the arguments (base64 body), and routing/tracking metadata (properties). Not inside: the function's code — only its registered name. The worker must already define that task.

④ Why does it matter that "Redis is not a queue"?

show answer

Because every queue-like guarantee — acks, redelivery, prefetch, priority — is emulated by kombu on top of plain Redis structures. The broker itself just stores strings; the cleverness is in the client.
