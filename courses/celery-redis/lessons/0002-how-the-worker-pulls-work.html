Celery + Redis · Lesson 2

How the worker gets the task — and where it goes next

Lesson 1 left a JSON string sitting in a Redis list. This lesson follows it the rest of the way: off the broker, through the worker's main process, and into the child that actually runs your code. The higher-level map of who hands what to whom.

The concept

"The Celery worker runs my task" is true the way "the post office delivers my letter" is true — it hides every interesting part. The process you launch with celery -A app worker is a supervisor. It runs no task code at all. Its whole job is to pull messages off Redis, do bookkeeping, and hand the work to other processes.1

The fact underneath this lesson:

Inside one worker there are two layers: a single main process that talks to the broker (the Consumer), and a pool of child processes that execute tasks (the execution pool). The broker→worker hop is one BRPOP. The interesting handoff is the second one — main process to child — and it happens over an OS pipe, not over Redis.2

The steps

Here's the full route of one message, from the Redis list to a return value. We'll walk each piece underneath.

                       MainProcess  (supervisor — runs NO task code)
 Redis                ┌──────────────────────────────────────────────────┐
 list   ─BRPOP(kombu)─▶  Consumer ─▶ build Request ─▶ reserve (prefetch)   │
"celery"              │     │                                              │
                      │     ├─ ETA / countdown / rate-limit ─▶  Timer      │
                      │     └─ ready to run now ──────────────▶  Pool ──┐  │
                      └──────────────────────────────────────────────│──┘
                                                                pipe   │
                                                  ┌────────────────────▼──────┐
                                                  │ child process (prefork)    │  add(2,3) → 5
                                                  └────────────────────┬───────┘
                                                                       │ return value
                                                                       ▼
                                                          Result backend  (separate store —
                                                                           NOT the broker)

One hop off the broker (BRPOP), a fork in the road (Timer vs Pool), then a second hop over a pipe into a child. The result does not go back on the queue.

Step 1 — The Consumer pulls from the broker

The main process runs a component called the Consumer. It owns the broker connection and, through kombu, issues the BRPOP from Lesson 1. It doesn't busy-poll — for the Redis transport the worker drives kombu from an async event loop (the Hub), so an idle worker sleeps on the socket and wakes the instant a message lands.1

Step 2 — Prefetch: it grabs more than it can run

The Consumer doesn't pull one message per free slot. It pulls ahead — worker_prefetch_multiplier × concurrency messages — and parks the surplus in an in-memory buffer, reserved. Default multiplier is 4, so a worker with concurrency 8 reserves up to 32 messages before any of them finish.3

  concurrency = 8 child slots,  prefetch_multiplier = 4
  ⇒ worker holds up to 8 × 4 = 32 messages in flight
       8 actively running   +   24 reserved in memory, waiting their turn

Prefetch trades fairness for latency: reserved messages start instantly when a slot frees, but they're stuck inside this worker even if another worker goes idle.

Step 3 — The message becomes a Request

For each message, the Consumer applies the task's strategy — a small handler built per registered task — which decodes the envelope and wraps it in a celery.worker.request.Request object: the in-memory representation of "this specific call, with these args, this id, this delivery tag."2 From here on the worker reasons about a Request, not raw JSON.

Step 4 — The fork in the road: Timer or Pool

The Request is then routed by when it should run:2

  • Has an ETA / countdown, or is rate-limited → it goes to the Timer, an in-memory scheduler that holds it until its due time, then releases it to the pool. (This is why a countdown task still occupies the worker — it's parked in memory, already pulled off the broker.)
  • Ready to run now → straight to the execution pool.

Step 5 — The pool hands the job to a child over a pipe

The execution pool is where concurrency actually lives. The default is prefork: a set of child processes forked at startup (via billiard, a multiprocessing fork). The main process serializes the job and writes it down a pipe to an idle child; the child runs your function and writes the return value back up the pipe.4

  POOL TYPE     unit of concurrency     good for
  ─────────     ───────────────────     ─────────────────────────
  prefork       child processes (def.)  CPU-bound, isolation, default
  threads       OS threads              I/O-bound, lighter than prefork
  gevent        greenlets (coroutines)  many concurrent I/O waits
  eventlet      greenlets (coroutines)  many concurrent I/O waits
  solo          none — main process     debugging; runs in-line, no pool

Concurrency is a property of the pool, not the consumer. Swapping --pool changes how step 5 executes; steps 1–4 are unchanged.

Step 6 — The result leaves by a different door

The child's return value goes to the result backend — a separate store (often Redis again, but a different set of keys, or a database) keyed by task id. It does not go back onto the broker queue. Broker = work going in; backend = results coming out. They're independent systems that happen to both be able to use Redis.

Why this matters

The two-layer shape explains the symptoms people actually hit.

1. "I have idle workers but tasks are stuck." Prefetch (step 2). A worker that grabbed 32 messages holds them in memory even while a fresh worker sits empty — the stuck tasks were reserved off the broker long ago. The fix is worker_prefetch_multiplier = 1 for long tasks, so the broker stays the source of truth for "what's unclaimed."

2. Concurrency lives in the pool, not the fetch. Raising prefetch doesn't add parallelism — it only deepens the in-memory buffer. Parallelism is the number of pool children/threads/greenlets. Conflating the two is the classic Celery tuning mistake.

3. The main process is a bottleneck and a single point of bookkeeping. Every message flows through one Consumer before fanning out to children. If your task code blocks the main process (it shouldn't — it runs in a child), or the pipe backs up, the whole worker stalls. It also means the main process is what acks messages and what restores them on shutdown (a later lesson).

Interview line "A Celery worker is two layers: one main process — the Consumer — that BRPOPs messages off Redis, turns each into a Request, and prefetches ahead into memory; and a pool of child processes that actually run the code, fed over a pipe. Concurrency is the pool size; prefetch is just how far ahead the consumer reads. Results leave through a separate backend, not the broker."
The trap Shallow: "the worker runs the task." The worker's main process runs none of your code — it's a broker client and dispatcher; a forked child runs the task, fed over a pipe. The deeper tell is naming prefetch: messages are pulled off Redis and reserved in memory long before a slot is free, which is why "stuck tasks + idle workers" happens and why prefetch_multiplier=1 fixes it.

Recall — answer out loud before peeking:

① What does the worker's main process actually do — and what does it not do?

show answer

It runs the Consumer: connects to the broker, BRPOPs messages, builds Requests, prefetches, dispatches, and does bookkeeping (acks). It does not run task code — child processes in the pool do.

② Where does a message go between the Consumer and the child process?

show answer

It's turned into a Request, then either parked in the Timer (if it has an ETA/countdown/rate-limit) or sent to the execution pool, which writes it down a pipe to an idle child.

③ What is prefetch, and what is it not?

show answer

It's how many messages the consumer reserves ahead into memory: prefetch_multiplier × concurrency (default multiplier 4). It is not parallelism — that's the pool size. More prefetch just deepens the buffer.

④ Where does a task's return value go?

show answer

To the result backend — a separate store keyed by task id (can be Redis, but different keys, or a DB). Never back onto the broker queue.
