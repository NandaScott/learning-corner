Signals: exactly when each one fires

Signals are Celery's hooks — connect a function and it runs at a fixed moment. The catch interviewers and bugs both exploit: different signals fire in different processes. This lesson lays every signal onto the journey from Lessons 1–3, and says which process it runs in.

The concept

A Celery signal is a blinker-style dispatch point: signal.connect(handler) and the handler runs synchronously when Celery send()s it.1 They're not magic — they're send() calls placed at exact lines in the publish path, the consumer, and the tracer.

The fact that makes signals tricky:

Signals fire in whichever process reaches that line — and the three groups live in three different processes. Publish signals fire in the producer (your Django web process). task_received fires in the worker's main process (the Consumer, Lesson 2). The execution signals fire in the child process running the task (the tracer, Lesson 3). Connect a handler in the wrong place and it simply never fires.

The grand timeline

One task, end to end, with every signal in order and the process it fires in. The grey handler calls are from Lesson 3 — shown so you can see exactly where the signals sit relative to them.

  ── PRODUCER process (whoever called .delay) ───────────────────────────
     before_task_publish        ← message built, NOT yet on the broker
        · · · LPUSH to Redis · · ·
     after_task_publish         ← message now in the list

  ── MAIN worker process · the Consumer (Lesson 2) ──────────────────────
     task_received              ← message popped, Request built

  ── CHILD process · the tracer (Lesson 3) ──────────────────────────────
     task_prerun                ← FIRST thing in the trace
       (before_start)           ← then the Lesson-3 handler
       (run — your body)
       (mark_as_* + on_* handler)
     task_success │ task_retry │ task_failure     ← exactly one
       (after_return)           ← terminal outcomes only (skipped on RETRY)
     task_postrun               ← LAST thing; ALWAYS sent, even on retry
  ─────────────────────────────────────────────────────────────────────

Two ends to anchor on: task_prerun is the first thing the tracer does (before before_start), and task_postrun is the last (after after_return) and fires on every outcome.

Group 1 — Publish signals (in the producer)

SignalFires exactly whenKey args
before_task_publishafter the message is built, before it's sent to the brokerbody, exchange, routing_key, headers, properties
after_task_publishafter the message has been sent to the brokerheaders, body, exchange, routing_key

Both run "in the process that sent the task"2 — i.e. your web request, not the worker. This is the trap below. (task_sent is the old, deprecated version — ignore it.)

Group 2 — Receipt (in the worker's main process)

SignalFires exactly whenProcess
task_receivedthe Consumer pops a message and builds a Request (Lesson 2, step 3)main worker process

This is the one signal that fires in the supervisor, not a child. It's your hook for "a worker has claimed this message" — before any child touches it.

Group 3 — Execution signals (in the child process)

SignalFires exactly whenvs. Lesson-3 handler
task_prerunstart of the tracebefore before_start()
task_successafter the result is stored, on successafter on_success()
task_retrywhen self.retry() is handledafter on_retry()
task_failureon an uncaught exceptionafter on_failure()
task_postrunend of the trace, all outcomesafter after_return()

So the signal/handler interleave is: task_prerun → before_start → run → on_success → task_success → after_return → task_postrun.3 The signals bracket the handlers on the outside at the ends, and trail them in the middle.

Group 4 — The error/edge signals

SignalFires when
task_revokeda task is revoked (cancelled) — before it runs, or terminated mid-run
task_rejectedthe worker gets a message it can't decode or route to any known queue
task_unknownthe message names a task this worker hasn't registered
task_internal_errorthe tracer itself errors (not your code — Celery's machinery)

task_rejected and task_unknown fire in the main process (the Consumer can't even build a Request); they're the failure modes of Lesson 1's "the worker must already define this task."

Worker lifecycle signals

Separate from any single task — these mark the worker process itself starting and stopping. The precise bit: some fire once in the main process, some fire once per child.

  STARTUP
    celeryd_init            ← the FIRST signal a worker sends (main process)
    worker_init             ← before the worker starts (main process)
    worker_process_init     ← in EACH pool child as it boots  ⚠ must not block >4s
    worker_ready            ← worker is now accepting tasks (main process)

  SHUTDOWN
    worker_shutting_down    ← main process, shutdown begins
    worker_process_shutdown ← in EACH pool child just before it exits (not guaranteed)
    worker_shutdown         ← main process, final

worker_process_init / worker_process_shutdown fire in every child — the right place to open/close per-process resources like a DB connection that must not be shared across forked children.4

Why this matters

1. Process locality decides whether your handler ever runs. Register a before_task_publish handler only in your worker startup code and it will never fire — publishing happens in the web process. Conversely, a worker_process_init handler placed where only the web app imports it does nothing. Signals fire where the code path runs, full stop.

2. task_postrun is the "runs on every attempt" hook that after_return isn't. Lesson 3's after_return skips retries; task_postrun fires on every outcome including RETRY. So for metrics like "count every execution attempt," use task_postrun — using after_return would undercount every retried task.

3. Forked children need per-process setup. Because the prefork pool forks children (Lesson 2), resources opened in the main process — DB connections, sockets — are unsafe to share. worker_process_init is where you reopen them per child. This is the single most common Celery-signals bug in Django apps.

Interview line "Signals fire in three places: before/after_task_publish in the producer, task_received in the worker's main process, and task_prerun…task_postrun in the child running the task. task_prerun is the first thing the tracer does and task_postrun the last — and task_postrun fires on every outcome, including retries, unlike after_return."
The trap Shallow answer: "signals fire in the worker." Some don't — before_task_publish and after_task_publish fire in whatever process called .delay() (your web app). Getting process-locality right — and knowing worker_process_init fires once per forked child, which is why you reopen DB connections there — is what separates someone who's debugged Celery from someone who's only read about it.

Recall — answer out loud before peeking:

① In which process does before_task_publish fire, and why does it matter?

show answer

The producer — whatever called .delay() (e.g. the Django web process), not the worker. A handler connected only in worker code never fires for it.

② Order these around the task body: task_prerun, before_start, task_postrun, after_return.

show answer

task_prerun → before_start → (run) → after_return → task_postrun. Prerun is first in the trace; postrun is last.

③ You want a counter that increments on every execution attempt, retries included. Signal or handler — which, and why?

show answer

task_postrun signal. It fires on all outcomes including RETRY; after_return skips retries and would undercount.

④ Which signal fires once per pool child, and what's it for?

show answer

worker_process_init (and worker_process_shutdown on exit). For per-process resources — e.g. reopening a DB connection that mustn't be shared across forked children.
