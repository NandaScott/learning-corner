Celery + Redis · Lesson 5

Acks are emulated: the in-flight ledger

Back to the broker. A real message broker remembers which messages are in-flight and lets a worker "acknowledge" one when it's done. Redis has no such thing. This lesson is how kombu fakes it with two extra keys — and why that fake is the whole basis of surviving a crash.

The concept

Recall Lesson 1: the worker does BRPOP to get a message. BRPOP is destructive — it atomically pops the element off the list and hands it over. The instant a worker takes a message, that message is gone from the queue.1

That's a problem. On a real broker (RabbitMQ/AMQP), a delivered message sits in an "unacknowledged" holding area until the consumer sends an ack; if the consumer dies first, the broker re-queues it. Redis has no ack, no holding area, no ACK command. So:

kombu emulates acknowledgement with a second data structure — an "unacked" ledger. The moment a message is popped, kombu writes a copy into two keys (a hash + a sorted set). "Acking" means deleting that copy. While the copy exists, the message is recoverable; once it's deleted, it's gone for good.2

The steps

Follow one message through the ledger. A message is only ever in one of three places: the queue list, the unacked ledger, or gone.

  1. BRPOP "celery"            ── pops the message OFF the list (destructive)
                                  the queue list no longer has it
  2. QoS.append(tag, msg)      ── record it as in-flight, in TWO keys:
        HSET unacked  tag → dumps([raw_message, exchange, routing_key])
        ZADD unacked_index  tag → time()         (delivery timestamp)
  ───────────────────────────────────────────────────────────────────
        ( task runs — Lesson 3 )
  ───────────────────────────────────────────────────────────────────
  3a. ack(tag)                 ── done: delete the copy, atomically
        pipeline:  ZREM unacked_index tag   +   HDEL unacked tag
        → message no longer exists ANYWHERE

  3b. reject(tag, requeue=True) ── put it back:
        restore_by_tag(tag, leftmost=True)  →  LPUSH back onto "celery"

Default keys are literally unacked (a HASH) and unacked_index (a ZSET). The hash holds the message body; the sorted set holds when it was handed out.2

Step 1 — BRPOP deletes the message from the queue

This is the fact everything hangs on. There's no "peek." The pop and the removal are the same atomic operation, so the queue list is not a record of in-flight work — the moment a worker takes a message, the list has forgotten it. Something else has to remember, or a crash mid-task loses the work silently.

Step 2 — append: write the in-flight copy

kombu's QoS.append records the just-popped message under a unique delivery tag, in two keys at once:2

  unacked        (HASH)    delivery_tag → [ the raw message, exchange, routing_key ]
                            └─ enough to fully reconstruct & re-route the message

  unacked_index  (ZSET)    delivery_tag → delivery timestamp (score)
                            └─ ordered by time, so "what's been out too long?" is one query

Two keys because they answer two questions: the hash answers "what was the message?"; the sorted set answers "how long has it been in-flight?" — the question Lesson 6's visibility timeout asks.

Step 3a — ack: delete the copy for good

Acknowledging is just removing the ledger entry — from both keys, in one pipelined round trip so they can't drift apart:2

  pipeline:  ZREM unacked_index    .   HDEL unacked    . execute()

After this, the message exists nowhere. There is no undo — an acked message that was never actually completed is simply lost.

Step 3b — reject / restore: push it back

If a message must go back (explicit reject-with-requeue, or the recovery paths in Lesson 6), restore_by_tag reads the body out of the unacked hash and LPUSHes it back onto its original queue (leftmost=True — to the head, so it's retried promptly), then clears the ledger entry. The message rejoins the list from Lesson 1 and flows through the whole pipeline again.2

When does the ack actually fire?

That's the lever everything else turns on, and it's configurable. By default (acks_late=False), the message is acked before the task runs — kombu deletes the ledger entry almost immediately after delivery.3 With acks_late=True, the ack is held until after the task returns. Same machinery, very different crash behaviour — which is exactly Lesson 7. For now, just hold the mechanism: ack = delete the in-flight copy.

Why this matters

1. The unacked ledger is the only memory that a popped message ever existed. The queue list dropped it at BRPOP. So whether a crash loses work comes down to one question: was the ledger entry still there when the worker died? If yes, it can be restored; if it was already acked, the work is gone with no trace.

2. That's why the default is "at-most-once-ish." Default early-ack deletes the ledger entry before your task body even starts. Crash mid-task and there's nothing to recover — the message was acked, the list forgot it, the ledger forgot it. Late-ack (Lesson 7) keeps the entry alive for the whole run so the crash is recoverable — at the cost of possible double execution.

3. Redis "queue reliability" is a client-side fiction — a good one. The server is still just storing a list, a hash, and a sorted set. All the delivery-tracking logic lives in kombu, running inside your worker. Nothing on the Redis side knows what an ack is. This is the through-line of the whole course: the broker is dumb; the cleverness is emulated next to it.

Interview line "Redis has no ack. BRPOP destructively removes the message, so kombu immediately copies it into an unacked hash plus an unacked_index sorted set keyed by a delivery tag. Acking just deletes that copy from both; rejecting LPUSHes it back. The ledger is the only record an in-flight message exists — which is the entire basis for recovering one after a crash."
The trap Shallow: "the worker acks the message when the task finishes." That's late ack — not the default. By default Celery acks before the task runs, so a mid-task crash loses the message (the ledger entry is already gone). The deep tell is connecting three facts: BRPOP already deleted it from the list, so the unacked ledger is its only home, so default early-ack means there's nothing left to restore.

Recall — answer out loud before peeking:

① After a worker BRPOPs a message, where does it live?

show answer

Not in the queue list — BRPOP removed it. It lives only in the unacked ledger: the unacked hash (tag → message body) and the unacked_index sorted set (tag → delivery time).

② What does "acking" physically do on Redis?

show answer

Deletes the ledger entry from both keys, pipelined: ZREM unacked_index <tag> + HDEL unacked <tag>. The message then exists nowhere.

③ Why are there two keys instead of one?

show answer

The hash stores what the message is (body + routing, to reconstruct it); the sorted set stores when it was delivered (scored by time) so kombu can find messages that have been out too long — the basis of the visibility timeout.

④ With the default ack setting, does a worker crash mid-task lose the message? Why?

show answer

Yes. Default acks_late=False acks (deletes the ledger entry) before the task runs, so after the crash there's no copy to restore. Late-ack would keep it recoverable.
