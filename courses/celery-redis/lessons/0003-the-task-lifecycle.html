The task lifecycle: which method runs when

Lesson 2 dropped the message into a child process and said "it runs your function." That's the one step we now open up. When the child executes a task, your function is wrapped in a tracer that calls a fixed sequence of handler methods — and moves the task through a fixed sequence of states. Here's the exact order.

The concept

Your @app.task function is never called bare. When a child process picks up a Request (Lesson 2, step 5), it runs it through a compiled tracer (celery.app.trace.build_tracer) that surrounds your code with lifecycle hooks.1 Those hooks are methods on the Task class you can override:

before_start → run → (record state) → one of on_success / on_retry / on_failure → after_return. The order is fixed and the same every time. Two facts trip people up, and we'll nail both: the outcome state is written to the backend before the outcome handler runs, and after_return does not run on a retry.1

The steps

This all happens inside one child process, synchronously, in one thread.2 Here's the timeline of one execution.

  child process · trace_task(uuid, args, kwargs)
  ─────────────────────────────────────────────────────────────────
  1.  before_start(task_id, args, kwargs)     ← blocks; run() waits for it
  2.  retval = run(*args, **kwargs)           ← YOUR @task function body
        │
        ├── returned normally ─────────▶  3a. backend.mark_as_done → SUCCESS
        │                                  4a. on_success(retval, …)
        │
        ├── called self.retry() ───────▶  3b. backend.mark_as_retry → RETRY
        │                                  4b. on_retry(exc, …)
        │
        └── raised an exception ───────▶  3c. backend.mark_as_failure → FAILURE
                                           4c. on_failure(exc, …, einfo)
  ─────────────────────────────────────────────────────────────────
  5.  after_return(status, retval, …)   ← runs for SUCCESS / FAILURE only
                                           SKIPPED for RETRY (and IGNORED/REJECTED)

Exactly one of the 3/4 branches fires per run. Step 3 (record state) always precedes step 4 (the handler) — the backend is updated before your handler sees the outcome.

Step 1 — before_start(task_id, args, kwargs)

The first hook, added in Celery 5.2. It runs before your function and blocks it — run() will not begin until before_start returns.3 Use it for per-execution setup (a request-scoped log context, a DB connection check). Anything slow here is latency added to every task.

Step 2 — run() — your function

run is your task body. @app.task turns the function you wrote into the run method of a Task subclass; calling the task ultimately calls run. It ends in exactly one of three ways, and that choice drives everything after it:

  • returns a value → success path
  • calls self.retry(...) → raises a special Retry exception → retry path
  • raises any other exception → failure path

Step 3 — The backend records the outcome first

Before any outcome handler runs, the tracer writes the result to the result backend (Lesson 2, step 6):1

  success →  mark_as_done(uuid, retval)     state = SUCCESS   (result = return value)
  retry   →  mark_as_retry(uuid, exc)       state = RETRY     (result = the exception)
  failure →  mark_as_failure(uuid, exc)     state = FAILURE   (result = exc + traceback)

This ordering is why on_success's docstring can promise "the result is already visible to clients" — the backend write happened on the line before.

Step 4 — Exactly one outcome handler

HandlerSignatureFires when
on_success(retval, task_id, args, kwargs)run() returned normally
on_retry(exc, task_id, args, kwargs, einfo)run() called self.retry()
on_failure(exc, task_id, args, kwargs, einfo)run() raised any other exception

Their return values are ignored — these are side-effect hooks (alerting, cleanup, metrics). When on_failure runs, the state in the backend is already FAILURE.

Step 5 — after_return(status, retval, task_id, args, kwargs, einfo)

The "finally" hook — but with a sharp edge. It runs for terminal states only. The tracer guards it with if state not in IGNORE_STATES, and IGNORE_STATES = {RETRY, IGNORED, REJECTED}.1 So:

  SUCCESS  → after_return runs   ✓
  FAILURE  → after_return runs   ✓
  RETRY    → after_return SKIPPED ✗   (the task isn't done — it'll run again)
  IGNORED / REJECTED → SKIPPED   ✗

"after_return always runs last" is true per terminal outcome — a retried task simply hasn't reached a terminal outcome yet, so its after_return only fires on the attempt that finally succeeds or fails.

The state machine

Alongside the method calls, the task moves through states — the thing AsyncResult.state reports. The handler outcomes above are the transitions into the terminal states.

  PENDING ──▶ RECEIVED ──▶ (STARTED) ──▶ ┬─▶ SUCCESS
   │            │                        ├─▶ FAILURE
   │            │                        └─▶ RETRY ──┐
   │            │                                    │ re-published as a new delivery
   │            └◀───────────────────────────────────┘
   └─ REVOKED (cancelled before it ran)

PENDING is implicit — it means "no record exists," so an unknown/typo'd id also reads PENDING. RECEIVED is set when the consumer picks the message up (Lesson 2). STARTED is only recorded if task_track_started is enabled (default off).4

Retry is a re-delivery, not a pause

self.retry() doesn't sleep the worker. It marks the task RETRY, re-publishes a fresh message (with an incrementing retries count and an optional countdown), and the current execution ends. That new message later flows through the entire pipeline again — Lesson 1's LPUSH, Lesson 2's consumer, and this lifecycle from the top. That's why after_return is skipped on RETRY: from the tracer's view this attempt is over, but the task isn't done.

Why this matters

1. Where you put cleanup logic depends on this order. Need teardown on every terminal outcome? after_return — but remember it skips retries. Need it on every attempt including retries? You can't use after_return; you need a signal that fires on all paths (Lesson 4's task_postrun).

2. "Result visible" timing is exact. Because step 3 precedes step 4, a client polling AsyncResult can observe SUCCESS and the return value before your on_success has finished running. Don't write code that assumes on_success is a gate before the result exists — it isn't.

3. Retries quietly multiply executions. Each retry is a brand-new trip through the whole system. before_start and run fire again from scratch; there's no resumption. Combined with at-least-once delivery (a later lesson), this is why task bodies must be idempotent.

Interview line "A task isn't just run() — the worker wraps it in a tracer: before_start, then run, then the backend records the state, then exactly one of on_success/on_retry/on_failure, then after_return — but after_return is skipped on retry, because a retry is a re-published new delivery, not a finished task."
The trap Two shallow beliefs to avoid. First: "after_return always runs." It doesn't — RETRY, IGNORED, and REJECTED are in IGNORE_STATES and skip it. Second: "on_success runs, then the result becomes available." Backwards — the backend is written before the outcome handler, so the result is already visible when on_success fires.

Recall — answer out loud before peeking:

① List the handler methods in execution order for a successful task.

show answer

before_start → run → (backend records SUCCESS) → on_success → after_return.

② Does after_return run when a task calls self.retry()? Why or why not?

show answer

No. RETRY is in IGNORE_STATES ({RETRY, IGNORED, REJECTED}). A retry re-publishes a new delivery, so this attempt isn't a terminal outcome — after_return only fires on the attempt that ends in SUCCESS or FAILURE.

③ When on_failure runs, what state is already in the backend?

show answer

FAILURE — the tracer calls mark_as_failure before the outcome handler, so the state (and the exception/traceback) is recorded first.

④ What does before_start block, and why care?

show answer

It blocks run() — your function won't start until before_start returns. Anything slow in it is latency added to every single task execution.
