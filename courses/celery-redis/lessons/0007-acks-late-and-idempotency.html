Celery + Redis · Lesson 7

acks_late, at-least-once & idempotency

The whole arc converges on one boolean. Lesson 5 said acking means deleting the in-flight ledger entry. Lesson 6 said unacked entries get redelivered. So the question "when do we ack: before or after the task runs?" decides your delivery guarantee. This lesson answers it, and explains why that forces your tasks to be idempotent.

The concept

Recall the ledger (Lesson 5): the ack deletes a message's only surviving copy. And the restore loop (Lesson 6): an unacked entry comes back. Put those together and the timing of the ack, relative to running your task, is the delivery guarantee.

The fact:

acks_late is a single boolean that chooses between at-most-once and at-least-once delivery, and there is no third option. Default False acks before the task runs, so a crash loses the task (at-most-once). True acks after, so a crash redelivers it (at-least-once). Celery on Redis cannot give you exactly-once, so if you choose at-least-once you must make the task safe to run more than once.1

The fork

One timeline, two ack points. The window between "task starts" and "ack" is the danger zone: a crash there is what differs.

  acks_late = False   (DEFAULT)
     BRPOP ─▶ ack (delete ledger entry) ─▶ run task ─▶ done
                      ▲
              crash AFTER here = work LOST (nothing to restore)   → AT-MOST-ONCE
              no duplicates, but a killed worker silently drops its tasks

  acks_late = True
     BRPOP ─▶ run task ─▶ done ─▶ ack (delete ledger entry)
                  ▲
          crash BEFORE the ack = ledger entry survives → Lesson 6 restores it
          → task runs AGAIN on another worker                    → AT-LEAST-ONCE
          recovers lost work, but the task can run 2+ times

Same machinery (Lesson 5's ledger, Lesson 6's restore). The only change is whether the ack happens on line 1 or line 4. That's the entire reliability decision.1

Default (acks_late=False) — at-most-once

The message is acked essentially on receipt, before your code runs. Its ledger entry is gone, so a worker crash mid-task leaves nothing to restore: the task is silently dropped. The upside: a task never runs twice from a crash. Good when re-running is worse than skipping (e.g. "send this one-off email" where a duplicate is worse than a miss, and you have other monitoring).

acks_late=True — at-least-once

The ack waits until the task returns. Crash before that and the entry lingers; Lesson 6's restore (immediately on clean shutdown, or after the visibility timeout on a hard kill) puts it back, and it runs again. The upside: lost work is recovered. The cost: the task may run more than once, so it has to tolerate that.

The edge cases (where "exactly when" matters)

Three settings refine what "after" means, and they're the difference between a textbook answer and a correct one.2

Setting (default)What it governs
acks_on_failure_or_timeout
(True)		With acks_late, a task that raises or times out is still acked, so ordinary failures are not redelivered (they're recorded as FAILURE; use retry to re-run). Set False to redeliver failed/timed-out tasks instead.
reject_on_worker_lost
(False)		If a child is killed abruptly (OOM, SIGKILL) mid-task, by default the message is acked, not redelivered: poison-pill protection, so a task that kills workers doesn't loop forever. Set True to re-queue it instead.
Reject(requeue=…)
(explicit)		Raise Reject in the task for manual control: requeue=True puts it back (guard with the request.delivery_info['redelivered'] flag to avoid infinite loops); requeue=False drops it (dead-letter on AMQP).

The subtle takeaway: even with acks_late=True, redelivery is not automatic on every failure. It happens on worker death (lost connection / clean restart / timeout), but a normal exception still acks. Late-ack protects against the worker dying, not against your code raising.

Why there's no exactly-once

Exactly-once would require acking the message and committing the task's side effects as one atomic step. They're separate systems, Redis and whatever your task writes to, so a crash can always land between "side effect committed" and "message acked." Either you ack first (risk losing the work) or you ack last (risk repeating it). There is no instant that's safe for both. Every distributed queue lives with this; Celery just exposes it as one switch.

Interview line "acks_late picks your delivery guarantee. Off, the default, acks before the task runs, so a crash loses it: at-most-once. On, it acks after, so a crash redelivers via the visibility-timeout restore: at-least-once. There's no exactly-once: you can't atomically commit the side effect and the ack across two systems, so at-least-once only works if the task is idempotent."

Idempotency: the contract at-least-once demands

If a task can run twice, running it twice must equal running it once. That's idempotency, and with acks_late it's the precondition that makes at-least-once safe.3 Common ways to get there:

TechniqueHow
Natural idempotenceWrite operations that don't care how often they run: UPDATE … SET status='paid', an upsert, S3 PUT. Re-running converges to the same state.
Idempotency key + dedupDerive a stable key from the work (e.g. order id), record "done" in a store, and short-circuit if the key's already there. Check the redelivered flag as a hint.
Database unique constraintLet the DB reject the second insert (unique on a business key); catch the violation and treat it as success.
Make non-idempotent steps retryable, not duplicableFor "charge a card", exactly the thing you can't run twice, push idempotency to the downstream API (most payment APIs take an idempotency key) rather than relying on the queue.

Why this matters — the whole course in one decision

1. The guarantee is a choice, not a default you inherit. Out of the box you have at-most-once: a crashed worker drops its in-flight tasks. If that's unacceptable, you opt into acks_late=True, and you pay for it by making tasks idempotent. Knowing it's a deliberate trade keeps you from assuming Celery "just doesn't lose things" (it does, by default) or "runs everything once" (it can't).

2. The settings interlock. acks_late + worker_prefetch_multiplier=1 (Lesson 2) is the canonical reliable config: don't reserve work you might crash on, and don't ack it until it's done. Pair it with a visibility timeout (Lesson 6) longer than your slowest task so the restore loop doesn't redeliver work that's merely slow.

3. It all rests on the dumb broker. Step back across the seven lessons: a Redis list (L1), a worker that pops and dispatches to children (L2), a tracer running lifecycle hooks and signals (L3–L4), an emulated ack ledger (L5), a restore loop (L6), and one boolean that turns those into a delivery guarantee (L7). Redis stored a list, a hash, and a sorted set. Every guarantee above that was built by kombu and Celery in the client.

The trap Shallow: "turn on acks_late and Celery won't lose or double-run tasks." Wrong on both halves. It doesn't give exactly-once, it moves you from at-most-once to at-least-once, trading dropped work for possible duplicates. And it doesn't redeliver on ordinary failures (those still ack) or even, by default, when a child is OOM-killed (reject_on_worker_lost=False). Knowing late-ack protects against worker death, demands idempotency, and still isn't exactly-once is the complete answer.

Recall — answer out loud before peeking:

① What delivery guarantee does each setting of acks_late give, and why?

show answer

False (default): ack before run → a crash loses the task → at-most-once. True: ack after run → a crash leaves the ledger entry to be restored → at-least-once.

② Why can't Celery on Redis offer exactly-once?

show answer

Acking and committing the task's side effects are separate systems; a crash can always fall between them. Ack-first risks loss, ack-last risks repeat, no instant is safe for both.

③ With acks_late=True, does a task that raises an exception get redelivered?

show answer

No, acks_on_failure_or_timeout=True (default) acks it anyway; it's recorded FAILURE. Redelivery happens on worker death, not ordinary exceptions. (Use retry to re-run on failure.)

④ Why does at-least-once force idempotency, and name one way to achieve it?

show answer

Because the task can run 2+ times, running twice must equal running once. E.g. upserts / natural idempotence, an idempotency key + dedup store, a DB unique constraint, or pushing an idempotency key to the downstream API.
