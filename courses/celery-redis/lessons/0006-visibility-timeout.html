Celery + Redis · Lesson 6

The visibility timeout: how lost work comes back

Lesson 5 left a question: a message popped by a worker lives only in the unacked ledger. So what happens to that ledger entry if the worker dies and never acks? Without an answer it sits there forever, the work lost with it. The visibility timeout is the answer, and it's also the source of Celery-on-Redis's most infamous bug.

The concept

The unacked ledger (Lesson 5) can leak. A worker BRPOPs a message, kombu records it in unacked + unacked_index, then the worker is kill -9'd mid-task. It never acks. That ledger entry now has no owner and would linger indefinitely. Something has to notice and recover it.

The fact:

Every worker runs a recovery loop every 10 seconds that re-queues any in-flight message older than the visibility timeout (default 3600s / 1 hour). "Visibility timeout" is borrowed terminology: on Redis there's no native invisibility. Mechanically it's just a deadline: an unacked ledger entry whose delivery timestamp is older than now − visibility_timeout is assumed lost, and kombu LPUSHes it back onto its queue.1

This is exactly what the unacked_index sorted set was built for (Lesson 5): it's keyed by delivery time, so "what's been out too long?" is a single range query.

The steps

The recovery loop, running in every worker, on a timer.

  register_with_event_loop:
     loop.call_repeatedly(10, maybe_restore_messages)   ── every 10 SECONDS:

  1. ceil = time() - visibility_timeout          (default: now - 3600s)
  2. with Mutex(unacked_mutex_key):               ── only ONE worker restores at a time
  3.    expired = ZREVRANGEBYSCORE unacked_index  ceil  0   withscores
            └─ every entry whose delivery timestamp is older than `ceil`
  4.    for tag in expired:
            restore_by_tag(tag)    →   LPUSH the message back onto its queue
  ─────────────────────────────────────────────────────────────────────────
  5. the message is back in the list → flows through the whole pipeline again
                                        (Lesson 1 BRPOP, Lesson 2 consumer, Lesson 3 run)

The check runs every 10s; the threshold is the visibility timeout (1h default). So a lost message is recovered within ~10s after its hour is up, not 10s after the crash.1

Step 1 — The deadline: now − visibility_timeout

Each pass computes a cutoff timestamp. The unacked_index scores are the times messages were delivered (Lesson 5). Anything delivered before the cutoff has been in-flight longer than the timeout allows, so it's presumed dead. Default cutoff is one hour ago.2

Step 2 — The mutex: only one worker restores

Every worker runs this loop, so without coordination ten workers would each re-queue the same expired message ten times. kombu takes a Redis-based lock (unacked_mutex_key) around the restore, so exactly one worker performs each restoration pass.1

Step 3 — The range query

ZREVRANGEBYSCORE unacked_index ceil 0 pulls every tag scored at or below the cutoff, the expired ones, in one shot (capped by unacked_restore_limit per pass). This is why the index is a sorted set and not part of the hash: the "expired?" question is a range scan over time.

Step 4 — Restore = LPUSH back

restore_by_tag reads the body out of the unacked hash and LPUSHes it back onto its original queue, then clears the ledger entry. From here it's an ordinary queued message again: there is no "resume"; the task runs from the top.

Clean shutdown vs hard kill — the Lesson 5 cliffhanger

This is where the two crash modes split, and it's a favourite interview probe.

  CLEAN shutdown (SIGTERM / warm shutdown / deploy restart)
     restore_at_shutdown = True  →  on channel close, restore_unacked()
     re-queues ALL in-flight messages IMMEDIATELY
     ⇒ near-instant redelivery, no waiting

  HARD kill (kill -9, OOM, power loss, network partition)
     no chance to run any cleanup
     ⇒ ledger entries just sit there until the 10s loop finds them
        EXPIRED — i.e. up to visibility_timeout (default 1 HOUR) later

So the visibility timeout only governs the ungraceful path. A normal deploy restart redelivers instantly; only a genuine crash makes you wait out the timeout.3

The infamous trap: ETA / countdown tasks

An eta/countdown task is pulled into a worker's memory and held there until its due time, and it is not acked until it starts executing.4 So its unacked ledger entry is live the entire time it waits.

  task.apply_async(countdown=7200)    # run in 2 hours
     t=0      worker A pops it, holds in memory, NOT acked
              unacked_index entry timestamp = t=0
     t=3600   visibility_timeout elapses. Worker B's restore loop sees an
              entry older than 1h, assumes it's lost → LPUSH back onto queue
     t=3600   worker B (or A) pops the "recovered" copy, ALSO holds it
     t=7200   BOTH copies fire  →  task runs TWICE

No worker crashed. The task was simply scheduled longer than the visibility timeout, so the recovery loop mistook a waiting task for a lost one. This is the canonical Celery-on-Redis duplicate-execution bug.4

The fixes, in order of preference: don't use long ETAs on Redis: use database-backed periodic scheduling (beat), which isn't subject to the timeout; or, if you must, raise visibility_timeout above your longest ETA/countdown/retry delay. Raising it is a real trade-off, though: it also delays redelivery of genuinely lost (hard-killed) tasks by the same amount.2

Why this matters

1. "Redelivery" is a client-side sweep, not a broker feature. The same theme as Lesson 5: Redis isn't tracking timeouts. Every worker is polling the sorted set every 10 seconds and re-pushing stale entries. The broker just stores keys.

2. The default 1-hour timeout is a latency floor on crash recovery. Hard-kill a worker running 50 tasks and those 50 messages are stuck for up to an hour before any other worker retries them. People expect instant failover; on Redis with defaults, ungraceful failure means up-to-an-hour delay.

3. Visibility timeout × task duration is a design constraint. Any task, running or merely scheduled, that stays unacked longer than the timeout will be redelivered and re-run. So either keep tasks shorter than the timeout, or raise the timeout, or use acks_late deliberately and make the task idempotent (Lesson 7). You can't ignore the relationship.

Interview line "On Redis there's no real ack and no real invisibility. Every worker runs a 10-second loop that scans the unacked_index sorted set for entries delivered more than visibility_timeout ago, default an hour, and LPUSHes them back onto the queue, under a mutex so only one worker does it. Clean shutdown re-queues in-flight work immediately; a hard kill makes you wait out the timeout."
The trap Shallow: "the visibility timeout is how long Redis hides a delivered message." There's no hiding: the message was destructively popped (Lesson 5); it's in the unacked ledger, not the queue. The deep tell is the duplicate-execution failure: an eta/countdown task waits unacked in memory, and if it waits longer than the timeout, another worker's recovery loop re-queues it and it runs twice, no crash required. That's the bug that separates the read-about-it crowd from the got-paged-at-3am crowd.

Recall — answer out loud before peeking:

① What runs every 10 seconds in each worker, and what does it look for?

show answer

The restore loop (maybe_restore_messages → restore_visible). It range-scans unacked_index for entries delivered more than visibility_timeout ago (default 1h) and LPUSHes them back onto their queue.

② Why does the restore happen under a mutex?

show answer

Every worker runs the loop. Without a lock, N workers would each re-queue the same expired message N times. The unacked_mutex_key lock lets exactly one worker restore per pass.

③ Clean SIGTERM vs kill -9: how fast does in-flight work come back?

show answer

Clean shutdown re-queues everything immediately (restore_at_shutdown=True → restore_unacked). Hard kill skips cleanup, so messages wait until the recovery loop sees them expire, up to the full visibility timeout (default 1h).

④ How can a task run twice with no worker crashing?

show answer

An eta/countdown longer than the visibility timeout: the task waits unacked in memory, the recovery loop mistakes the stale entry for a lost message and re-queues it, so two copies eventually fire. Fix: avoid long ETAs on Redis (use beat) or raise visibility_timeout above the longest delay.
