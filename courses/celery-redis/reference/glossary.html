Glossary

Canonical vocabulary for this course. Every lesson uses these words to mean exactly this. Verified against the kombu source and Celery docs — not generic message-queue theory.

The pieces

Broker
The message store that sits between producers (your web process) and consumers (workers). With Redis, the broker is just a Redis server — a key-value store, not a purpose-built queue. Everything queue-like is built on top by kombu.
kombu
Celery's messaging library. It owns the Redis "transport" — the code that turns Redis primitives (lists, sorted sets, pub/sub) into something with queue and acknowledgement semantics. When this course says "the Redis transport," it means kombu's kombu/transport/redis.py.
Producer
Whatever calls task.delay(...) / task.apply_async(...) — usually a Django view or another task. It serializes a message and pushes it onto the broker.
Worker
A long-running process that pulls messages off the broker and executes the task functions. One worker can run many child processes/threads (its concurrency).
Message
The serialized unit of work on the broker. A JSON envelope: headers (task name, task id), a base64-encoded body (the args/kwargs), and properties (routing info + a delivery_tag). Not a Python object — a string of bytes Redis stores verbatim.
Task
The Python function decorated with @app.task. A message is a request to run a task with specific arguments; the task is the code.
Queue
Physically, a Redis LIST keyed by the queue's name (default celery). "Putting a task on the queue" = LPUSH onto that list. There is no special queue object — it's a list.

Redis operations

LPUSH / BRPOP
Produce / consume. The producer LPUSHes a message onto the left of the list; the worker BRPOPs from the right (blocking pop — waits if the list is empty). Push-left + pop-right = FIFO order.
Sorted set (ZSET)
A Redis structure mapping members to a numeric score, kept ordered by score. kombu uses one (unacked_index) to track when each in-flight message was delivered, so it can find ones that have been out too long.

Reliability machinery (defined in L3–L4)

Acknowledgement (ack)
The signal "this message is done, delete it for good." AMQP brokers have this natively; Redis does not. kombu emulates it (L3).
Delivery tag
A unique id kombu assigns each delivered message so it can track and later ack/restore that specific in-flight copy.
unacked / unacked_index
The two Redis keys behind emulated acks. unacked is a HASH (delivery_tag → the message payload). unacked_index is a ZSET (delivery_tag → delivery timestamp). Ack removes from both; nothing removes them if the worker dies — which is what makes recovery possible (L3).
Visibility timeout
How long a delivered-but-unacked message stays "invisible" before kombu assumes the worker died and re-queues it. Default 3600s (1 hour). The source of the classic duplicate-execution trap (L4).
Prefetch
How many messages a worker reserves at once beyond the ones it's actively running (worker_prefetch_multiplier). Reserved messages count as unacked (L2).
acks_late
Ack after the task finishes instead of when it's received. Turns delivery from at-most-once into at-least-once — survives crashes, but can double-run. Demands idempotent tasks (L5).

Worker anatomy (L2)

Main process / supervisor
The process you launch with celery worker. Runs the Consumer and does bookkeeping; runs no task code itself.
Consumer
The component in the main process that owns the broker connection, BRPOPs messages, turns each into a Request, prefetches, and dispatches to the pool.
Request
celery.worker.request.Request — the in-memory object a message becomes once received: this specific call, its args, id, and delivery tag.
Execution pool
The child processes (or threads/greenlets) that actually run task code. Default is prefork (forked child processes via billiard). Concurrency = pool size.
Prefetch
Messages reserved into the main process's memory ahead of free slots: worker_prefetch_multiplier × concurrency (default multiplier 4). Prefetch ≠ concurrency.
Result backend
The separate store (Redis with different keys, or a DB) where return values land, keyed by task id. Distinct from the broker — work goes in via the broker, results come out via the backend.

Task lifecycle (L3)

Tracer
celery.app.trace.build_tracer — the compiled wrapper a child runs your task through. It calls the handler methods and sends the task signals in a fixed order.
Handler methods
Overridable Task methods, in order: before_start → run → (state recorded) → one of on_success/on_retry/on_failure → after_return. before_start blocks run; after_return is skipped for RETRY/IGNORED/REJECTED.
run()
Your @app.task function body — the decorator makes it the run method of a Task subclass.
Task states
PENDING (implicit — no record), RECEIVED, STARTED (only with task_track_started), SUCCESS, FAILURE, RETRY, REVOKED.
IGNORE_STATES
{RETRY, IGNORED, REJECTED} — the states for which the tracer skips after_return. The reason "retry doesn't run after_return."
Retry
self.retry() raises a Retry exception, records RETRY, and re-publishes a fresh message. Not a pause — a new delivery that flows through the whole pipeline again.

Signals (L4)

Signal
A blinker-style dispatch point Celery send()s at a fixed line. signal.connect(fn) runs fn synchronously there. Fires in whichever process reaches the line.
Publish signals
before_task_publish / after_task_publish — fire in the producer process (whoever called .delay()), around the broker send.
task_received
Fires in the worker's main process when the Consumer builds a Request.
Execution signals
Fire in the child, in trace order: task_prerun (before before_start) → task_success/task_retry/task_failure → task_postrun (after after_return; sent on every outcome, including retry).
Worker-lifecycle signals
celeryd_init (first) → worker_init → worker_process_init (per child) → worker_ready; shutdown mirrors it. worker_process_init/_shutdown fire once per forked child — where per-process resources (DB connections) are opened/closed.

Delivery guarantees (L5–L7)

Visibility timeout
The deadline an unacked ledger entry has before it's presumed lost and re-queued. Default 3600s. A 10-second restore loop in each worker scans unacked_index for entries delivered longer ago than this and LPUSHes them back.
Restore
restore_by_tag — reading a message body out of the unacked hash and LPUSHing it back onto its queue. Triggered by the timeout sweep, an explicit reject-with-requeue, or clean shutdown (restore_at_shutdown).
acks_late
If False (default) the message is acked before the task runs (at-most-once); if True, after it returns (at-least-once). The single switch that picks the delivery guarantee.
At-most-once
Each message runs zero or one time. A worker crash drops in-flight work; no duplicates. The default (acks_late=False).
At-least-once
Each message runs one or more times. A worker crash redelivers it; duplicates possible. Requires idempotent tasks. Enabled by acks_late=True.
Idempotency
The property that running a task twice equals running it once. The precondition that makes at-least-once safe. Celery on Redis offers no exactly-once, so this is the task author's responsibility.
reject_on_worker_lost / acks_on_failure_or_timeout
The edge knobs. By default an abruptly-killed child's task is acked not redelivered (poison-pill protection), and an ordinary task failure under acks_late is still acked. So late-ack redelivers on worker death, not on every error.
