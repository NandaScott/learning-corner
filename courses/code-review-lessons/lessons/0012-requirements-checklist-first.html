Code Review Lessons · Arc 6 · Lesson 12

Requirements checklist first

"Include the customer's details" was in the ticket and missing from the first draft. It slipped past personal review and AI-assisted review both, and surfaced late — an extra day of rework. A missing requirement is invisible to code review; only the ticket says what should exist.

The mistake

Work started without a complete, enumerated list of what the ticket asked for. One requirement — "include the customer's details" — was missed in the first draft. It then survived every review, personal and AI-assisted, because review looks at the code that exists and asks whether it's correct. No reviewer flagged the absence, because there was nothing present to flag. It was caught late, and cost an extra day.

This is the one catch in the course that no amount of code-quality review would have found. The other eleven lessons sharpen the review of code on the page. This one is about code that never made it onto the page — and the only artifact that knows it should have is the ticket.

what code review checks          what it CANNOT check
──────────────────────          ─────────────────────
is the code that EXISTS          is required code ABSENT?
correct, clean, tested?            → nothing present to flag
   ✓ catches bugs                  → reviewer sees a coherent diff
   ✓ catches bad names             → the gap is invisible
   ✗ catches nothing missing

only the TICKET says what should exist.
no enumerated list → an omission has nothing to fail against → it ships

A missing requirement has nothing to fail against until someone reads the ticket beside the diff.

How to catch it — enumerate the ticket, then check the diff against it

Before writing code, read the ticket and pull every requirement into an explicit checklist — from the description, the acceptance criteria, and any linked spec. Keep it visible while you work. Before opening the PR, walk the checklist against the diff and confirm each item is implemented. That's a spec-adherence pass, separate from code-quality review.

  "did I build everything asked?"  ≠  "is what I built good?"

Spec adherence and code quality are different questions; the checklist is what makes the first one answerable.

Step 1 — Enumerate before the first draft, not after

At the start of a ticket, turn its description, acceptance criteria, and any linked spec into a list of checkboxes. The point of doing it first is that the list becomes the thing an omission can fail against. Without it, a missed requirement has no counterpart — there's nothing that says "this was supposed to be here," so nothing surfaces the absence until it's expensive.

Step 2 — Keep the list visible while you build

The checklist isn't a one-time reading of the ticket; it's a reference you keep beside the work. Requirements are easy to hold in mind at the start and easy to lose as the implementation grows and attention narrows to the hard parts. A visible list is what carries a low-drama requirement like "include the customer's details" all the way to the end, past the interesting problems that crowd it out.

Step 3 — Run the spec gate separately from the quality gate

Before requesting review, walk the checklist against the diff: is each item actually implemented? Treat "did I build everything that was asked?" as a distinct gate from "is what I built good?" Code review — human or AI — answers the second well and the first not at all, because it reasons about present code. Only a deliberate item-by-item pass against the enumerated requirements answers the first.

Why it matters

Code review checks whether the code that exists is correct; it structurally cannot check whether required code is absent, because an absence presents nothing to react to. A reviewer reads a coherent diff, finds it reasonable, and approves — while a whole requirement sits unbuilt and unmentioned. That's why this failure survives every review and surfaces late, where it's most expensive: the earlier lessons all catch things visible in the diff, and this one, by definition, isn't. The enumerated checklist is the fix precisely because it moves the requirement out of your head and onto paper, where an omission finally has something to fail against. It's the same principle as Lesson 10 — enumerate the real fields, don't render from memory — lifted from the domain object up to the ticket: the spec, not your recollection of it, is the checklist.

Key line Code review checks whether present code is correct; only the ticket says what should exist. Enumerate the ticket's requirements into a checklist before the first draft, keep it visible, and walk it against the diff before the PR — a spec gate separate from the quality gate.
The common trap The near-miss is trusting review — your own, or an AI's — to catch a missing requirement. Review reasons about the code in front of it; a requirement that was never implemented leaves nothing in the diff to notice, so it passes clean. Thoroughness of review can't substitute for an enumerated spec, because the two answer different questions.

show answer

An enumerated requirements checklist drawn from the ticket, built before the first draft — so an omission has something concrete to fail against, and walked against the diff before opening the PR.

show answer

The spec gate — "did I build everything asked?" (checklist vs diff) — and the quality gate — "is what I built good?" (code review). Code review answers the second, not the first.
