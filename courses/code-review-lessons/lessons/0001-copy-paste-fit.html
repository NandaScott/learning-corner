Copy a pattern only after checking it fits

The most-cited catch on the PR. An existing method was reused wholesale into a new endpoint, and it carried assumptions that were true at the origin and false at the destination.

The mistake

The quote endpoint needed a template context, and an obvious donor was already in the codebase: Order.order_receipt_context. It got reused wholesale, down to the same cache key, and its currency handling came along with it.

Two of those decisions did not survive contact with the new use. The app only supports USD, and BigCommerce exposes no display currency, so the currency-formatting logic had nothing to do. And the cache key belonged to the receipt concern; sharing it across a quote and a receipt lets one path serve another path's cached bytes. One review comment made the point directly: when you copy code, ask whether it still makes sense for the new use before you keep it.

order_receipt_context           reused as-is for the quote
─────────────────────           ──────────────────────────
build context dict        →     build context dict          ✓ shared behaviour
format transacting currency →   format currency (USD-only)  ✗ dead logic here
cache under receipt key    →    cache under receipt key     ✗ two concerns, one key

The copied block fit its origin. Only the first line fit the destination.

How to catch it — annotate before you paste

The habit that prevents this runs at review-of-your-own-diff time, one copied block at a time.

1. Read the donor      understand what each line does at its origin
2. Annotate each line  write "why does this belong here?" against every line
3. Cut the origin-only lines  drop anything that only made sense where it came from
4. Factor the shared part     if the whole block is shared, extract it — don't copy it

The output is either a smaller, justified block, or a single shared function both call sites use.

Step 1 — Read the donor

Before pasting, read the source method as if you were reviewing it. An old codebase carries decisions made by many hands, for contexts that may no longer hold. The currency handling in order_receipt_context was written for a reason that applied to receipts; nothing guarantees it applies to a quote.

Step 2 — Annotate each line with why it belongs

Go line by line and answer one question against each: why does this belong in the new use? The answer for the context-building lines was "a quote needs a context too." The answer for the currency formatting was "…it doesn't, we're USD-only." That silence is the signal.

Step 3 — Cut the origin-only lines

Any line whose only justification is "it was in the original" gets removed. Copied code encodes someone else's assumptions into your feature; the currency logic encoded a multi-currency assumption the quote path does not have. Cutting it is not tidying. It removes behaviour that was never wanted here.

Step 4 — Factor shared behaviour instead of duplicating it

If the block genuinely is shared behaviour, the fix is not a cleaner copy but a single source of truth both paths call. Two copies mean two places to fix the next time the behaviour changes, and they drift. The cache key is the sharp version of this: reusing the receipt's key across two concerns couples them through the cache. A quote and a receipt are different concerns and must not answer to the same key.

Why it matters

The underlying confusion on the PR was between two things a currency value can mean: the transacting currency (what money changed hands in) and the display currency (what to render). The copied code blurred them, because at the origin they happened to coincide and the distinction never had to be drawn. Carried into a USD-only quote with no BigCommerce display currency, the copied handling was answering a question nobody asked.

This is the recurring habit the mission names: following conventions too closely. The codebase's convention was "contexts look like order_receipt_context," and matching it felt safe. Matching a convention is not the same as fitting the use. The convention is a starting point to interrogate, not a template to satisfy.

Key line Copied code fits its origin, not its destination. Annotate every pasted line with why it belongs here; if a line only made sense where it came from, cut it, and if the whole block is shared behaviour, factor it out instead of duplicating it.
The common trap The trap is reusing the cache key. Dropping a stray line of dead currency code is cheap to spot in review. Sharing a cache key across two concerns looks harmless (the code runs, the tests pass) until a quote serves a receipt's cached bytes or vice versa. A copied key is a copied identity: it silently declares two different things to be the same thing.
