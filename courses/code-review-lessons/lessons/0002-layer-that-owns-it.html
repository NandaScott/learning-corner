Code Review Lessons · Arc 1 · Lesson 2

Put logic at the layer that owns it

Lesson 1 asked whether a copied block belonged in the new use at all. This asks a sharper question about the block that survives: which layer should hold it.

The mistake

The template context for the quote was generated by a method on the model: cart.sales_quote_context(). The rendering that used that context lived at the view. So a single concern, "assemble the payload this template needs," was split across two layers. A review comment named it directly: the context was generated on the model while the rendering that used it lived at the view, so why was the context model-level?

The method also did little to earn its place on the model. A review comment noted that sales_quote_context did nothing but add to a dict. It stuffed additional_notes into a context dictionary. That is a view concern wearing a model method's clothes.

            MODEL                          VIEW
            ─────                          ────
  cart.sales_quote_context()   ┐
    build template context     │  ── the context and the render
    stuff additional_notes     │     are one job, split in two
                               ┘
                                  render(template, context)

  the caller must reach into the model to get a payload
  that only the view will ever use

One responsibility, prepare-and-render a template, with its two halves on different objects.

How to place it — ask which layer owns the responsibility

Before choosing the file for a new method, run three checks in order. Each is cheap and settles the placement.

1. Name the responsibility   what is this method actually for?
2. Find the owning layer     which layer is responsible for that kind of thing?
3. Check the co-location     does the thing that uses this output live in that layer too?

If the output is consumed at the view and produced at the model, checks 2 and 3 disagree, and that disagreement is the smell.

Step 1 — Name the responsibility, not the data

The method's job was "build the context a template needs." Name it that way and its nature is obvious: it is presentation assembly. It exists to feed a render. When a model method exists only to build a dict for a template, the name of what it does already points away from the model.

Step 2 — Match the responsibility to its owning layer

Models expose domain behaviour: what a cart is and what it can do as a business object. Views own presentation: turning domain state into a response. Template-context assembly is presentation. If rendering is a view-level concern, then generating the context that rendering consumes is too. They are the same job at two moments.

Step 3 — Check that producer and consumer sit together

The tell is spatial. The context is produced on the model and consumed at the view, and nothing else ever reads it. Output that only one layer uses should be produced in that layer. When the producer and the sole consumer live on different objects, you have leaked one concern across a boundary, and the caller now has to know both halves.

Why it matters

Splitting one concern across two layers drops cohesion and couples the model to presentation. The model now knows about templates; a change to what the quote template needs forces an edit to the model. The "where does this live?" question, asked later by the next maintainer, gets a wrong answer, because the code says the model owns quote presentation when the view does.

This is the same pull as Lesson 1, one level up. There, the pull was to match a convention line-for-line. Here it is to hang new behaviour off the nearest object that already had methods, the model, rather than the object whose responsibility it actually is. Convenience of attachment is not ownership.

Key line If rendering is a view concern, generating its context is too. A model method that exists only to build a dict for a template is a view concern in the wrong file. Place logic with the layer that owns the responsibility, not the object that was nearest.
The common trap The trap is treating "it's about the cart, so it goes on the Cart model" as the deciding test. The subject of the data is not the owner of the behaviour. A quote's context is about a cart but is for a template; ownership follows the responsibility (presentation), not the noun in the method name.

Recall — answer out loud before peeking:

① A model method does nothing but build a dict for a template. What does that signal?

show answer

That it is presentation/context assembly and belongs at the view, not the model. Models expose domain behaviour; building a template payload is a view concern.

② What is the spatial tell that a concern has been split across layers?

show answer

The output is produced in one layer and consumed only in another (context on the model, render at the view). Producer and sole consumer should sit in the same layer.

③ Why is "it's about the cart, so put it on Cart" the wrong test?

show answer

The subject of the data isn't the owner of the behaviour. Ownership follows the responsibility (here, presentation), not the noun the data describes.
