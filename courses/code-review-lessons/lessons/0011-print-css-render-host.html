Print CSS follows the render host

The quote PDF template was written to screen assumptions — pixels for spacing, fonts installed on the dev laptop. It renders on a Linux server whose font inventory and coordinate system are different. "Looks right on my machine" was unverified.

The mistake

The PDF template used px for spacing and named fonts that happened to be installed on the author's machine, with a bare sans-serif fallback. But the PDF is generated on a Linux server, not in the author's browser. That server has its own installed-font list (what fc-list reports), and print media has a physical coordinate system — inches and points — that px doesn't map onto cleanly. The design targeted the wrong environment.

The result is output that silently looks different in production: fonts substituted because the named one isn't on the server, spacing that drifts because pixels aren't a print unit, and a sans-serif fallback that resolves to whatever the render host calls its default — not what the author saw locally.

How to catch it — design against the render environment

For print/PDF output, use physical units and fonts you've confirmed exist on the rendering host. Structural spacing in in, fonts and fine spacing in pt. Pick a font that fc-list reports on the server (or ask what ships there), with a generic fallback — and don't assume sans-serif resolves the same everywhere.1

Print media has a physical coordinate system and a fixed font inventory — design to both.

Step 1 — Use physical units, because print is physical

Screen layout is measured in pixels; print output is measured in inches and points. In CSS the absolute units are anchored so that 1in = 96px = 72pt and 1pt = 1/72in.1 For a document that ends up on a physical page, express structural spacing in in and font sizes and fine spacing in pt. Those units mean the same physical thing regardless of the render host; px is a screen abstraction that only invites the renderer to reinterpret it.

Step 2 — Name a font the server actually has

A font-family is a request, not a guarantee: the renderer uses the named font only if it's installed. On a Linux render server the installed set is whatever fc-list reports — often not the Helvetica/Arial family a laptop has. Pick a font confirmed present on that host (for example Nimbus Sans, a common Linux metric-compatible substitute), and treat "it's on my machine" as irrelevant to what the server will do.

Step 3 — Don't trust the generic fallback to mean one thing

A generic family like sans-serif is a real fallback, but it resolves to whatever the render host calls its default sans-serif — which is not necessarily the one you saw locally. It stops the render from failing; it does not make the output match your screen. So a fallback covers "something readable renders," not "renders as designed." Verify the primary font exists rather than leaning on the generic to save you.

Why it matters

The render environment, not the author's browser, decides the final output — and for anything rendered server-side, "looks right on my machine" is a statement about the wrong machine. Print media has a physical coordinate system (inches and points) and a fixed font inventory that lives on the server, so a template designed against local screen defaults produces output that silently diverges in production: a substituted typeface, spacing that shifted because pixels aren't a print unit, a fallback that resolved to a stranger's default. None of it errors; it just comes out wrong on the page a customer receives. The fix is cheap and specific — confirm the render host's fonts with fc-list, use in/pt, and treat the local preview as unverified until it's checked against the server.

Key line Server-rendered print follows the render host's rules, not your screen's. Use physical units (in for structure, pt for fonts), name a font fc-list confirms on the server, and treat "looks right locally" as unverified until checked against production.
The common trap The near-miss is adding a sans-serif fallback and calling the font problem handled. The fallback only guarantees something renders — it resolves to the host's default, not the face you designed against, so the PDF still comes out looking different. Handling the font means confirming the named one exists on the render server, not backstopping it with a generic that means different things on different machines.

Recall — answer out loud before peeking:

① You're writing CSS for a PDF generated on a Linux server. Which units, and why not px?

show answer

in for structural spacing, pt for fonts and fine spacing — print media is physical (1in = 72pt), and px is a screen abstraction the renderer reinterprets.

② Your named font looks fine locally. What must you check before trusting it in the server-rendered PDF?

show answer

That the font is installed on the render host — run fc-list on the server (or ask what ships there). "It's on my laptop" says nothing about the server's inventory.

③ Does a sans-serif fallback make the output match your screen?

show answer

No. It guarantees something readable renders, but resolves to the host's default sans-serif — not the face you designed against. It prevents failure, not divergence.
