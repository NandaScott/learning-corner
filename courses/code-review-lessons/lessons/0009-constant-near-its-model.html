Code Review Lessons · Arc 5 · Lesson 9

Put a constant next to its model

A limit that governs the cart was defined in the serializer, because that's where it was first needed. Correct value, wrong home. A rule about a thing should live with the thing it constrains.

The mistake

A cart-related constant — a limit describing the Cart — was declared in the serializer module. That was the place it happened to be used first, so it landed there. But the constant isn't about serialization; it's about the cart. Anyone reasoning about what a cart may hold would open the model, not the serializer, and wouldn't find it.

The value is right and the code runs. The cost is discoverability. A rule that lives away from the concept it governs is hard to find from the concept, easy to miss, and easy to duplicate or contradict when the next person, not finding it on the model, defines their own.

declared where first used              declared where it belongs
─────────────────────────             ──────────────────────────
serializers.py                         models.py
  MAX_CART_ITEMS = 50    ← the rule       class Cart(...):
  class CartSerializer:                       MAX_ITEMS = 50   ← the rule
      ...  uses MAX_CART_ITEMS                 ...

reader asks "what limits a cart?"      reader asks "what limits a cart?"
  → opens models.py → not there          → opens models.py → right there
  → may redefine it elsewhere

Place a constant by the concept it constrains, not by the line that first reads it.

How to catch it — ask what the constant is about

When you introduce a constant, ask one question before choosing its home: what domain object does this describe? Put it on or beside that object. The point of first use is where you notice you need it — not where it belongs.

introducing a constant → ask:

   "what domain object does this value constrain / describe?"
                    │
                    ▼
        put it ON or NEXT TO that object
                    │
        (point of first use = where you noticed it,
         not where it lives)

The placement test is aboutness, not first-touch.

Step 1 — Separate "where I first needed it" from "what it's about"

The pull is to define a value at the line that first reads it, because that's the least friction in the moment. But the line that reads a value and the concept the value describes are different things. A cart limit is about the cart wherever it's first consumed. First-use is a fact about your editing session; aboutness is a fact about the domain.

Step 2 — Move the constant to its concept

Locate the constant on the model or domain object it constrains — as a class attribute on Cart, or beside the model definition. Now it's discoverable from the concept it governs: someone asking "how many items can a cart hold?" reads the cart and finds the answer, instead of grepping the codebase and hoping.

Step 3 — Recognise this as cohesion, not style

This is the high-cohesion principle: data and the rules about that data should hang together, so a change to the concept touches one place. A misplaced constant is a small cohesion leak, and cohesion leaks compound — the further a rule drifts from its concept, the more likely a second, contradicting copy appears. (Same co-location instinct as temporal coupling in Lesson 5: there, behaviour that always runs together; here, a rule that belongs with its data.)

Why it matters

High cohesion is what makes a codebase navigable: everything about one concept is in one place, so you can reason about the concept by reading one place. A constant defined at its point of first use quietly erodes that. The value is invisible from the model it constrains, which means the next person can't find it, can't trust it's the only copy, and may well write a second one that says 60 where the first says 50. Now the cart has two contradicting limits and no single source of truth. Placing the constant with its concept costs nothing at write-time and pays back every time someone reasons about that concept — which is every time the concept changes.

Key line A constant belongs with the domain object it describes, not at the line that first reads it. Ask "what is this value about?" — put it there, so anyone reasoning about the concept finds the rule with the concept.
The common trap The near-miss is defining the constant at its point of first use and calling that "declare close to usage." Declaring a variable near its use is right for a local; a domain rule's "usage" is the concept it governs, not the first line that happens to read it. First-use placement optimises your editing session; aboutness placement optimises every future reader of the concept.

Recall — answer out loud before peeking:

① You need a new limit while editing the serializer. What's the one question that decides where it goes?

show answer

"What domain object does this value describe/constrain?" It's about the cart, so it goes on/beside the Cart model — not in the serializer where you first needed it.

② Point-of-first-use placement runs fine. What does it cost?

show answer

Discoverability and cohesion: the rule is invisible from the concept it governs, so it's easy to miss and easy to duplicate or contradict. The value being correct doesn't fix that.

③ Why is "declare close to usage" not a defence here?

show answer

That rule is for locals. For a domain rule, the relevant "usage" is the concept it constrains, not the first line that reads it. A cart limit's home is the cart.
