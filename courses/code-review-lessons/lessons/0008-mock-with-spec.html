Mock a real callable with a spec

A test proved the wrong thing. It asserted the new input reached a function that no longer had to match the real one. A bare mock accepts any call, so the test stays green even after the real signature moves out from under it.

The mistake

The test for the quote view mocked render_sales_quote_html with a plain MagicMock, then asserted the mock was called with additional_notes. That reads like a real check. It isn't. A bare MagicMock overwrites the target's signature with one that accepts anything, so the assertion proves "additional_notes was passed to the thing we mocked" — not "the call matches the real function's actual signature."

The hole opens the moment the real function drifts. Rename additional_notes, reorder its parameters, drop it — and production breaks, while the test stays green, because the mock happily accepts a call the real function would reject.

bare MagicMock                         autospec'd mock
──────────────                         ───────────────
signature: accepts anything            signature: copied from the real callable

  mock(additional_notes=x)   ✓ pass      mock(additional_notes=x)   ✓ pass
  mock(anything_at_all=y)    ✓ pass      mock(anything_at_all=y)    ✗ TypeError
  mock(wrong, positional)    ✓ pass      mock(wrong, positional)    ✗ TypeError

real fn renames the arg →              real fn renames the arg →
  test still GREEN, prod BROKEN          test goes RED at the mismatched call

The green test was asserting wiring, not correctness against the real contract.

How to catch it — make the mock as faithful as the thing it stands in for

When you mock a real function or method, create the mock with a spec so it carries the real signature. In unittest.mock that means autospec=True on patch, or create_autospec(target) directly. A call that doesn't match the real signature then raises TypeError inside the test.1

# signature-blind — asserts wiring only
with patch("app.views.render_sales_quote_html") as render:
    ...
    render.assert_called_once_with(cart, additional_notes=notes)

# autospec — the mock enforces the real signature
with patch("app.views.render_sales_quote_html", autospec=True) as render:
    ...
    render.assert_called_once_with(cart, additional_notes=notes)
    # if render_sales_quote_html no longer takes additional_notes,
    # this call raises TypeError — the test catches the drift

One keyword turns a false-green into a failing test the day the real signature changes.

Step 1 — Name what the assertion actually proves

An assert_called_once_with against a bare mock proves only that your code made a particular call. It says nothing about whether that call is one the real callable would accept. Two separate facts — "my code calls it this way" and "this way is valid" — collapse into one, and only the first is checked. The mock is standing in for the real thing but keeping none of its constraints.

Step 2 — Give the mock the real signature

create_autospec(fn) builds a mock whose call signature is copied from fn; patch(..., autospec=True) does the same for the patched target. Now the two facts are checked separately: the assertion still proves your call, and the spec proves the call is valid against the real signature. Verified against the CPython docs — an autospec'd mock "will have their arguments checked and will raise a TypeError if they are called with the wrong signature."1

Step 3 — Reach for it deliberately, not as a blanket refactor

Autospec has a real cost: it inspects the target at mock-creation, and it can't see attributes set dynamically at runtime (a signature built from *args, **kwargs, or attributes assigned inside __init__ that introspection can't reach). Where a codebase already leans on hand-written mocks, introduce autospec at the mocks that stand in for real callables whose signatures you care about — not as a sweep that rewrites every mock in the suite. Default to it when mocking a real callable; fall back to a bare mock with a specific reason.

Why it matters

A mock is only as good as its fidelity to the real thing it replaces. A signature-blind mock gives you a test that passes for the wrong reason: it certifies that two pieces of your own code agree with each other, while the actual contract — does this call match the function that runs in production? — goes unchecked. That is the most expensive kind of test, because it reads as coverage and delivers none where it counts. When the real signature drifts, an honest suite goes red at the mismatch and points at the exact line. The signature-blind suite stays green and ships the break. Autospec converts signature drift from a production surprise into a test failure, which is the whole reason the test exists.

This is the testing edge of the mission's recurring habit. Following the codebase's mocking convention too closely — "we use plain MagicMock here" — inherits its blind spot. The convention is a starting point to interrogate: when the mock stands in for a real callable, matching the convention of not spec'ing it is matching a gap.

Key line A bare mock asserts your code calls it a certain way; it never asserts that way is valid against the real signature. Mock real callables with autospec=True / create_autospec so a mismatched call raises TypeError in the test instead of breaking silently in production.
The common trap The near-miss is trusting a green assert_called_once_with as proof the call is correct. It only proves the call happened the way you wrote it — and you wrote both sides. Against a signature-blind mock, "called with additional_notes" survives even after the real function drops additional_notes. Fidelity to the real signature is the missing half.
