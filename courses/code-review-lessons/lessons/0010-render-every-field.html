Render every field the object carries

The address block in the quote left out the phone number. The address object carries it and the document needs it. The template rendered a remembered address, not the real one.

The mistake

The quote PDF rendered an address block — street, city, region, postcode — and dropped the phone number. Not because the address lacked one: the address object carries a phone field, and the document is a customer-facing quote that wants it. The template rendered a mental picture of "what an address looks like" instead of the fields the address object actually has.

This is the failure mode where the code works, the page renders, and the output is quietly incomplete. Nothing errors. A field the consumer expects is simply absent, and the only way to notice is to compare the output against the object's real shape — which is exactly the step that got skipped.

rendered from memory                   rendered from the model
────────────────────                   ───────────────────────
Address:                               Address:
  name                                   name
  street                                 street
  city, region  postcode                 city, region  postcode
  ─ (phone omitted)                       phone   ← carried, and needed

output renders fine, silently          output matches the object's
missing a field the doc needs          real field set

A partial render from memory drops fields the consumer expects, with no error to flag it.

How to catch it — enumerate the object's real fields

Before rendering a composite domain object — address, contact, line item — list its fields from the model, then decide which the output requires. Render from that list, not from a remembered shape. Treat every field as needed until you've shown it isn't.

rendering a composite object into a template:

   1. open the model → enumerate its ACTUAL fields
        name · street · city · region · postcode · phone · email
   2. for each field: does THIS document need it?
        phone → yes (customer-facing quote)
        internal_id → no
   3. render the required set — default a field to "needed"
      until proven irrelevant

Output completeness is a spec concern: the object's field list is the checklist.

Step 1 — Get the field list from the object, not your head

Open the model (or serializer, or dataclass) and read what the object actually holds. A rendered-from-memory address is a guess about the object's shape, and the guess tends to include the fields you always picture and omit the ones you don't — like a phone number on an address. The object's own field list is the authoritative shape; anything else is a partial recollection.

Step 2 — Decide each field in or out, on purpose

Walk the enumerated fields and make an explicit call for each: does this document need it? Default a field to "needed until proven irrelevant." That default matters — the failure here wasn't deciding phone was unnecessary, it was never considering phone at all. An omission you decided is fine; an omission you never saw is the bug.

Step 3 — Treat the omission as a spec miss, not a style nit

A missing field isn't a formatting preference; it's the output not meeting what the consumer expects. The quote is supposed to carry the customer's contact details, and a quote without the phone number fails that quietly. Rendering completeness is part of the spec the document has to satisfy, and it's checked by comparing render against real shape.

Why it matters

Rendering from a mental model instead of the real object is the same root fault as coding against a convention or an imagined future: you're designing against something other than the actual thing in front of you. Here the actual thing is the domain object's field set, and the mind's version of it is lossy in a predictable direction — it keeps the fields you always think of and drops the ones you don't. The result passes every check that asks "does the code run?" and fails the only check that matters: "does the output carry what the object holds and the reader needs?" That check has to be run deliberately, against the model's real fields, because nothing else will surface it until a customer reads a quote with no way to phone them back.

Key line Render a composite object from its real field list, not a remembered shape. Enumerate the model's fields, decide each in or out on purpose, and default a field to "needed" until proven irrelevant — a field you never considered is the one that goes missing.
The common trap The near-miss is rendering the fields that "obviously" make up an address and calling it done. The obvious fields are exactly the ones memory keeps; the dropped phone number is the one it doesn't. "Looks like a complete address" is a judgement from memory. "Matches the address object's fields" is a judgement from the model — only the second catches the gap.
