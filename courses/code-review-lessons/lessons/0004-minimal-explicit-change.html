Code Review Lessons · Arc 2 · Lesson 4

Prefer the minimal, explicit change

Four catches with one shape: the change was bigger, cleverer, or more hand-built than the problem required. Each smaller alternative was already available.

The mistake

Four moves on the PR reached past the smallest change that would work:

  • An entire validation function was rewritten, when pulling ['.html', '.txt'] out and optionally appending would have done the job.
  • A field→validator map was hand-built (and defined in two places) where Django already hands you the validators.
  • A SimpleNamespace was used where a plain dict fit.
  • A single brittle validator branched internally where two explicit, named validators would read clearer.

None is a bug. Each expands the diff, the review surface, and the number of things that can drift, in exchange for nothing the problem asked for.

How to catch it — three questions before you restructure

1. Smallest-diff question   what is the smallest change that solves this?
                            justify any rewrite bigger than that.
2. Framework question       am I building a lookup the framework already gives me?
                            (Model._meta.get_field(...), field.validators, ...)
3. Explicit question        would two named things read clearer than one that branches?
                            prefer explicit over clever.

Run them before restructuring, not after. The default is the smallest change; anything larger needs a reason.

Step 1 — Ask for the smallest diff that works

The validation function did not need rewriting. It needed one list pulled out and optionally extended. A larger-than-necessary rewrite hides the actual change inside churn, so a reviewer cannot see what really moved, and it takes on risk the task never required. Before restructuring anything, state the smallest change that solves the problem, and justify any rewrite bigger than that against it.

Step 2 — Use the framework's own accessor

The hand-built field→validator map is the sharp one, because Django already holds that mapping. A model field's validators are reachable directly:

# hand-rolled, and defined in two places → drifts
FIELD_VALIDATORS = {
    "document": [FileExtensionValidator(["pdf"])],
    ...
}

# the framework already knows — one source, no map to maintain
field = MyModel._meta.get_field("document")
field.validators        # == [*default_validators, *_validators]

Verified in Django 6.0: Options.get_field(name) returns the field; Field.validators is [*default_validators, *_validators]: the validators you attached, already collected.

A duplicated definition is two things to keep in sync, and they will not stay in sync. When you catch yourself building a lookup table the framework can already produce, stop and use the accessor. The framework is the single source of truth for its own metadata.

Step 3 — Prefer explicit over clever

Two of the catches are the same instinct in different clothes. SimpleNamespace over dict reached for a richer type where the plain one carried the data fine. One branching validator over two named ones folded two ideas into one that has to decide which it is at runtime. Two explicit validators say what they each check in their names; one that branches makes the reader trace the branch. Explicit is better than implicit, and it is the first line of the language's own guidance.

Why it matters

This is the recurring habit the mission calls out from the other side: following conventions too closely also means reaching for the impressive-looking construct because it looks like what "real" code does. The smallest change looks too small to be the answer. It usually is the answer. A reviewer can hold a small, explicit diff in their head and verify it; a rewrite, a hand-built map, and a clever type each cost them the effort of proving nothing else changed.

Note the boundary against Lesson 6, which is coming: that lesson is about not building for requirements that do not exist yet. This one is about the size and clarity of the change in front of you. They rhyme but are distinct: one is diff size and legibility, the other is design horizon.

Key line Make the smallest change that solves the problem, and justify any rewrite bigger than that. If you're building a lookup the framework already gives you (_meta.get_field().validators), stop and use it. Prefer two explicit named things over one that branches.
The common trap The near-miss is "explicit means more code, and more code is safer to review." Not here. Two named validators are more lines than one that branches, but each states its single check in its name, so the reader verifies them at a glance. The hand-built map is also "more code" and is worse, because it duplicates what the framework owns. Explicit is about legibility per unit, not volume.

Recall — answer out loud before peeking:

① What's the default size of a change, and what needs justifying?

show answer

The default is the smallest change that solves the problem. Any rewrite bigger than that needs a reason justified against the smallest version.

② You're about to hand-build a {field: validators} map on a Django model. What stops you?

show answer

The framework already holds it: Model._meta.get_field(name).validators. Hand-building duplicates framework metadata and the copies drift. Use the accessor.

③ One validator that branches, or two named validators: which, and why?

show answer

Two named ones. Each states its single check in its name; the branching one hides two ideas behind a runtime decision. Explicit over clever.
