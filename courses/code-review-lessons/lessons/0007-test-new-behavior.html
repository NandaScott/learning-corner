Code Review Lessons · Arc 4 · Lesson 7

Test the new behaviour, even when neighbours lack tests

The catch that closes the loop. New behaviour shipped without coverage, on the reasoning that the adjacent legacy code had none either. The tests went in afterward, in a later commit.

The mistake

The new sales_quote view action and the additional_notes path had no tests. Only the model helper was exercised at all. The reasoning, stated plainly at the time, was: "there are no tests for order_receipt either, so I didn't write any." An automated check flagged the missing view test; a reviewer noted the helper was untested and could not confirm the notes even reached the context. The tests were added in a later commit.

The gap that matters is the last one. Without a test, there was no proof that additional_notes, the actual new input, made it through to the output. The contract was asserted by nobody.

"order_receipt has no tests, so I skipped tests for sales_quote"

        propagates the gap  ──────────────►  instead of closing it

what went unproven:
   • feature gating works
   • auth / ownership is enforced
   • missing-case handling
   • response content type is right
   • additional_notes actually reaches the output   ← the new input, unverified

"The old code isn't tested" is a description of a gap, not a licence to add another.

How to catch it — cover the observable contract

When you add an endpoint or action, write action-level tests as part of the same PR, whatever the neighbours have. Cover what an outside caller can observe.

the observable contract of a new action:

  1. Feature gating     off when the flag/condition says off
  2. Auth / ownership   the wrong user can't reach it
  3. Missing cases      absent/invalid input fails as intended
  4. Response shape     correct status and content type
  5. Input → output     the new input (additional_notes) reaches the result

Point 5 is the one the review specifically could not confirm. It's the reason the test exists.

Step 1 — Test the contract, not the implementation

Action-level tests assert what a caller observes: given this request, this response. Feature gating, auth and ownership, missing and invalid cases, the response content type. These are the promises the endpoint makes to the outside, and they are what a test is for.

Step 2 — Assert the input reaches the output

The specific catch on the PR was that nobody could confirm additional_notes reached the context. That is the single most important assertion for this change, because it is the change: pass notes in, prove they appear in the rendered output. Untested new behaviour means no proof the contract holds, and this contract was the whole point of the endpoint.

Step 3 — Treat "nothing to copy" as a reason to write the first test

The reasoning to disarm is "there's no existing test to copy from." The Boy Scout rule cuts the other way: leave the code cleaner than you found it. Absence of tests on adjacent legacy code is not permission to skip; it is the reason to write the first one. Copying conventions too closely includes copying the convention of not testing, and that convention should not be matched.

Why it matters

"The old code isn't tested" propagates a gap forward instead of closing it. Every new untested action makes the next "well, this one isn't tested either" easier to justify, and the coverage hole grows by inheritance. More concretely: without the test, the endpoint might not do the one thing it was built to do, and no one would know until a customer saw a quote with the notes missing. The reviewer said it exactly: they could not confirm the notes reached the context. The test is what converts "probably works" into "provably works."

This is the last face of the mission's recurring habit. Following conventions too closely, here, meant inheriting the neighbour's missing tests. The counter-move is the same as it has been all course: the convention is a thing to interrogate, not a rule to satisfy. When the convention is "we didn't test this," you break it.

Key line Absence of tests on adjacent legacy code is not a licence to skip. Cover the new behaviour's observable contract (gating, auth, missing cases, response shape, and that the input actually reaches the output) in the same PR, and treat "nothing to copy" as the reason to write the first test.
The common trap The near-miss is testing that the helper runs, and calling it covered. In the review, the model helper was the one thing exercised, and it still left the real question open, because "the helper returns a dict" doesn't prove the notes reach the rendered response. Test the observable contract end to end, not the internal function that happens to be easiest to call.
