Code Review Lessons · Reference

Glossary

The shared vocabulary for this course. Every lesson uses these terms exactly as defined here. Print it; keep it next to the diff.

TermDefinition
Copied code fits its origin, not its destinationA pasted block was written to satisfy the context it came from. Assumptions true there (multi-currency, a shared cache key) may be false where it's pasted. (Lesson 1)
Annotate-before-pasteThe habit of writing "why does this belong here?" against each copied line, then cutting any line whose only answer is "it was in the original." (Lesson 1)
Single source of truthEach piece of behaviour has one authoritative definition. If a block is genuinely shared, factor it out and call it from both sites rather than copying. Copies drift. (Lesson 1, DRY)
Transacting vs display currencyThe currency money changed hands in vs the currency to render. In the review they were conflated; carried into a USD-only quote with no BigCommerce display currency, the copied currency handling was dead logic. (Lesson 1)
Shared cache keyReusing one cache key across two concerns couples them through the cache: one path can serve the other's cached bytes. A copied key is a copied identity. (Lesson 1)
The layer that owns itThe layer responsible for a kind of responsibility. Domain behaviour → model. Presentation / template-context assembly → view. Place logic by responsibility, not by which object was nearest. (Lesson 2)
Split concern (spatial tell)One responsibility whose halves live on different objects. The tell: output produced in one layer and consumed only in another (context on the model, render at the view). (Lesson 2)
Subject ≠ ownerThe noun a method's data is about (the cart) is not the layer that owns the behaviour (presentation). "It's about the cart, so it goes on Cart" is the wrong placement test. (Lesson 2)
Intention-revealing nameA name that states what the thing holds or does without a comment. If an inline comment names a variable better than its identifier, the comment is the rename. (Lesson 3)
Scope-change passWhen a change broadens what a function covers, re-read its name and docstring in that same change and update both, as the last edit before commit. (Lesson 3)
Stale docstringA docstring accurate when written but not since. It misinforms worse than none because the reader trusts it, and it hides behind its former truth. (Lesson 3)
Smallest-diff question"What is the smallest change that solves this?" The default change size; any rewrite larger than that must be justified against it. (Lesson 4)
Framework accessorThe framework's own way to retrieve metadata it already holds, used instead of a hand-built duplicate. Django: Model._meta.get_field(name).validators. (Lesson 4)
Field.validatorsDjango field property = [*default_validators, *_validators]: the validators you attached, already collected. Verified in Django 6.0. Removes any need to hand-build a field→validator map. (Lesson 4)
Explicit over cleverTwo named validators that each state one check beat one that branches at runtime; a plain dict beats a SimpleNamespace where the data is plain. Legibility per unit, not volume. (Lesson 4, PEP 20)
Temporal couplingTwo functions bound by "A must run before B, every time," where nothing enforces the order: the contract lives only in each caller's head. (Lesson 5)
Reverse-dependency examinationReading a function from its callers: who calls it and what they call right before and after. How coupling (a property of use, not of the body) is found. (Lesson 5)
Universal invariant (merge test)Merge A and B only if every caller runs A then B with nothing between. If any caller runs A alone, they're genuinely separate. (Lesson 5)
Speculative generalityStructure (a param, hook, layer, abstraction) added for a requirement that doesn't exist yet. Untested, usually the wrong shape when the real need lands. (Lesson 6)
YAGNIYou Aren't Gonna Need It. Add abstraction when a second real use appears, not in anticipation. The deciding question: does any current caller need this? (Lesson 6)
Cheap to type, expensive to carryThe real economics of a speculative hook: trivial to add, but untested, wrong-shaped, and in everyone's way until proven needed. The cheap moment to add it is when a real second use reveals its shape. (Lesson 6)
Observable contractWhat an outside caller can see of a new action: feature gating, auth/ownership, missing/invalid cases, response status and content type, and that the input reaches the output. What action-level tests assert. (Lesson 7)
Input-reaches-output assertionThe test proving the new input (e.g. additional_notes) actually appears in the rendered response: the one thing the PR's review couldn't confirm. (Lesson 7)
Boy Scout ruleLeave the code cleaner than you found it. Absent tests on adjacent legacy code are the reason to write the first test, not permission to skip. (Lesson 7)
Signature-blind mockA bare MagicMock standing in for a real callable: it accepts any arguments, so an assertion against it proves your code made a call, not that the call matches the real signature. Gives a false-green that survives signature drift. (Lesson 8)
Autospec / create_autospecBuilding a mock whose call signature is copied from the real callable (patch(..., autospec=True) or create_autospec(fn)), so a wrong-signature call raises TypeError in the test. Verified against the CPython unittest.mock docs. Introduce it deliberately, not as a blanket refactor. (Lesson 8)
Aboutness vs first-use placementA constant's home is the domain object it describes (aboutness), not the line that first reads it (first-use). First-use optimises the editing session; aboutness optimises every future reader of the concept. (Lesson 9)
Render from the model, not memoryEnumerate a composite object's real fields from its definition before rendering, and default each field to "needed until proven irrelevant." Memory is lossy in a predictable direction — it keeps the usual fields and drops the rest (a phone on an address). (Lesson 10)
Output completenessA spec concern, not a style nit: the output carries every field the domain object holds and the consumer needs. Checked by comparing the render against the object's real field set, since no error flags a silently missing field. (Lesson 10)
Render hostThe environment that actually produces server-side output (e.g. the Linux box generating a PDF), whose font inventory and coordinate system — not the author's browser — decide the final result. "Looks right on my machine" is unverified for it. (Lesson 11)
Physical units (in / pt)Absolute CSS length units for print: in for structural spacing, pt for fonts and fine spacing, with 1in = 96px = 72pt. They mean the same physical size on any render host; px is a screen abstraction the renderer reinterprets. (Lesson 11)
fc-listThe fontconfig command that reports the fonts installed on a host. Run it on the render server to confirm a font-family exists there (e.g. Nimbus Sans) before trusting it; a generic sans-serif fallback resolves to the host's default, not your designed face. (Lesson 11)
Requirements checklistAn explicit, enumerated list of the ticket's requirements (description + acceptance criteria + linked spec), built before the first draft and walked against the diff before the PR. Gives an omission something concrete to fail against. (Lesson 12)
Spec gate vs quality gateTwo distinct pre-review checks: the spec gate asks "did I build everything asked?" (checklist vs diff); the quality gate asks "is what I built good?" (code review). Review answers the second and structurally cannot answer the first. (Lesson 12)
Following conventions too closelyThe mission's recurring habit: matching what the codebase already does (a context shape, a missing test, an impressive construct, a bare mock) instead of interrogating whether it fits. A convention is a starting point to question, not a rule to satisfy. (Lessons 1, 4, 7, 8)
