Pre-PR self-review checklist

The compressed payoff of the twelve lessons: the checks to run on your own diff before it goes up. Each is phrased as the question to ask, the tell that answers it, and the move. Print it; run it cold on the next PR.

#AskTell → move
1 Did I paste any existing block? For each copied line: why does this belong here? A line's only answer is "it was in the original" → cut it. The whole block is shared behaviour → factor to one source, don't copy. Never reuse a cache key across two concerns. L1
2 For each new method: which layer owns this responsibility? The output is produced in one layer and consumed only in another (context on the model, render at the view) → move it to the consuming layer. A model method that only builds a template dict belongs at the view. L2
3 Do my names reveal intent? Did any function's scope broaden this change? An inline comment out-names the variable → rename to the comment, delete it. Scope broadened → re-read name and docstring, update both, as the last edit before commit. L3
4 Is this the smallest change that works? Am I hand-rolling what the framework holds? A rewrite bigger than the change needs → justify it or shrink it. Hand-built {field: validators} map → use Model._meta.get_field(name).validators. Prefer two named validators over one that branches; a dict over SimpleNamespace. L4
5 Did I split a function? Read its callers: does B always run right after A? Every caller runs A then B, nothing between → merge them (fold B into A, or one call that runs both). If any caller runs A alone, leave them separate. Use reverse-dependency examination. L5
6 Did I add a param / hook / layer "for later"? Does any current caller need it? No real caller uses it today → cut it. Add the abstraction when a second concrete use lands and reveals its shape. Minimal ≠ non-speculative: a small addition for a nonexistent caller still goes. L6
7 Did I test the new behaviour's observable contract, regardless of neighbour coverage? Cover gating, auth/ownership, missing cases, response status/content type, and that the input reaches the output. "No adjacent test to copy" → write the first one. Testing the helper ≠ testing the contract. L7
8 Did I mock a real function/method? Does the mock carry its real signature? Bare MagicMock standing in for a real callable → use autospec=True / create_autospec so a wrong-signature call raises TypeError. A green assert_called_once_with proves your call, not that it's valid. Introduce autospec deliberately, not as a sweep. L8
9 Did I add a constant? What domain object is it about? Defined at its point of first use (e.g. in the serializer) → move it onto/beside the model it constrains, so a reader finds the rule with the concept. First-use placement ≠ "declare close to usage" for a domain rule. L9
10 Rendering a composite object? Did I enumerate its real fields, not a remembered shape? Rendered from memory → open the model, list its actual fields, decide each in/out on purpose, default to "needed until proven irrelevant." A silently missing field (the phone on an address) is a spec miss, not a style nit. L10
11 Is this output rendered server-side (PDF/print)? Did I design to the render host, not my screen? Screen assumptions (px, locally-installed fonts) → use physical units (in for structure, pt for fonts) and a font fc-list confirms on the server. A sans-serif fallback prevents failure, not divergence. "Looks right locally" = unverified. L11
12 Before the first draft: did I enumerate the ticket's requirements? Before the PR: did I walk them against the diff? No enumerated checklist → an omission has nothing to fail against and survives every review. Build the checkbox list from the ticket first, keep it visible, and run the spec gate ("built everything asked?") separately from the quality gate ("is it good?"). L12

The lines under all twelve

Key line Two threads run under the twelve. A convention is a starting point to interrogate, not a rule to satisfy — most catches came from matching what was already there (a context shape, a missing test, an impressive construct, an imagined future, a bare mock) instead of asking whether it fit. And enumerate from the real source, not your memory — the model's fields, the render host's fonts, the ticket's requirements. Both are the same move: design against the actual thing in front of you, not a remembered or convenient stand-in for it.
