How BigCommerce Works · Lesson 8

The order object

Lesson 1 minted the order at the end of the pipeline and left it as incomplete. This lesson is the whole object: its complete status lifecycle, everything it carries, the two ways it can be born, and how a consignment becomes a tracked shipment. Everything the earlier lessons built — catalog, line items, pricing, discounts, consignments — comes to rest here.

The concept

The order is the durable record: the source of truth for money and fulfillment once a purchase is committed. Unlike the cart, it is meant to persist and to read the same later as it did the day it was placed. And because fulfilling it takes time — payment confirms, items ship, refunds happen — an order is not static. It moves through a lifecycle, and a status field is how the whole business reads where it is.

How BigCommerce handles it

An order carries a copied-in snapshot of the purchase plus a set of sub-resources, and it advances through a fixed list of statuses. Start with the lifecycle, because status is how everything else coordinates.

① The status lifecycle

Every order sits at exactly one of fifteen statuses, each with a fixed numeric status_id and a string status.1 (Merchants can rename the shopper-facing label via custom_label, but the id and system_label are fixed.1)

The happy path threads a few of these, and the split between automatic and manual transitions is the thing to hold:

   checkout converts                 payment confirmed              seller ships items
        │                                  │                              │
        ▼          (auto)                  ▼        (auto)                ▼   (seller action)
   Incomplete/Pending ──▶ Awaiting Payment ──▶ Awaiting Fulfillment ──┬─▶ Partially Shipped ─▶ Shipped
      (0 / 1)                  (7)                    (11)             │        (3)                (2)
                                                                      └─▶ Completed (10, digital)

   seller actions branch off at any point:  Refunded(4) · Partially Refunded(14)
                                            Cancelled(5) · Declined(6) · Manual Verification(12)

   ⚠ status does NOT auto-revert: "deleting a shipment does not move the order out of shipped status."

Payment transitions are automatic (7 → 11). Fulfillment and refunds are seller actions. Nothing rewinds on its own.

② What the order carries

An order bundles the committed purchase. Its own fields include identity and origin (cart_id back to the cart, channel_id, order_source), a full set of totals held as strings (subtotal_ex_tax/subtotal_inc_tax, total_ex_tax/total_inc_tax, shipping_cost_*, coupon_discount, discount_amount, refunded_amount), payment state (payment_method, payment_status), and a copied-in billing_address — the snapshot from Lesson 3.2 The rest hangs off as sub-resources, each returned as a {url, resource} pointer you expand: products (the line items), shipping_addresses, consignments, coupons, and fees.2 Payment transactions and refunds live on separate v3 endpoints.3

③ Two ways an order is born

The pipeline path — a checkout converting — is one way. The other is creating an order directly, with no prior cart: the manual or phone-order path via POST /v2/orders.4 It requires a billing_address plus either products or consignments, accepts catalog products or custom ones, and takes customer_id: 0 for a guest.4 One consequential difference: "the V2 Orders API will not trigger the typical Order Email when creating orders" — to send it, "you can instead create a cart and convert that cart into an order."5 Orders are also editable after creation, via PUT.4

④ Consignments become shipments

Lesson 7's consignment was the plan; the shipment is the act. "Once an order has products, a billing address, and a shipping address, you can create an order shipment"5 — a record with a tracking_number, tracking_carrier, shipping_method, and an items list of what went in it. Multiple partial shipments are allowed, and each drives the status from Awaiting Fulfillment toward Partially Shipped and then Shipped. The consignment named the intended grouping; the shipment records the real dispatch and its tracking.

Where it surfaces in the API

Orders are a v2 API; transactions and refunds are surfaced on v3.3

  • Order — GET/POST/PUT /v2/orders/{id}; sub-resources at /products, /shipping_addresses, /consignments, /coupons, /fees.2
  • Statuses — /v2/order_statuses lists the fixed table (id, name, custom_label).1
  • Shipments — /v2/orders/{id}/shipments.5
  • Transactions / refunds — GET /v3/orders/{id}/transactions and the v3 refund endpoints.3

The pipeline, whole

Step back and the eight lessons are one object graph. A product and its variants (L2) define what can be bought; a line item (L3) freezes a chosen variant plus its price (L4) into a cart; discounts (L5) and the promotions engine (L6) adjust the cart total; consignments (L7) plan how it ships; and conversion mints the order (L1, L8) — the durable record all of that comes to rest in. The order's cart_id points back to where it started, its products are the frozen line items, its consignments are the fulfillment plan turning into shipments, and its status is how payment and fulfillment stay coordinated over time. The cart was disposable; the order is the truth.

Key line "An order is the durable record with a fifteen-status lifecycle: payment transitions are automatic (Awaiting Payment → Awaiting Fulfillment on confirmation), fulfillment and refunds are seller actions, and nothing auto-reverts. It carries totals, payment state, and a snapshot billing address on itself, with products, consignments, coupons, transactions, and shipments as sub-resources — and it can be born either from a converting checkout or created directly for phone orders."
The trap Assuming order creation and status behave like the storefront. Two real gotchas: first, an order created through the V2 API does not send the customer's order-confirmation email — if you need it, create a cart and convert it instead; and the default status_id on API create is documented inconsistently, so always set it explicitly. Second, status is not fully automatic or reversible — fulfillment and refunds are manual seller actions, and undoing one doesn't rewind the status ("deleting a shipment does not move the order out of shipped status"). And keep consignment (the plan) distinct from shipment (the tracked dispatch).

Recall — answer out loud before peeking:

① Which order-status transition is automatic on payment confirmation, and what are the two statuses?

show answer

Awaiting Payment (7) → Awaiting Fulfillment (11), automatically once payment is confirmed.

② You create an order via the V2 API. Does the customer get the order-confirmation email? How do you make sure they do?

show answer

No — V2 order creation doesn't trigger the order email. To send it, create a cart and convert that cart into an order instead.

③ What's the difference between a consignment and a shipment on an order?

show answer

The consignment is the intended grouping (items + destination + fulfillment method, planned at checkout); the shipment is the actual dispatch record with a tracking number. Partial shipments advance the status toward Shipped.

④ An order's products, consignments, and coupons aren't inline objects in the order response. How are they returned?

show answer

As sub-resource pointers ({url, resource}) that you expand — via a separate request or an include param. The order's own fields are identity, totals, payment, and the copied billing address.
