How BigCommerce Works · Lesson 6

The promotions rule engine

Lesson 5 placed automatic and coupon promotions on the map and said the discount logic "lives in the promotion." This lesson opens that box. A BigCommerce promotion is a small rule engine — an ordered list of rules, each an action guarded by a condition — plus a set of controls for how promotions stack. Learn the vocabulary and every real-world offer becomes expressible.

The concept

Every promotion is fundamentally "if the cart looks like this, do that." To express real offers — spend $50 for free shipping, buy one get one free, 20% off the cheapest item, wholesale gets a fixed price — you need two small languages: one for describing when an offer applies (the condition) and one for describing what it does (the action). And once a store runs several promotions at once, you need rules for which ones combine and which one wins.

How BigCommerce handles it

A promotion holds an ordered list of rules. The documentation is precise about how they run: the rules field is "an ordered list of rules to be executed until the first applicable one applies a discount successfully and the rest will be skipped."1 Each rule is one action plus an optional condition:

  PROMOTION
   ├─ redemption_type: AUTOMATIC | COUPON      (Lesson 5 — gates the whole thing)
   ├─ rules: [                                 ordered; FIRST applicable rule wins,
   │    {                                        the rest are SKIPPED
   │      condition: { … }   ── WHEN            cart / and / or / not
   │      action:    { … }   ── WHAT            cart_value | cart_items | gift_item
   │      apply_once, stop                                | shipping | fixed_price_set
   │    }, … ]
   ├─ can_be_used_with_other_promotions        ]  how it STACKS with
   ├─ stop                                      ]  OTHER promotions
   └─ status, max_uses, start/end_date, channels[], customer{group_ids}

              condition matches?  ──yes──▶  apply action  ──▶  discount lands in
                                                                cart discounts[] / coupons[]
                                                                (Lesson 5)

A promotion is a tiny rule DSL: ordered rules, each a guarded action, wrapped in stacking controls.

① Rules run in order; the first that applies wins

This is the mechanical heart. The rules array is evaluated top to bottom, and evaluation stops at "the first applicable one" that "applies a discount successfully."1 The rest are skipped. So a promotion's rule list is not "apply everything that matches" — it is "find the first match and stop." A rule with no condition applies unconditionally, which is how a storewide offer is written.

② Conditions — describing when

A condition is built from one of four types: a cart condition that "uses the cart contents," and the logical wrappers and, or, and not.2 A cart condition can require a minimum_spend or a minimum_quantity, and can target specific items through a matcher that accepts products, variants, brands, categories, a product_option, or a product_custom_field.2 The boolean wrappers let you compose: "brand X and at least 2 items," "category A or category B."

③ Actions — describing what

There are five action types, each a differently-shaped discount.2 cart_value discounts the whole cart. cart_items discounts eligible line items and carries the sharp options — a strategy of LEAST_EXPENSIVE or MOST_EXPENSIVE, a quantity, and its own items matcher (this is how "20% off your cheapest eligible item" is written). gift_item adds a free product or variant. shipping grants free_shipping over given zone_ids. fixed_price_set forces a fixed price for a set of items. The discount amount itself is expressed as a fixed_amount or a percentage_amount.2

④ A worked example — buy one, get one free

BOGO is a condition plus a gift_item action: the condition requires at least one of the product in the cart (minimum_quantity: 1 with an items matcher), and the action gives one free (gift_item with quantity: 1).2 Set the rule's apply_once to false and it repeats — buy two, get two. That one pattern, with different conditions and actions, covers most of the offer catalog a store will ever need.

⑤ Stacking and priority — when many promotions run

Three separate controls decide how promotions combine. A rule-level stop: "if you set stop to true on a rule, and it is applied successfully, you will end the evaluation."2 A promotion-level stop, which "determines if you want to calculate any other promotions after this one."2 And can_be_used_with_other_promotions, which "when set to false, only apply this promotion if there are no discounts applied already, and stop executing other promotions if this promotion applies successfully."2 Priority is documented as an ordering concept — "a lower number means higher priority"2 — though it does not appear as a field in the JSON examples, so treat rule order and the stop flags as the reliable levers.

Where it surfaces in the API

  • Promotions — POST/GET /v3/promotions; the rules, redemption_type, status, max_uses, customer, and stacking fields all live on this object.1
  • Coupon codes — POST /v3/promotions/{promotion_id}/codes for a COUPON promotion (Lesson 5).2
  • Result — the resolved discount appears on the cart's line-item discounts[] and cart-level coupons[] (Lesson 5).2
// A COUPON promotion: one rule, condition + cart_items action, exclusive stacking
// POST /v3/promotions
{
  "name": "Planeswalker Spark Sale",
  "redemption_type": "COUPON",
  "can_be_used_with_other_promotions": false,   // exclusive: don't stack
  "currency_code": "USD",
  "status": "ENABLED",
  "max_uses": 500,
  "rules": [
    {
      "condition": {
        "cart": { "items": { "products": [129] }, "minimum_quantity": 2 }   // WHEN
      },
      "action": {
        "cart_items": {                                                     // WHAT
          "discount": { "percentage_amount": "20" },
          "strategy": "LEAST_EXPENSIVE",     // 20% off the cheapest eligible item
          "quantity": 1,
          "items": { "products": [130] }
        }
      },
      "apply_once": true,
      "stop": true                            // first success ends evaluation
    }
  ]
}

Condition + action inside one rule. strategy: LEAST_EXPENSIVE is what makes "off your cheapest item" expressible.

// BUY ONE GET ONE FREE — an automatic gift_item promotion
{
  "name": "Buy one and get one free",
  "redemption_type": "AUTOMATIC",
  "rules": [
    {
      "condition": { "cart": { "items": { "products": [130] }, "minimum_quantity": 1 } },
      "action":    { "gift_item": { "product_id": 130, "quantity": 1 } },
      "apply_once": false,       // repeats: buy 2 get 2, buy 3 get 3
      "stop": false
    }
  ],
  "status": "ENABLED"
}

BOGO = a minimum_quantity condition and a gift_item action. apply_once:false makes it repeat.

Why a rule engine

The condition/action split is what lets one object model an open-ended catalog of offers without a special case per offer. Every promotion a store will run — tiered spend thresholds, brand-targeted deals, BOGO, free shipping to some zones, wholesale fixed prices — is the same shape, just different conditions and actions. The ordered-rules-first-match rule keeps a single promotion's outcome unambiguous, and the three stop/stacking controls extend that determinism across many promotions at once. And because redemption_type sits outside the rules, the exact same engine serves both an always-on automatic sale and a code-gated coupon — which is precisely the unification Lesson 5 pointed at. The engine decides the number; the cart's discounts[] and coupons[] report it.

Key line "A promotion is an ordered list of rules, each an action (cart_value / cart_items / gift_item / shipping / fixed_price_set) guarded by a condition (cart with minimum_spend/minimum_quantity/item matchers, composed with and/or/not). Rules run top-down and the first that applies wins; three controls — rule stop, promotion stop, and can_be_used_with_other_promotions — govern stacking across promotions."
The trap Expecting every matching rule in a promotion to apply. It doesn't — the rules array runs until the first applicable rule applies a discount successfully, then the rest are skipped. If you want two effects, that's two rules where the first doesn't stop, or two promotions. The related half: stacking across promotions is governed by three distinct flags (rule-level stop, promotion-level stop, and can_be_used_with_other_promotions), not one — and priority, though documented as "lower = higher," isn't a reliable JSON field, so lean on rule order and the stop flags.

Recall — answer out loud before peeking:

① A promotion has three rules and the cart matches all three. How many apply, and which?

show answer

One — the first applicable rule that applies a discount successfully. The rest are skipped. Rules are first-match, not apply-all.

② Name the two halves of a rule, and the four condition types.

show answer

A rule = a condition (when) + an action (what). Condition types: cart, and the boolean wrappers and, or, not.

③ Which action, with which option, expresses "20% off your single cheapest eligible item"?

show answer

cart_items with strategy: LEAST_EXPENSIVE, quantity: 1, a percentage_amount discount, and an items matcher for eligibility.

④ Name the three controls that govern how a promotion stacks with others.

show answer

Rule-level stop, promotion-level stop, and can_be_used_with_other_promotions (false = exclusive).
