How BigCommerce Works · Lesson 1

The cart → checkout → order pipeline

Before any single object makes sense, you need the map. In BigCommerce a purchase is not one thing that changes state — it is three distinct objects handed off in sequence: a cart, a checkout, and an order. This lesson walks the whole pipeline, so every later lesson has a place to sit.

The concept

Every ecommerce system moves a purchase through the same three stages, whatever it calls them. First a container holds what the shopper wants to buy. Then the shopper commits: they say who they are, where it ships, and how they will pay. Finally the system records what was agreed, so it can be fulfilled and accounted for. Container, commitment, record.

Some platforms model this as one object with a status that advances (a single "order" that starts as a draft). BigCommerce does the opposite: it uses three separate objects, each owning one stage, with an explicit handoff between them. Knowing where one ends and the next begins is most of the mental model.

How BigCommerce handles it

The three objects are the cart, the checkout, and the order. They describe the same purchase at three points in its life, and they even share an identity — the checkout is addressed by the same ID as the cart it grew from.2

   ①  CART                    ②  CHECKOUT                     ③  ORDER
   ─────────                  ────────────                    ─────────
   line items                 cart + billing address          the durable record
   prices, discounts          + consignment(s):               status: incomplete
   NO customer data                address                          │
        │                          line items                      │ payment
        │  add a                    shipping option                 ▼
        │  billing address     ────────────────────           Awaiting Fulfillment
        └──────────────▶  every physical consignment  ──POST──▶
                          needs a selected shipping         Create
                          option before it can convert      Order

One purchase, three objects. The cart is disposable scratch space; the order is the permanent record. The checkout is the staging area in between.

① The cart — the container

A cart is "a collection of items, prices, discounts, etc." and, crucially, "it does not contain customer-related data."2 That is the defining fact: a cart knows what is being bought and what it costs, but not who is buying or where it goes. It is mutable, cheap, and disposable — shoppers add and remove things, abandon it, come back. Nothing about a cart is a commitment.

② The checkout — the commitment

A cart becomes a checkout the moment customer commitment is attached. In BigCommerce, "you can change a cart to a checkout by adding a billing address."1 The checkout is the same underlying purchase plus the things a cart deliberately lacks: a billing address, and one or more consignments — each a shipping address, its line items, and a chosen shipping option.1 This is the staging area where a loose basket becomes a shippable, payable transaction.

③ The conversion gate — shipping must be resolved

A checkout cannot become an order while shipping is undecided. The rule is exact: "A checkout containing physical products is not eligible to become an order until each of its consignments contains a selected_shipping_option."3 Consignments get a full lesson later; for now hold the gate: no chosen shipping on every consignment, no order.

④ The order — the record

Converting the checkout creates an order. Created through the API, "the order status is set to incomplete," and once payment succeeds "the order status is then automatically changed to Awaiting Fulfillment."1 The order is the durable artifact: unlike the cart, it is the permanent record of what was agreed, and it is what the rest of the business — fulfillment, refunds, reporting — acts on.

Where it surfaces in the API

Each stage is a different API resource, which is the clearest proof they are genuinely separate objects.

  • Cart API — create a cart, add and remove line items. Returns a cart ID (and a redirect URL you can hand to the storefront to resume checkout).1
  • Checkout API — take that cart ID, add a billing address (this is what promotes it to a checkout), then add consignments and select shipping options.1
  • Create an Order endpoint — a POST that converts the eligible checkout into an order and returns the new order ID.1

Two axes to keep straight from the start. First, the same three resources exist in both the REST Storefront API (customer-scoped, client-side, for the shopper's own session) and the REST Management API (server-side, store-wide) — same objects, different access model. And a naming tell that follows from it: the Storefront API returns camelCase fields (billingAddress), the Management API returns snake_case (billing_address). Second, the shared identity: the ID you use to address the checkout "will correspond with the cart ID returned by the Create a Cart endpoint."1 Cart and checkout are two views of one record; the order is the new, permanent object minted at conversion.

The three objects, scrubbed down to the fields that carry the story:

// ① CART — Management API, snake_case. Knows what, not who.
{
  "id": "a1b2c3d4-…",          // the id the checkout will reuse
  "customer_id": 0,            // 0 = guest; a cart carries no real customer
  "base_amount": 149.99,
  "cart_amount": 139.99,       // after cart-level discounts
  "line_items": {
    "physical_items":    [ { "product_id": 111, "variant_id": 222, "quantity": 2 } ],
    "digital_items":     [],
    "gift_certificates": [],
    "custom_items":      []
  }
}

A cart: line items and money, no identity. (Line-item types are Lesson 3.)

// ② CHECKOUT — Storefront API, camelCase. checkoutId == cartId.
{
  "id": "a1b2c3d4-…",
  "billingAddress": { "firstName": "Jane", … },   // ← promoted cart to checkout
  "cart": { "id": "a1b2c3d4-…", "cartAmount": 112.93 },
  "consignments": [
    {
      "id": "5ba12…",
      "address": { "city": "Austin", "countryCode": "US" },
      "lineItemIds": [ "69791a88-…" ],
      "selectedShippingOption": { "id": "bb3c…", "cost": 69.94 },  // ← the gate
      "availableShippingOptions": [ … ]
    }
  ],
  "grandTotal": "211.49"
}

A checkout is the cart plus the two things a cart lacks: a billing address and consignments.

// ③ ORDER — Management API, snake_case. The permanent record.
{
  "id": 218,
  "status": "Awaiting Fulfillment",  // was "incomplete" at creation
                                     //   (status has a numeric twin, status_id — Lesson 8)
  "cart_id": "a1b2c3d4-…",           // ← links the order back to its cart
  "total_inc_tax": "69.8400",
  "currency_code": "USD",
  "payment_status": "authorized",
  "billing_address": { "first_name": "Jane", … },   // copied in — a snapshot
  "products":     { "resource": "/orders/218/products"     },   // sub-resources,
  "consignments": { "resource": "/orders/218/consignments" }    //   fetched separately
}

The order carries cart_id back to its origin, and copies the address in rather than linking — the record must not change when the cart does.

Why three objects

The split pays off because the three stages have genuinely different needs. The cart must be fast and throwaway — most carts are abandoned, and holding customer data on every abandoned basket would be wrong on both privacy and storage grounds, which is exactly why a cart carries none. The checkout is where validation lives: addresses, shippability, tax, the shipping-option gate. The order must be immutable enough to be the source of truth for money and fulfillment. One object trying to be all three would be mutable and permanent, anonymous and identified, all at once. BigCommerce refuses that by drawing two hard lines: add a billing address (cart → checkout) and resolve shipping then convert (checkout → order).

Key line "A purchase in BigCommerce is three objects, not one: a cart (items, no customer, disposable), a checkout (cart + billing address + consignments, the staging area), and an order (the permanent record). A billing address promotes cart to checkout; a resolved shipping option on every consignment lets checkout convert to order."
The trap Thinking the cart "becomes" the order by changing status, the way a single draft-order object would. It does not. The cart and checkout share an ID and are two views of the same pre-purchase record, but the order is a separate object created at conversion. And the boundary that matters between cart and checkout is not payment — it is the billing address. Adding customer commitment, not taking money, is what turns a basket into a checkout.

① Name the three objects in the pipeline, in order, and the one-word job of each.

show answer

Cart (container), checkout (commitment / staging), order (record). Cart holds items with no customer data; checkout adds billing address + consignments; order is the durable post-conversion record.

② What single action turns a cart into a checkout?

show answer

Adding a billing address. Not payment — commitment. The checkout keeps the same ID as the cart.

③ What must be true of every consignment before a checkout with physical products can convert to an order?

show answer

Each consignment must contain a selected_shipping_option. No chosen shipping on all of them, no order.

④ Immediately after the API creates an order, what is its status, and what changes it next?

show answer

incomplete at creation; a successful payment automatically moves it to Awaiting Fulfillment.
