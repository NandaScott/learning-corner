How BigCommerce Works · Lesson 7

Consignments

Lesson 1 named the consignment as the checkout→order gate and moved on. Now the full object. A consignment is how BigCommerce plans fulfillment: it groups line items with a destination and a way to get there, generates the shipping options for that destination, and later becomes the order's fulfillment record. It is also where one cart can be split across several addresses and methods.

The concept

A single order does not always go to one place by one method. Part ships to a home address, part to an office; some items are collected in store; a digital item just downloads. So the checkout needs a unit smaller than the order — a grouping of "these items, going this way, to there." That unit is the consignment. Every real shipping arrangement is one or more of them.

How BigCommerce handles it

BigCommerce defines a consignment as "a list of physical products that will travel together to the purchaser, and it specifies how those items can and will ship."1 Three things make it work: an allocation of line items, a destination, and a fulfillment method whose options the server generates.

  CHECKOUT
   ├─ consignment A ──▶ address 1  + lineItems[…]  + selectedShippingOption   ┐
   ├─ consignment B ──▶ address 2  + lineItems[…]  + selectedShippingOption   │ ≥ 1 per address
   └─ consignment C ──▶ pickupOption(location) + lineItems[…]                  ┘ (pickup = BOPIS)

   creating a consignment (address + items)  ──generates──▶  availableShippingOptions[]
                                                             (based on the destination + weight)
                                              ──choose one──▶ selectedShippingOption   ← Lesson 1's gate

   convert to ORDER  ──▶  /orders/{id}/consignments  (object structure):
                          { shipping:[…], pickups:[…], downloads:[…], email:{…} }
                          each shipping consignment tracks items_total / items_shipped

A consignment plans one shipment. Multiple consignments split a cart; the object persists onto the order as the fulfillment record.

① Splitting a cart across addresses

Consignments are how ship-to-multiple works. "An order that ships to multiple addresses will have at least one consignment per address. It is also possible to create multiple consignments for the same address."1 You allocate items by listing them on each consignment — a line_items array of {item_id, quantity} (Management) — and the create endpoint accepts an array of consignments, building the whole split in one call. There is a cost to splitting: "the REST Management API responds quickly when the checkout contains one consignment. Each additional consignment increases the amount of time the API takes to finish."1

② Shipping options — available vs selected

The two shipping fields are not symmetric. available_shipping_options are computed by the server, not supplied by you: "a newly created consignment returns a list of fulfillment options available based on the destination address."1 They only exist once the consignment has an address and items, and you must ask for them explicitly — "append include=consignments.available_shipping_options to the endpoint so that you will have the information to select one."1 Then choosing one sets selected_shipping_option — Lesson 1's conversion gate. Each option carries id, type, description, cost, transitTime, and an isRecommended flag.2

③ Why you can't change address and pick a method at once

There are "two distinct kinds of consignment updates. The first selects a fulfillment option. The second can update the recipient's address and adjust the list of included line items. You must choose one type of consignment update… You can't do both in the same API call."1 The reason is causal, not arbitrary: "changing the address and weight can change available fulfillment options." Selecting an option you computed against the old address would be selecting from a stale menu, so the API forbids it — change the destination first, let the options regenerate, then choose.

④ Pickup consignments (BOPIS)

Not every consignment ships. In-store pickup is a first-class consignment type: instead of an address plus a shipping option, a pickup consignment supplies a pickupOption (a pickupMethodId naming a location) plus its line items.2 "Required fields for shipping: address and lineItems. Required fields for pickup: pickupOption and lineItems."2 One order can mix them — some items shipped, some picked up — as separate consignments.

⑤ From checkout consignment to order consignment

When the checkout converts, its consignments persist onto the order at /orders/{id}/consignments. In the object structure, an order's consignments are grouped by fulfillment method into four keys: shipping, pickups, downloads, and email.3 Each shipping consignment carries the fulfillment detail — shipping_method, costs, and crucially items_total and items_shipped, the inline counters that track dispatch progress. (The consignment is the intended grouping; Lesson 8 covers the shipment — the actual dispatch record with tracking — and how it advances the order status.)

Where it surfaces in the API

Consignments live on the checkout in both APIs (camelCase in Storefront, snake_case in Management — the Lesson 1 casing split) and persist onto the order.

  • Create / update — checkout consignments via REST Storefront and REST Management; create takes an array (multi-address in one call); options come back only with include=consignments.available_shipping_options.1
  • Select an option — a separate update sending shipping_option_id (never combined with an address change).1
  • On the order — GET /orders/{id}/consignments. Pass consignment_structure=object: the default array structure "is legacy and will be deprecated from 1 Feb 2026."3
// CHECKOUT CONSIGNMENT — Storefront, camelCase (scrubbed)
{
  "id": "5ba12…",
  "address": { "firstName": "Jace", "city": "Austin", "countryCode": "US", … },
  "lineItemIds": ["00a8e1c3-996f-4786-96ca-2a8a887b6648"],
  "availableShippingOptions": [                 // server-generated from the address…
    { "id": "d09e…", "type": "shipping_flatrate", "description": "Flat Rate",
      "cost": 5, "transitTime": "", "isRecommended": true }
  ],
  "selectedShippingOption": {                   // …one of them, chosen = the gate
    "id": "d09e…", "type": "shipping_flatrate", "description": "Flat Rate", "cost": 5
  }
}

availableShippingOptions is computed and must be explicitly included; selectedShippingOption is what lets the checkout convert.

// POST .../checkouts/{id}/consignments
[
  { "address": { "first_name": "Jace",    "city": "Austin", "country_code": "US", … },
    "line_items": [ { "item_id": "00a8e1c3-…", "quantity": 1 } ] },
  { "address": { "first_name": "Liliana", "city": "Austin", "country_code": "US", … },
    "line_items": [ { "item_id": "00a8e1c3-…", "quantity": 1 } ] }
]

// then, SEPARATELY, select an option (never in the same call as an address change):
// PUT .../consignments/{consignmentId}
{ "shipping_option_id": "9241669174884c2f2e83b3adabf03f83" }

One array creates the whole split. Selecting a method is a distinct update — because moving the address regenerates the options.

// ORDER CONSIGNMENTS — object structure, snake_case
// GET /orders/{id}/consignments?consignment_structure=object
{
  "shipping": [
    { "id": 1, "first_name": "Jace", "city": "Austin",
      "shipping_method": "Flat Rate", "cost_inc_tax": "5.0000",
      "items_total": 1, "items_shipped": 0,           // ← dispatch progress, inline
      "line_items": [ { "resource": "/orders/129/products/12" } ] }
  ],
  "pickups":   [ /* in-store pickup consignments */ ],
  "downloads": [],
  "email":     { "gift_certificates": [] }
}

On the order, consignments are grouped by fulfillment method. Pass consignment_structure=object — the legacy array form retires 1 Feb 2026.

Why consignments exist

The consignment is the checkout's fulfillment-planning unit, and giving it its own object is what makes three otherwise-hard things simple. Multi-address orders are just multiple consignments, so nothing elsewhere in the pipeline needs a special "split shipment" mode. The conversion gate from Lesson 1 stops being arbitrary once you see that shipping options are generated from the destination — an order can't be priced or promised until each destination has a chosen method, which is exactly "every consignment needs a selected_shipping_option." And mixed fulfillment — ship some, pick up some, download some — falls out of the same object carrying different methods. The consignment plans the shipment at checkout; it persists onto the order as the record fulfillment works from.

Key line "A consignment groups line items + a destination + a fulfillment method; multiple consignments split a cart (at least one per address). The server generates available_shipping_options from the destination — you must include them explicitly — and choosing one sets the selected_shipping_option that lets the checkout convert. Address/line-item changes and option selection are separate calls, because moving the address regenerates the options."
The trap Treating shipping options as data you set, and updates as free-form. Neither holds. available_shipping_options are computed by the server from the address and weight, exist only after items and an address are present, and aren't returned unless you include them. And you cannot change the address (or line items) and select a shipping option in one call — they are two distinct update kinds, because changing the destination regenerates the option menu. One more: on the order, always pass consignment_structure=object; the default array structure retires 1 Feb 2026.

Recall — answer out loud before peeking:

① A cart ships to two addresses. Minimum number of consignments, and how are items assigned to each?

show answer

At least two — one consignment per address (more are allowed). Items are allocated by each consignment's line_items list of {item_id, quantity}.

② Where do available_shipping_options come from, and what must you do to see them?

show answer

The server generates them from the destination address (and item weight), only after the consignment has an address + items. You must request them with include=consignments.available_shipping_options.

③ Why can't you change a consignment's address and select its shipping option in the same API call?

show answer

Because changing the address/weight regenerates the available options — selecting one in the same call would pick from a stale menu. They are two distinct update kinds.

④ On the order, what four keys group consignments in the object structure, and what param do you pass?

show answer

shipping, pickups, downloads, email. Pass consignment_structure=object (the legacy array form deprecates 1 Feb 2026).
