How BigCommerce Works · Lesson 3

Line items

Lesson 1 put items in a cart; Lesson 2 said a purchasable thing is a variant. This lesson is the join between them: the line item, one row on the receipt. It is where a catalog variant plus its modifier choices becomes a concrete "2 × this, at this price" — and, crucially, where that becomes a frozen record rather than a live link back to the catalog.

The concept

A line item is one row of an order: a specific thing, a quantity, and a price. Every cart and every order is, at bottom, a list of these rows plus some totals.

The subtle part is time. The catalog changes — prices move, products get renamed, items are discontinued. A line item must not move with it. If a shopper adds a headphone at $99 and the merchant raises the catalog price to $119 an hour later, the cart still owes $99. And a placed order must read the same next year as it did the day it shipped. So a line item cannot be a live pointer into the catalog; it has to capture what was true when the row was created. A line item is a snapshot.

How BigCommerce handles it

BigCommerce sorts a cart's rows into typed buckets, builds each row from a catalog variant (or from nothing, for a one-off), and freezes the details onto the row. Three things to hold:

   CART.line_items
   ├─ physical_items[]     ← variants with type "physical" (ship; need a consignment)
   ├─ digital_items[]      ← variants with type "digital"  (no shipping)
   ├─ gift_certificates[]  ← a stored-value certificate, not a catalog product
   └─ custom_items[]       ← a row with NO catalog link (name + price typed in)

   Building one physical row from the catalog:

     product_id  ─┬─ + variant_id            → pick the SKU directly
                  └─ + option_selections[]   → when the product has MODIFIERS
                                               (encodes the variant AND the
                                                modifier choices together)
                          │
                          ▼
     the row captures name, sku, price, chosen options  ── frozen ──▶ snapshot

Rows are typed by kind; each is built from a variant (or typed in), then its details are frozen onto the row.

① The four line-item types

A cart's line_items object splits rows into four arrays by kind: physical_items, digital_items, gift_certificates, and custom_items.1 The first two come straight from Lesson 2's product type — a physical variant lands in physical_items and is what forces a consignment and shipping option back in Lesson 1; a digital variant lands in digital_items and skips shipping entirely. Gift certificates are stored value, not a catalog product. Custom items are the escape hatch, below.

② How a catalog product becomes a row

Adding a catalog row needs a product_id and a quantity. To say which variant, there are two mutually exclusive ways, and this is the part people get wrong. If the product is a plain variant pick, send its variant_id. But "when a product has modifiers, omit the variant_id and instead use the option_selections array to describe both the variant and the modifier selections."2 That is the payoff of Lesson 2's split: variant options and modifiers are different objects, so a row that involves modifiers cannot be named by a SKU alone — option_selections is what carries the engraving text or the insurance checkbox alongside the size-and-color choice.

③ Custom items — a row with no catalog behind it

Sometimes the thing sold is not in the catalog: a phone order, a one-off, a fee. A custom item is a line item built from typed-in values — it "is not added to the catalog" and requires sku, name, quantity, and list_price, with no product_id at all.2 (An image_url is accepted but does not render on the storefront.2) A custom item is the clearest proof of the snapshot idea: it is all snapshot and no catalog link, and the rest of the pipeline treats it like any other row.

④ The snapshot — why the row is frozen

Three facts in the model show the row captures rather than links. First, when adding a catalog row you may override name and list_price right on the line2 — a live pointer to the catalog could not allow that. Second, custom items carry full name and price with no catalog reference at all. Third, recall from Lesson 1 that an order exposes its own /orders/{id}/products sub-resource, separate from the catalog: the order keeps its own copy of every row. Change the catalog price tomorrow and existing carts and orders are untouched, because they never held a pointer — they held a copy.

Where it surfaces in the API

Rows live inside the cart's line_items object (Lesson 1's cart example), are created through the cart's items endpoint, and persist onto the order as the /products sub-resource.

  • Add rows — POST /v3/carts/{cartId}/items, with a line_items array (catalog rows) and/or a custom_items array.2
  • Read rows — they come back inside the cart's line_items, split into the four typed arrays.1
  • On the order — GET /v2/orders/{id}/products, the order's own frozen copy of its rows.1

The request body — the two ways to name a variant, side by side, plus a custom row:

// POST /v3/carts/{cartId}/items
{
  "line_items": [
    { "product_id": 111, "variant_id": 222, "quantity": 2 },   // plain: name the SKU

    { "product_id": 333, "quantity": 1,                        // has modifiers:
      "option_selections": [                                   //   NO variant_id
        { "option_id": 102, "option_value": "205" },           //   the variant facet
        { "option_id": 88,  "option_value": "J.D." }           //   the modifier (engraving)
      ] }
  ],
  "custom_items": [
    { "sku": "PHONE-ORDER-1", "name": "Repair fee",            // no product_id:
      "quantity": 1, "list_price": 40.00 }                     //   nothing in the catalog
  ]
}

Two ways in: variant_id for a plain pick, option_selections when modifiers are in play. Custom items carry their own name and price.

// A returned physical row, scrubbed — note it's a copy, not a link
{
  "id": "69791a88-…",
  "product_id": 333,
  "variant_id": 334,
  "name": "Premium Headphones",     // captured — survives later catalog renames
  "sku": "HEADPHONES-001-RED",
  "quantity": 1,
  "list_price": 99.99,              // captured — survives later price changes
  "sale_price": 99.99,
  "extended_list_price": 99.99,     // list_price × quantity
  "options": [                      // the chosen variant + modifier selections, frozen
    { "name": "Fit",       "value": "Over-ear" },
    { "name": "Engraving", "value": "J.D." }
  ]
}

The row holds its own name, list_price, and chosen options. The lineItemIds a consignment referenced back in Lesson 1 point at rows like this one.

Why line items are their own object

The line item is where the two halves of this course meet: it takes a variant from the catalog (Lesson 2) and places it into the pipeline (Lesson 1), and by freezing the details at that moment it lets the two halves change independently. The catalog stays free to evolve — reprice, rename, discontinue — without rewriting anyone's cart or falsifying anyone's past order. The four types let one row list carry everything a purchase can contain: things that ship, things that download, stored value, and one-offs that were never catalog items at all. And option_selections is the seam that keeps Lesson 2's variant-versus-modifier distinction intact all the way to the receipt: the row records not just which SKU, but every choice the shopper made about it.

Key line "A line item is one frozen row: a variant (or a typed-in custom item) plus quantity, with its name, price, and chosen options captured at add-time — not linked live to the catalog. Rows come in four types (physical, digital, gift certificate, custom); catalog rows name their variant by variant_id, or by option_selections when modifiers are involved."
The trap Thinking a line item is a live reference to a product, so editing the catalog would change carts and orders. It does not — the row is a snapshot, which is exactly why an order still reads correctly after the catalog moves on. The second, mechanical half of the same trap: when a product has modifiers, sending variant_id is wrong. You must omit it and use option_selections, because a SKU alone cannot carry the modifier choices (the engraving text, the insurance checkbox) — only option_selections encodes the variant and the modifiers together.

Recall — answer out loud before peeking:

① Name the four line-item types in a cart's line_items.

show answer

physical_items, digital_items, gift_certificates, custom_items.

② A product has modifiers. How do you specify what the shopper chose when adding it, and what must you not send?

show answer

Send option_selections (which encodes both the variant and the modifier choices). Do not send variant_id — omit it. A SKU alone can't carry the modifier picks.

③ A merchant raises a product's catalog price after a shopper added it to their cart. What happens to the existing line item, and why?

show answer

Nothing — it keeps its captured list_price. The row is a snapshot, not a live link to the catalog, so catalog changes don't rewrite it.

④ What is a custom item, and which field does it lack that a catalog row has?

show answer

A line item with no catalog link — it carries its own sku, name, quantity, list_price. It lacks a product_id (and any variant_id).
