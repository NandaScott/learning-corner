How BigCommerce Works · Lesson 5

The discounts landscape

"Discount" is one word hiding four different machines. Lesson 4 already met one of them — the catalog sale price. This lesson maps the whole terrain: sale price vs automatic promotion vs coupon, and the fact that BigCommerce runs two separate coupon systems. The goal is a taxonomy sharp enough that you always know which object and which API you are talking about.

The concept

Shoppers experience every price cut as "a discount," but the systems behind them differ on three axes that decide how you build and reason about them: where the cut is applied (baked into the catalog, or applied to the cart), whether a code is required, and which object owns the logic. Blur those axes and you double-count savings, or reach for the wrong API.

How BigCommerce handles it

Four kinds, along those axes:

  KIND                     applied where     needs code?   owned by
  ─────────────────────────────────────────────────────────────────────────
  1  Sale price            CATALOG (item)    no            Catalog API (sale_price)
  2  Automatic promotion   CART              no            Promotions API (AUTOMATIC)
  3  Coupon promotion      CART / checkout   YES           Promotions API (COUPON) + code
  4  Legacy v2 coupon      CART / checkout   YES           Coupons API (/v2/coupons)

                    ▲ catalog          ▲──────── cart layer ────────▲
              price is already cut     discount decided when the cart
              before a cart exists     is evaluated (Lesson 6's engine)

  Two of these carry a CODE — and they come from two DIFFERENT systems (see step ④).

Four machines. The split that matters: catalog (kind 1) vs cart (kinds 2–4), and the two separate code-bearing systems.

① Sale price — a catalog fact, not a discount event

The first "discount" isn't an event at all. A product or variant sale_price "will be used instead of value in the price field when calculating the product's cost,"1 and it feeds calculated_price from Lesson 4 before any cart exists. No code, no rule, no promotion object — it is catalog data. Treat it as a lower price, not as a discount applied to an order.

② Automatic promotions — cart rules with no code

The remaining three act on the cart. A promotion gives "discounts based on whether the shopper has met specific criteria such as ordering a certain amount, purchasing certain brands, or being a repeat customer."2 An automatic promotion needs no code: "the shopper does not need to satisfy any special conditions, and the promotion will apply automatically."2 It fires the moment the cart matches its rules.

③ Coupon promotions — the same engine, gated by a code

The difference between automatic and coupon-gated is a single field on the promotion: redemption_type, either AUTOMATIC or COUPON.2 For a coupon promotion you additionally "send a request to the Create a coupon code endpoint,"2 and — the key mental move — that code carries no discount logic. The discount lives in the promotion's action (Lesson 6); the coupon code is only the key that unlocks it. One field flips the same rule engine between "applies on its own" and "applies when this code is entered."

④ Two coupon systems — the trap in the terrain

Here is the fact that trips people up: BigCommerce has two coupon systems. The modern one is the Promotions coupon code above — a thin object of code plus usage counters, whose discount lives in the parent promotion. The older one is the standalone v2 Coupons API, where the coupon object itself carries the discount: a type (such as per_total_discount) and an amount.3 Same shopper-facing idea, "enter a code, get money off," but two different objects in two different APIs. When you see "coupon," your first question must be: which system?

⑤ How discounts surface on the cart and order

Because the kinds differ, they show up in different places, and BigCommerce keeps them deliberately separate. On the cart, promotion/order discounts land in a line item's discounts array, while coupons land in the cart-level coupons array.4 And the cart's discountAmount is narrower than it looks — it is "order based discounted amount only — Coupon discounts and product based discounts are excluded."4 On the finished order, redeemed coupons appear in the coupons sub-resource, each with its code, amount, and a type.5

Where it surfaces in the API

  • Sale price — sale_price on /v3/catalog/products and variants (Lesson 4).1
  • Promotions (automatic & coupon) — /v3/promotions; coupon codes at POST /v3/promotions/{id}/codes.2
  • Legacy coupons — the v2 /v2/coupons Coupons API, a separate system.3
  • On the cart / order — cart line-item discounts[] and cart-level coupons[]; order /orders/{id}/coupons.4

The centerpiece contrast — the same "code for money off," from the two systems:

// LEGACY v2 COUPON — the coupon carries its OWN discount logic
// GET /v2/coupons/{id}
{
  "id": 1,
  "name": "$5 off",
  "code": "S2549JM0Y",
  "type": "per_total_discount",      // ← the discount lives HERE
  "amount": "5.0000",                // ← and HERE
  "min_purchase": "0.0000",
  "applies_to": { "entity": "categories", "ids": [0] },
  "max_uses": 0,
  "enabled": true
}

// PROMOTIONS COUPON CODE — a thin trigger; discount lives in the PROMOTION
// POST /v3/promotions/{promotion_id}/codes   → returns:
{
  "data": {
    "id": 1,
    "code": "SPARK20",               // ← just the key…
    "current_uses": 0,
    "max_uses": 200,
    "max_uses_per_customer": 1
  },                                 // …no type, no amount: the promotion holds those
  "meta": {}
}

Two coupons, two systems. The v2 coupon is self-contained; the Promotions code is a bare key to a promotion's rule.

// ON THE FINISHED ORDER — GET /orders/{id}/coupons
[
  { "id": 3, "coupon_id": 1, "order_id": 120,
    "code": "S2549JM0Y",
    "amount": 5,        // "$3 returns as 3; 5% returns as 5" — read type to interpret
    "type": 2,          // integer enum here (2 = per_total_discount)
    "discount": 5 }
]

Redeemed coupons on the order. Note amount is ambiguous without type — a percent and a dollar amount can print identically.

Why the taxonomy matters

The four kinds exist because the cuts happen at genuinely different times and places. A sale price must be true before a cart exists — it is what the storefront displays and what a line item freezes at add-time (Lessons 3–4), so it cannot be a cart rule. Automatic and coupon promotions are cart-time decisions, computed when the cart is evaluated, which is why they need the rule engine of the next lesson. And the two coupon systems are simply history: the flexible modern engine lives in Promotions, with the coupon reduced to a trigger, while the older self-contained coupon persists for compatibility. Keeping them straight is what lets you read a cart's totals correctly — discountAmount, discounts[], and coupons[] are three different numbers, on purpose.

Key line "Four kinds of discount on three axes (catalog vs cart, coded vs not, which object): catalog sale_price; automatic promotions; coupon promotions; and legacy v2 coupons. Automatic vs coupon is one field, redemption_type. And there are two coupon systems — the modern Promotions code is a bare trigger whose discount lives in the promotion; the legacy v2 coupon carries its own type and amount."
The trap Collapsing "discount" into one idea, in two specific ways. First, assuming there is one coupon concept — there are two systems (Promotions coupon codes vs the v2 Coupons API), and the code's meaning is opposite in each: a trigger in one, the whole discount in the other. Second, reading a cart's discountAmount as total savings — it is order-based discounts only and excludes coupon and product discounts, which live in coupons[] and in item pricing. Both mistakes make the math wrong.

Recall — answer out loud before peeking:

① Which of the four kinds is applied in the catalog, before a cart exists, and needs no code?

show answer

The product/variant sale_price. It feeds calculated_price and is catalog data, not a cart-time discount event.

② What single field decides whether a promotion is automatic or requires a coupon code?

show answer

redemption_type — AUTOMATIC (no code) or COUPON (a code created at /promotions/{id}/codes unlocks it). It's immutable after creation.

③ In the modern Promotions system, where does a coupon code's discount logic live — on the code, or somewhere else?

show answer

Not on the code. The code is a bare trigger (code + usage counters); the discount logic lives in the parent promotion's action. Contrast the legacy v2 coupon, which carries its own type/amount.

④ A cart's discountAmount is $10. Is that the shopper's total savings? Why or why not?

show answer

No. discountAmount is order-based discounts only — it excludes coupon discounts (in coupons[]) and product/catalog discounts (in item pricing). Total savings can be higher.
