How Stripe Works · Lesson 2

The PaymentIntent state machine

Lesson 1 walked the path a card payment takes through the banking system. The PaymentIntent is Stripe's API object for tracking one payment's progress along that path. It is a state machine, and this lesson walks every state and every transition.

The concept

Getting paid used to look like one synchronous API call: send card details, get back a charge or an error. That model breaks the moment the path from Lesson 1 needs a detour. The issuer may demand cardholder authentication mid-payment. Some payment methods take days to confirm. A declined attempt may be retried with a different card. None of that fits in one request-response.

Stripe's answer is to represent the payment as an object with a status that changes over its lifecycle: the PaymentIntent. It "tracks a payment from creation through checkout, and triggers additional authentication steps when required."1 Two design facts define how it is used:

One PaymentIntent per cart, not per attempt. "Each PaymentIntent typically correlates with a single shopping cart or customer session."1 If checkout is interrupted, the integration reuses the same PaymentIntent rather than creating a new one, so the object accumulates the history of failed attempts for that cart.

Attempts are Charge objects hanging off the intent. "A PaymentIntent might have more than one Charge object associated with it if there were multiple payment attempts."1 The intent is the goal ("collect $50 for this cart"); each charge is one try at reaching it.

Using it takes two actions: the server creates the PaymentIntent (with amount and currency), then the payment is confirmed, which starts the actual authorization from Lesson 1. Confirmation usually happens from the customer's browser, using a client secret: a unique key returned at creation that lets the client read the intent's safe fields (status, amount) and drive it forward, without exposing sensitive ones.1

The states

The full machine. Statuses are exact API strings; every PaymentIntent you ever inspect is at exactly one of them.2

          create (server, amount + currency)
                      │
                      ▼
      ┌── requires_payment_method ◀───────────────┐
      │               │  payment method attached  │
      │               ▼                           │  attempt fails:
      │    requires_confirmation                  │  back to the start
      │               │  confirm                  │
      │               ▼                           │
      │      requires_action ── (only if the      │
      │               │          issuer demands   │
      │               │          e.g. 3D Secure)  │
      │               ▼                           │
      │          processing ──────────────────────┘
      │               │
      │               ├──▶ requires_capture ──▶ succeeded   (manual capture)
      │               └──▶ succeeded                        (default)
      │
      └──▶ canceled   (terminal; reachable from the non-terminal states)

The happy path runs down the middle in seconds. The loop on the left is what makes the object worth having.

requires_payment_method — the starting state

A freshly created PaymentIntent knows the amount and currency and nothing about how it will be paid. It sits at requires_payment_method until payment details are attached.2 This is also the state a failed attempt returns to: a decline does not kill the intent, it rewinds it, ready for the customer to try another card.

requires_confirmation — details attached, not yet fired

With a payment method attached, the intent is ready to be confirmed. Most integrations never observe this state because they attach and confirm in one step; it exists for flows that separate "customer picked a card" from "run the payment now."2

requires_action — the machine pauses for the customer

Confirmation starts the authorization request from Lesson 1 (Stripe to network to issuer). If the issuer demands cardholder authentication before answering, such as a 3D Secure challenge, the intent parks at requires_action and its next_action field says what has to happen, for example a redirect URL for the customer to complete.2 Skipped entirely when no authentication is demanded. Lesson 3 is about what happens inside this state.

processing — Stripe is working

Required actions are done and Stripe is executing the payment. For cards this state passes in moments. For asynchronous payment methods, such as bank debits, an intent can sit at processing for days while the money network does its slow work.2

requires_capture — the hold from Lesson 1, as a status

An integration using manual capture lands here after authorization succeeds: the issuer's hold is in place and the intent waits for the capture instruction.2 This status is the API surface of Lesson 1's authorize-now-capture-later window. If the hold expires uncaptured, the funds release and the intent moves to canceled.

succeeded — the payment is complete

Capture happened; "the funds are in your account."2 Terminal. Read that phrase with Lesson 1 precision: the funds are in (or on their way into) the Stripe balance, on the days clock. succeeded says nothing about the bank account.

canceled — the other terminal state

The intent was invalidated: canceled explicitly before completion, or automatically, for instance when an uncaptured authorization expired. A canceled intent cannot be used for further attempts.2

Why a state machine

Lay the two lessons side by side and the mapping is one-to-one. requires_action and processing cover Lesson 1's steps 2 through 4, the authorization conversation. requires_capture is the gap between steps 4 and 5, the standing hold. succeeded is step 5 complete, with settlement and payout still ahead of it. The PaymentIntent models the seconds clock and hands off to the balance for the days clock.

The failure loop is the part the old one-call model could not express. Because a decline rewinds to requires_payment_method instead of ending anything, one object carries a whole checkout session: three declined cards and a fourth that worked are four Charge attempts under one intent, and the merchant's records hold one payment, not four.1

Key line "A PaymentIntent is the state machine for one cart getting paid; charges are its attempts. Declines rewind it, actions pause it, and succeeded means the balance, not the bank."
The trap Two shallow readings. First: succeeded does not mean money reached the bank account; it means capture completed and the funds belong to the Stripe balance, with settlement timing and payout still to run. Second: a failed payment does not put the intent into a "failed" status; there is no such status. It returns to requires_payment_method, because the intent models the cart, and the cart can still be paid.

Recall — answer out loud before peeking:

① A customer's card is declined. What status is the PaymentIntent in now, and why that one?

show answer

requires_payment_method, the starting state. A decline rewinds the intent rather than terminating it, because the intent represents the cart, which can still be paid with another method.

② What is the relationship between a PaymentIntent and Charge objects?

show answer

One intent per cart or checkout session; one Charge per payment attempt. Retries create multiple Charges under the same intent.

③ Which status corresponds to Lesson 1's standing authorization hold, and how does an intent get there?

show answer

requires_capture. A manual-capture integration lands there when authorization succeeds; the intent waits for the capture call, and moves to canceled if the hold expires first.

④ The API says succeeded. Where is the money?

show answer

Captured and headed into the merchant's Stripe balance (pending, then available on the T+X settlement clock). Not yet in the bank account; that takes the payout.
