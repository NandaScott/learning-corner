How Stripe Works · Lesson 4

Webhooks, the source of truth

The last two lessons built a state machine that keeps moving after the API call returns and after the customer leaves. This lesson covers how a merchant's server finds out what happened: the Event object, the delivery contract, and why the signature check is not optional.

The concept

Most of the interesting transitions in this course happen when nobody is on the line. A bank debit confirms days after checkout. A customer disputes a charge weeks later. A 3DS challenge completes on the issuer's page, not the merchant's. A subscription renews at 3 a.m. Stripe's docs put it directly: webhooks exist so you can "respond to asynchronous events, such as when a customer's bank confirms a payment, a customer disputes a charge, or a recurring payment succeeds."1 The request-response cycle cannot deliver these outcomes, because there is no open request when they occur.

The mechanism inverts the arrow. Instead of the merchant polling Stripe, Stripe calls the merchant: an HTTPS POST to a URL the merchant registered, carrying an Event object as JSON. An Event has a type naming what happened (payment_intent.succeeded, charge.dispute.created), a data.object snapshot of the affected resource, and a unique id.1

That inversion has a consequence worth sitting with: the webhook endpoint is a door on the public internet that accepts POSTs from anyone. The whole delivery contract below exists to make that door safe and reliable.

The steps

One event, from state change to safely processed.

1. State changes      intent hits succeeded            Stripe creates Event (unique id)
2. Delivery           Stripe POSTs JSON → endpoint     HTTPS, Stripe-Signature header
3. Verify             HMAC check on the signature      reject anything that fails
4. Acknowledge        return 2xx FAST                  before any heavy logic
5. (no 2xx?)          exponential-backoff retries      for up to 3 days (live mode)
6. Process            dedupe by event id               assume no ordering

Steps 3, 4, and 6 are the merchant's obligations. Getting any of them wrong produces a specific, classic bug.

Step 1 — A state change mints an Event

Every consequential change in the account produces an Event object. The Lesson 2 machine is the main producer here: transitions like reaching succeeded or rewinding to requires_payment_method each have a corresponding event type, which is how the machine reports its moves to the outside world.

Step 2 — Stripe POSTs to the registered endpoint

Stripe sends the Event as a JSON payload over HTTPS to each endpoint registered for that event type.1 Endpoints subscribe to the types they care about; everything else is noise they never receive.

Step 3 — Verify the signature before believing anything

Because the endpoint is publicly reachable, the payload alone proves nothing. "Without verification, an attacker could send fake webhook events to your endpoint to trigger actions like fulfilling orders, granting account access, or modifying records."1 The proof of origin rides in the Stripe-Signature header:

Stripe-Signature:  t=1492774577, v1=5257a869e7ecebe...

verification:
  signed_payload = t + "." + raw_request_body
  expected       = HMAC-SHA256(endpoint_signing_secret, signed_payload)
  compare(expected, v1)          ← constant-time comparison
  check |now - t| within tolerance  (default 5 minutes)

The signing secret is shared between Stripe and this one endpoint. Only a holder of the secret can produce a valid v1, and the timestamp inside the signed payload blocks replaying old captures.1

The official libraries wrap this in one call (construct_event), which is the recommended path; the mechanism above is what that call does.

Step 4 — Return 2xx before doing the work

The endpoint "must quickly return a successful status code (2xx) prior to any complex logic that could cause a timeout."1 Acknowledge first, then fulfill the order, send the email, update the records. A handler that does the slow work first risks timing out, and a timeout is a failed delivery.

Step 5 — Failed deliveries retry for three days

Any non-2xx response, or no response, and Stripe re-attempts delivery "for up to three days with an exponential back off" in live mode (a sandbox retries three times over a few hours).1 This is the reliability half of the contract: a deploy, a crash, or an outage does not lose the event, it delays it. It also means every handler must expect to see the same event more than once, because a timeout after successful processing still triggers a retry.

Step 6 — No ordering, no uniqueness: handle both

"Stripe doesn't guarantee the delivery of events in the order that they're generated."1 One action often mints several events (a subscription payment produces invoice and charge events together), and they can land in any order. The docs' prescriptions: build handlers that do not depend on sequence, and "guard against duplicated event receipts by logging the event IDs you've processed" and skipping ones already seen.1

Why webhooks are the source of truth

Put the three clocks from Lesson 1 next to this machinery. The API response from confirming a payment reports the seconds clock, and only as far as it ran while the request was open. Everything on the days clock, and every seconds-clock outcome that completed after the customer left, arrives by webhook or not at all. That is why fulfillment keyed off the synchronous response is fragile, and fulfillment keyed off payment_intent.succeeded, verified and deduplicated, is the pattern Stripe's docs push throughout.

Each obligation maps to an attack or a failure. Skip verification: forged events, free goods. Work before acknowledging: timeouts read as failures, and retries double-process. Assume order or uniqueness: bugs that appear only under retry and concurrency, which is to say, only in production.

Key line "The API response tells you what happened while you waited. Webhooks tell you what happened after you stopped waiting: verify the signature, ack fast, dedupe by event id, assume no order."
The trap The shallow model treats webhooks as optional notifications layered over a complete API. For asynchronous outcomes they are the only channel; the API response ends at the end of the request. The deep answer also names the security inversion: a webhook endpoint is unauthenticated inbound HTTP, so the HMAC signature check is the authentication, and processing an unverified payload means an attacker can mint payment_intent.succeeded at will.

Recall — answer out loud before peeking:

① Why can't the API response to a confirm call be the source of truth for payment outcomes?

show answer

Because outcomes keep happening after the request closes: async payment methods confirm days later, 3DS completes off-site, disputes arrive weeks on. No open request exists when they occur, so Stripe must push them: webhooks.

② An endpoint returns 500. What does Stripe do, and for how long?

show answer

Retries delivery with exponential backoff for up to three days in live mode. Sandbox: three retries over a few hours.

③ How does signature verification actually work, and what attack does it stop?

show answer

Stripe sends Stripe-Signature: t=…, v1=… where v1 = HMAC-SHA256 of "timestamp.body" under the endpoint's signing secret. The server recomputes and compares constant-time, and checks the timestamp (5-minute default tolerance) against replay. It stops forged events from anyone who found the public URL.

④ Two guarantees Stripe does NOT make about delivery, and the countermeasure for each?

show answer

No ordering (write handlers that don't depend on sequence) and no single delivery (log processed event ids, skip duplicates).
