How Stripe Works · Lesson 5

Idempotency, or how retries become safe

Lesson 4 covered Stripe retrying deliveries to the merchant. This lesson is the mirror image: the merchant retrying requests to Stripe, and the mechanism that makes retrying a money-moving request safe instead of terrifying.

The concept

Networks fail in an inconvenient place: after the request was sent, before the response came back. A merchant's server POSTs a payment request, the connection drops, and now the state of the world is unknowable from where it sits. Maybe Stripe never got the request. Maybe Stripe executed it fully and the confirmation died in transit. With most APIs the safe move is unclear: not retrying risks an unpaid order, retrying risks charging the customer twice.

An operation is idempotent when performing it twice has the same effect as performing it once. Stripe's API makes its POST requests idempotent through the Idempotency-Key header: a unique string the client generates and attaches to a request. The mechanism behind it is storage, not deduplication magic: Stripe saves "the resulting status code and body of the first request made for any given idempotency key, regardless of whether it succeeds or fails," and "subsequent requests with the same key return the same result."1

A retry with the same key is therefore not a second attempt. It is a question: what happened the first time?

The steps

One payment request surviving a network failure.

1. Generate key      client mints UUID v4              one key per logical operation
2. Send              POST + Idempotency-Key header     (POST only; GET/DELETE need none)
3. First arrival     Stripe executes the request       saves {status, body} under the key
4. Failure           response lost in transit          client cannot know the outcome
5. Retry, same key   Stripe finds the key              REPLAYS the saved response
                                                       nothing re-executes
6. Expiry            keys removable after 24h          same key later = brand-new request

The stored response is the whole trick. Steps 3 and 5 are the same HTTP exchange from the client's view; only Stripe knows one executed and one replayed.

Step 1 — One key per logical operation

The client generates the key, and its scope is the thing to get right: one key per operation the business means to happen once, not per HTTP attempt. Stripe suggests "V4 UUIDs, or another random string with enough entropy to avoid collisions," up to 255 characters, and warns against building keys from sensitive data like email addresses.1

Step 2 — POST only

Keys go on POST requests. "Don't send idempotency keys in GET and DELETE requests because it has no effect. These requests are idempotent by definition"1: reading a resource twice reads it twice; deleting a deleted thing is still deleted. POST is the verb that creates and mutates, so POST is where "twice" differs from "once."

Step 3 — First execution, and what gets saved

The first time a key arrives, Stripe runs the request and records the outcome. One boundary matters: "we save results only after the execution of an endpoint begins. If incoming parameters fail validation, or the request conflicts with another request that's executing concurrently, we don't save the idempotent result."1 Rejected-at-the-door requests leave no record, and can be retried freely.

Step 4 — The unknowable failure

The timeout hits. From the client's seat, outcomes "executed" and "never arrived" are indistinguishable. This is the exact situation the mechanism exists for, and the client's move is mechanical: retry, same key.

Step 5 — The replay

Stripe sees a key it knows and returns the stored status code and body. If the first request created a payment, the retry returns that same payment, not a second one. And the replay is faithful to a fault: a stored error is replayed too, "including 500 errors."1 The key answers "what happened," and what happened may be a failure; a genuinely new attempt at the operation needs a new key.

Step 6 — Keys expire

Stripe can remove keys "after they're at least 24 hours old," and reusing a key after removal "is treated as a new request."1 The protection is a 24-hour window, sized for retry storms and queue backlogs, not an eternal ledger.

Why this completes the reliability picture

Set this lesson next to Lesson 4 and the symmetry is exact. Stripe-to-merchant: deliveries retry for days, so the merchant deduplicates by event id. Merchant-to-Stripe: requests retry on failure, so Stripe deduplicates by idempotency key. Both directions accept duplicate attempts as inevitable and make duplicate effects impossible, each side keeping a record keyed by an identifier the sender controls.

It also closes a loop from Lesson 2. Stripe's docs list "no double charges" among the reasons PaymentIntents exist: one intent per cart is idempotency lifted to the object level. The header protects a single API call; the intent's design protects the whole checkout session, because re-confirming an already-succeeded intent has nowhere bad to go in the state machine.

Key line "An idempotency key turns a retry from 'do it again' into 'tell me what happened the first time.' Stripe stores the first response under the key for 24 hours and replays it verbatim, errors included."
The trap The shallow phrasing says keys "prevent duplicate requests." They prevent nothing; the duplicate request arrives and gets answered, with the recorded outcome instead of a re-execution. The deep answer knows the edges of the guarantee: stored errors replay as-is (a new attempt needs a new key), nothing is stored when validation rejects the request, and the record lives 24 hours, after which the same key silently becomes a fresh request.

Recall — answer out loud before peeking:

① Your POST creating a payment times out. What do you do, and what makes it safe?

show answer

Retry with the same idempotency key. If the first request executed, Stripe replays its stored response and nothing runs twice; if it never arrived, the retry simply executes first.

② What exactly does Stripe store under a key, and name a case where nothing is stored.

show answer

The first request's status code and response body, success or failure. Nothing is stored when execution never began: parameter-validation failures, or a conflict with a concurrently executing request.

③ The first request returned a 500 and it was stored. What does a same-key retry get, and what should a new attempt use?

show answer

The same stored 500, replayed. A deliberate new attempt at the operation needs a fresh key.

④ Why are idempotency keys pointless on GET and DELETE?

show answer

Those verbs are idempotent by definition: repeated reads and repeated deletes already have the same effect as one. Only POST creates or mutates in a way where twice differs from once.
