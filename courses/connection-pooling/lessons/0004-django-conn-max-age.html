Connection Pooling · Lesson 4

Django's CONN_MAX_AGE and the per-worker trap

Django's built-in connection feature isn't a pool, and that trips people up. Knowing exactly what it is, and what it can't do, is the heart of any Django pooling answer.

The concept

By default, Django opens a fresh database connection at the start of every request and closes it at the end, paying the full Lesson 1 handshake every single time. Its built-in remedy is persistent connections, controlled by one setting:

DATABASES["default"]["CONN_MAX_AGE"] = ...

  0      → close the connection at the end of each request   (default)
  N      → keep it open and reuse it for N seconds, then close
  None   → keep it open forever (unlimited)

A positive value (or None) tells a worker: don't close this connection, hold it and reuse it on the next request.1

Interview line "CONN_MAX_AGE is persistent connections, not pooling. It lets one worker reuse one long-lived connection across requests, so it removes the per-request handshake cost, but it does nothing to reduce the number of connections. There's no shared pool, no queue, no multiplexing."

That distinction is everything. A pool (Lesson 3) hands a small set of connections around among many requests. CONN_MAX_AGE just stops a worker from throwing its own connection away. To see why that matters, you have to look at how Gunicorn runs.

The steps

Follow one connection through a sync Gunicorn deployment.

1. Gunicorn forks W worker processes — each is a separate OS process
2. A request hits worker #3 → Django opens a connection (pays the handshake once)
3. Request ends → CONN_MAX_AGE > 0, not expired → worker KEEPS the connection open
4. Next request on worker #3 → reuse it (optional health check first)
5. Connection older than CONN_MAX_AGE → closed at request end, reopened next time
6. Meanwhile workers #1, #2, #4 … each hold their OWN connection, independently

Step 1 — Each worker is its own process

Gunicorn scales by forking W independent worker processes (this is the same fork from Lesson 1, one level up). Processes don't share memory, and therefore they don't share database connections. Whatever Django does with connections, it does separately inside each worker.2

Step 2 — First request opens a connection

Connections are lazy: a worker opens one only when it first needs the database, paying the Lesson 1 setup cost. A sync worker handles one request at a time, so it needs exactly one connection per database alias.

Step 3 — Request ends, connection stays

This is the whole point of CONN_MAX_AGE. With the default 0, Django closes here. With a positive value, it keeps the connection open and parks it on the worker for reuse.

Step 4 — Next request reuses it

The next request on that worker skips steps 1–7 of Lesson 1 entirely: the handshake is already done. If CONN_HEALTH_CHECKS = True, Django first pings the connection to confirm it's still alive (the same liveness idea as psycopg's check from Lesson 3); without it, a connection gone stale after a DB restart will raise on first use.1

Step 5 — Expiry

Once the connection's age passes CONN_MAX_AGE, Django closes it at the end of that request and opens a fresh one next time. None means this never happens: the connection lives as long as the worker does.

Step 6 — Every other worker does the same, independently

With W workers you have W independent connections. And with CONN_MAX_AGE = None, those are W connections held open permanently, whether or not any work is happening.

Why this is a trap

Persistent connections trade away setup cost, but they do not reduce connection count, and they can increase the pressure on Lesson 2's max_connections ceiling.

          ┌── worker 1 ──▶ conn ──┐
gunicorn ─┼── worker 2 ──▶ conn ──┼──▶ Postgres   total backends held
          ┼── worker 3 ──▶ conn ──┤                = W   (× threads, × app
          └── worker W ──▶ conn ──┘                  instances behind an LB)

No shared pool. Each worker pins its own backend. An idle worker with CONN_MAX_AGE=None still occupies a connection slot + ~5–10 MB on the server (Lesson 2) doing nothing.

Do the arithmetic that bites people:

  • Sync workers: total connections ≈ W workers.
  • Threaded workers (gthread): ≈ W × threads_per_worker: each thread runs a request, so each needs its own connection.
  • Multiple app instances behind a load balancer: multiply again by instance count.

So "I set CONN_MAX_AGE=None for performance" plus "we autoscaled to 12 instances × 8 workers" demands ~96 permanently-open backends. And Postgres defaults to max_connections = 100. That's the FATAL: sorry, too many clients already from Lesson 2, arriving in production.

The trap CONN_MAX_AGE removes handshake latency but pins a backend per worker. It is the right tool for cutting per-request overhead, and the wrong tool for surviving a high connection count. When you have lots of workers/instances, the fix is a real pool below all the workers, not a bigger CONN_MAX_AGE. That's Lesson 5.

What about Django 5.1's "pool": True?

Django 5.1+ can use psycopg3's real pool inside each worker via OPTIONS={"pool": True} (requires psycopg[pool]).1 That gives you genuine pooling, but it's still per worker process, so your total is W × pool_size: the exact footgun from Lesson 3. It helps a threaded worker share connections across its threads; it does not unify connections across workers or instances. (Also: with ASGI/async, disable CONN_MAX_AGE and rely on a pool instead.1)

Recall — answer out loud before peeking:

① Is CONN_MAX_AGE connection pooling? Explain in one sentence.

show answer

No. It's persistent connections: one worker reusing one long-lived connection across requests. It cuts handshake cost but doesn't share, queue, or reduce the number of connections.

② You run 10 Gunicorn sync workers with CONN_MAX_AGE=None. Roughly how many Postgres backends are held, and when?

show answer

About 10, one per worker, held open permanently even while idle, because workers don't share connections and None never expires them.

③ Why can a larger CONN_MAX_AGE make a max_connections problem worse?

show answer

It pins each worker's backend open longer (or forever), so idle workers keep occupying connection slots + RAM, pushing total held connections toward the ceiling instead of releasing them.
