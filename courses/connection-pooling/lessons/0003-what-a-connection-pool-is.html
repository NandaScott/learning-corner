Connection Pooling · Lesson 3

What a connection pool actually is

Lessons 1 and 2 built the problem: opening connections is slow, holding too many is fatal. This is the payoff: the one idea that fixes both at once.

The concept

A connection pool is a managed set of already-open database connections that an app keeps around and hands out on demand, instead of opening a fresh one per request and closing it after.

The shift in mental model is small but total: a connection stops being something you create and destroy, and becomes something you borrow and return. The pool opens its connections once, up front, and then those same backend processes are reused thousands of times.

  WITHOUT pool                       WITH pool
  ───────────                        ─────────
  request → open conn → query        request ─┐
          → close                             │   ┌──────────────┐
  request → open conn → query        request ─┼─▶ │  POOL        │ ─▶ Postgres
          → close                             │   │ [c1][c2][c3] │   (few real
  request → open conn → query        request ─┘   └──────────────┘    backends)
          → close
  (pay setup EVERY time)             (open once, borrow/return forever)

The pool sits between your app and Postgres, multiplexing many requests onto a small, fixed set of real connections.

The steps

Here's the full life of one request through a pool. We'll walk each step underneath.

1. Pool init        on startup, open N connections, leave them idle
2. Acquire          request asks the pool for a connection ("check out")
3a.  → idle one?    hand it over instantly
3b.  → all busy?    wait in a FIFO queue...
3c.  → timeout?     ...until a slot frees OR the wait limit expires → error
4. Use              run the query on the borrowed connection
5. Reset            roll back any open txn, clear session state
6. Release          return the connection to the pool ("check in"), still open

The connection is never closed at step 6: that's the whole point. It goes back in the pool, warm, for the next request.

Step 1 — Pool initialization

When the app starts, the pool opens a configured number of connections and parks them idle, ready to use. It pays the Lesson 1 handshake cost once, at boot, not on the request hot path.

Step 2 — Acquire (check out)

When a request needs the database, it doesn't connect to Postgres; it asks the pool for a connection. This is the "checkout." The request now holds that connection exclusively until it gives it back.1

Step 3 — Idle, queue, or timeout

What happens next depends on the pool's state:

  • A connection is idle → handed over instantly. The common, fast path.
  • All connections are busy → the request waits in a FIFO queue for one to be returned.1
  • The wait exceeds the checkout timeout → the request gives up with an error rather than hanging forever.

This queue is a feature, not a bug: it lets the app absorb a burst of requests without opening a burst of connections at the database.

Step 4 — Use

The request runs its SQL on the borrowed connection, exactly as it would on a freshly-opened one. No difference from the query's point of view.

Step 5 — Reset (or invalidate)

Before the connection goes back, the pool decides between two very different actions depending on the connection's health:

return connection
  ├─ connection broken?   → INVALIDATE: discard it, spawn a fresh one
  │     (never clean a corpse — a dead socket can't be reset)
  └─ connection healthy?  → RESET: clean it, return it to the pool
        ├─ open / aborted transaction?  → ROLLBACK
        └─ optional reset callback runs (must leave conn idle, else discarded)

Reset = "clean it for reuse." Invalidate = "throw it away." A failed query triggers reset; a failed connection triggers invalidation.

The reset matters because, without it, the next borrower could inherit a half-finished transaction, or its table and row locks, from a stranger.

The trap — psycopg3 defaults psycopg3's default reset is ROLLBACK only. It does not run DISCARD ALL, so leftover session state survives into the next borrower: SET variables, prepared statements, temp tables, LISTEN channels.4 You get transaction safety for free, but not session isolation. To get the latter, pass a reset callback: 
def full_reset(conn):
    conn.execute("DISCARD ALL")   # temp tables, prepared stmts, SET, LISTEN — all cleared

ConnectionPool(conninfo, reset=full_reset)

Detecting a broken connection happens in two ways. Optimistically (the default, free): an operation fails, the connection is flagged broken and discarded on return. Pessimistically: pass check=ConnectionPool.check_connection to run a liveness ping before handing a connection to a client, psycopg's equivalent of pool_pre_ping. It costs a network round trip per checkout, so it's off by default; turn it on if your database does failovers.4 (Lesson 5's "pooling modes" govern when the reset is even safe to do.)

Step 6 — Release (check in)

The connection is returned to the pool, still open. No fork, no auth, no teardown. It sits idle until the next request checks it out, and the cycle repeats.

Why this works

One mechanism, both problems from Lessons 1 and 2 solved:

ProblemHow the pool kills it
Setup is slow (L1: ~20–100 ms fork + auth per connection)Connections are opened once at startup and reused. The handshake leaves the request hot path entirely.
Too many backends (L2: the max_connections ceiling + throughput curve)The pool caps connections at its configured size. 10,000 incoming requests still map onto, say, 20 real backends: they queue instead of overwhelming Postgres.

That second point is the subtle one. The pool decouples request concurrency (how many users are hitting you) from connection concurrency (how many backends Postgres must run). You keep Postgres parked on the good part of Lesson 2's throughput curve no matter how spiky your traffic gets.

Interview line "A pool decouples request concurrency from connection concurrency. Requests borrow and return a small fixed set of pre-warmed connections, so you skip the per-request handshake and you never blow past max_connections: extra load queues instead of forking more backends."

How big should the pool be?

Bigger is not better. That's straight from Lesson 2. The pool size should aim for the database's throughput sweet spot, not your request count. A common starting rule of thumb:2

pool_size ≈ (cpu_cores × 2) + effective_spindles

A starting point, not a law: for an SSD-backed 4-core DB that's ~8–10. Tune from there. The instinct to "set it to 500" recreates the exact Lesson 2 problem the pool was meant to prevent.

The trap Your total load on Postgres is pool_size × number of app processes, not just pool_size. Twenty Gunicorn workers each with a pool of 20 = 400 connections at the database. This is the single biggest pooling footgun in a multi-process Django deploy. And it's exactly what Lesson 4 tackles.

Recall — answer out loud before peeking:

① What changes about a connection's "lifecycle" once you introduce a pool?

show answer

It stops being create-then-destroy and becomes borrow-then-return. Connections are opened once at startup and reused; check-in returns them open, not closed.

② A pool decouples two kinds of concurrency. Name them.

show answer

Request concurrency (how many users hit the app) from connection concurrency (how many real backends Postgres runs). Excess requests queue instead of opening more connections.

③ Why does the pool reset a connection before checking it back in, and what does psycopg3 reset by default?

show answer

So the next borrower doesn't inherit a stranger's dirty connection. psycopg3's default reset is ROLLBACK only (clears the transaction + its locks); session state like SET/temp tables/prepared statements survives unless you add a DISCARD ALL reset callback.

④ Connection comes back dead (DB restarted). Reset or invalidate, and what's the difference?

show answer

Invalidate: a broken connection is discarded and replaced, never reset. Reset cleans a healthy connection for reuse; invalidate throws a dead one away. A failed query → reset; a failed connection → invalidate.
