Connection Pooling · Lesson 5

PgBouncer and pooling modes

Lesson 4 left you stuck: every worker pins its own backend, and no Django setting unifies them. The fix is to move the pool out of the app entirely, and the cost of doing so is the most-tested topic in the subject.

The concept

PgBouncer is a connection pool that lives as a separate process between your app and Postgres. It's tiny and single-threaded (event-driven, no fork-per-connection), so it can accept thousands of cheap client connections while holding only a small pool of real Postgres backends, and hand those few backends around among all the clients.1

The thing that breaks Lesson 4's trap: PgBouncer sits below every worker and every app instance at once. There's one shared pool for the whole fleet, not one per worker. So Postgres's backend count stops depending on how many workers or instances you run.

  LESSON 4 (per-worker)                 LESSON 5 (pool below the fleet)
  ─────────────────────                 ──────────────────────────────
  worker 1 ─▶ conn ─┐                    worker 1 ─┐
  worker 2 ─▶ conn ─┼─▶ Postgres         worker 2 ─┤        ┌──────────┐
  worker 3 ─▶ conn ─┤   (W backends,     worker 3 ─┼──────▶ │ PgBouncer│─▶ Postgres
  worker W ─▶ conn ─┘    grows with W)   worker W ─┘        └──────────┘   (~20 backends,
                                         (1000s of cheap client conns)      FIXED)

Many cheap client connections in; a small, fixed set of expensive server connections out.

Interview line "PgBouncer is a pool below all the app processes, so Postgres's backend count is decoupled from worker and instance count entirely. A thousand client connections across the fleet ride on maybe twenty real backends."

The steps

How one request rides through PgBouncer. Notice that step 5, when the backend is returned, is what matters most.

1. App workers connect to PgBouncer (cheap — no fork, no per-conn backend)
2. A request begins its database work
3. PgBouncer assigns a real Postgres backend from its small pool
4. Queries run on that borrowed backend
5. PgBouncer returns the backend to the pool  ← timing depends on the MODE
6. Postgres only ever sees pool_size backends, no matter how many clients

Step 1 — Workers connect to PgBouncer

The app's connections now point at PgBouncer, not Postgres. A client connection to PgBouncer is cheap: PgBouncer isn't Postgres, so there's no fork and no backend process behind each one (Lesson 1). It can hold thousands of idle client connections happily, which is exactly why every worker, thread, and instance keeping a connection open is no longer a problem.

Step 2 — A request begins its database work

Holding a client connection to PgBouncer ties up nothing on Postgres by itself. A real backend isn't involved until the worker actually starts running SQL, so an idle worker (Lesson 4's pinned-backend villain) now costs a Postgres backend exactly zero.

Step 3 — Assigning a backend

When a client needs the database, PgBouncer lends it one of its pooled Postgres backends. If all are busy, the client waits. Same queue idea as Lesson 3, just now shared across the entire fleet instead of trapped inside one worker.

Step 4 — Queries run on the borrowed backend

PgBouncer speaks the Postgres wire protocol and is transparent: the app runs its queries exactly as if connected directly. From the query's point of view there's no pooler in the path at all, which is also why the breakages in Step 5 are so easy to miss until they bite.

Step 5 — Returning the backend: the pooling mode

The single most important PgBouncer setting is pool_mode: it decides when a borrowed backend goes back to the pool. More aggressive return = fewer backends needed = better multiplexing, but it breaks more of Postgres's session-scoped features.1

sessionwhen the client disconnectsnothing: full Postgres compatibilityyou need every session feature; least multiplexing
transactionat each COMMIT/ROLLBACKsession-scoped state (see below)the default choice for web apps, with the best multiplexing

What transaction mode breaks — and why

In transaction mode, two consecutive transactions from the same client may land on different backends. So anything that lives in the session rather than the transaction silently breaks:1

  • Session-level prepared statements: prepared on one backend, gone on the next.
  • SET / RESET session variables: don't persist across transactions.
  • LISTEN / NOTIFY: the channel is bound to a specific backend.
  • Session-scoped advisory locks: held by a backend you may not get back.
  • Server-side cursors (incl. WITH HOLD): local to a backend; a later fetch may hit a different one.

Step 6 — Postgres only ever sees pool_size backends

This is the payoff. No matter how many client connections arrive, thousands across the fleet, Postgres only runs the small fixed number of backends in PgBouncer's server-side pool (default_pool_size). Backend count is now a knob you set to land on Lesson 2's throughput sweet spot, not a number that drifts upward with every worker, thread, and instance you add.

Why it works — and the Django wiring

This is the structural fix to Lesson 4. PgBouncer decouples Postgres backend count from app process count globally: your 96-worker fleet from Lesson 4 now rides on a fixed ~20 backends, comfortably under max_connections and parked on Lesson 2's good part of the throughput curve.

But to run Django in transaction mode (the one you actually want), you have to stop relying on the session features it breaks:

# settings.py — Django behind PgBouncer in transaction mode
DATABASES["default"].update({
    "CONN_MAX_AGE": 0,                  # let PgBouncer own persistence — don't double-pool
    "DISABLE_SERVER_SIDE_CURSORS": True, # .iterator() would otherwise break (Django #28062)
    # psycopg3: also disable server-side prepared statements
    "OPTIONS": {"prepare_threshold": None},
})

DISABLE_SERVER_SIDE_CURSORS=True is the famous one: Django 1.11 made QuerySet.iterator() use server-side cursors, which transaction pooling can't keep alive.2

The trap Don't double-pool. If PgBouncer owns persistence, set Django's CONN_MAX_AGE=0 (or use Django's native pool or PgBouncer, not both layers fighting). And never point an app that uses LISTEN/NOTIFY, session advisory locks, or long server-side cursors at a transaction-mode pooler without knowing those features are now broken.
The whole course in one breath A Postgres connection is a forked backend process (L1) → so connections are RAM/CPU-bound with a hard ceiling and a throughput curve that bends down (L2) → a pool reuses a small fixed set instead of forking per request (L3) → but Django's CONN_MAX_AGE is per-worker persistence, not a shared pool, so it pins a backend per worker (L4) → PgBouncer is the shared pool below the whole fleet, and transaction mode multiplexes hardest at the cost of session-scoped features (L5).

Recall — answer out loud before peeking:

① Why does PgBouncer fix Lesson 4's per-worker trap when CONN_MAX_AGE can't?

show answer

It's one shared pool below all workers and instances, so Postgres backend count is decoupled from app process count. CONN_MAX_AGE is per-worker, so it scales with worker count.

② In transaction mode, name three things that break and the one reason behind all of them.

show answer

Prepared statements, SET vars, LISTEN/NOTIFY, advisory locks, server-side cursors. All because consecutive transactions can land on different backends, so session-scoped state doesn't persist.

③ Which two Django settings let it run safely in transaction mode, and why?

show answer

DISABLE_SERVER_SIDE_CURSORS=True (server-side cursors can't survive across transactions) and CONN_MAX_AGE=0 (let PgBouncer own persistence instead of double-pooling).
