Connection Pooling · Lesson 1

The connection handshake, step by step

Before you can talk about pooling, you have to know what physically happens when an app connects to PostgreSQL. So we open the box first — every later lesson builds on this sequence.

The concept

"Connecting to the database" sounds like one instant action. It isn't. It's a multi-step negotiation between your app and Postgres — and the key fact underneath all of it:

PostgreSQL serves every client connection with its own dedicated operating-system process. Not a thread, not a lightweight handle: a full backend process with its own private memory, forked by a supervisor process called the postmaster when the connection opens, and living until it closes.1

That single design choice is the root cause of every connection problem you'll ever debug. The steps below are the play-by-play of how that process is born — and each one has a cost.

The steps

With no pool, a naïve app runs this entire sequence for every request, then throws the backend away and starts over on the next one. Here's the map; we'll walk each step underneath.

1. TCP handshake       client ↔ postmaster   SYN / SYN-ACK / ACK  (1 round trip)
2. (TLS handshake)     if sslmode on         encryption negotiated
3. Startup packet      client → postmaster   "proto v3, user=X, db=Y"
4. postmaster FORKS    →  new backend process   ← the expensive bit
5. Authentication      backend ↔ client      challenge/response
6. Backend setup       load role, session GUCs, warm catalog caches
7. ReadyForQuery       backend → client      ...only now can you run SQL
8. ── run the query ──
9. Close               backend process dies, its ~5–10 MB freed

Steps 1–7 are pure overhead — paid before a single byte of your query is processed.

Step 1 — TCP handshake

Your app first opens a plain network socket to Postgres (default port 5432) using TCP's three-way handshake: SYN → SYN-ACK → ACK. That's one full network round trip before Postgres has even heard what you want. On a LAN it's sub-millisecond; across cloud availability zones it can be several milliseconds.

Step 2 — TLS handshake (only if encrypted)

If your connection requires SSL/TLS (sslmode=require and friends), client and server now negotiate encryption: agree on a cipher, exchange keys, verify certificates. This adds more round trips plus asymmetric cryptography (CPU-heavy public-key math). Skipped entirely on unencrypted connections — but most production setups encrypt.

Step 3 — Startup packet

The client sends its first Postgres-protocol message: the startup packet, declaring the protocol version, the target user, and the database. This is how the postmaster learns who is connecting and to what — everything before this was just plumbing.2

Step 4 — The fork (the expensive bit)

Now the postmaster forks a brand-new backend process to handle this one connection. This is the costliest step by far — and the part people miss when they imagine a connection is "just a socket."

          ┌───────────────┐
client ──▶ │  postmaster   │  (listens on 5432, accepts the socket)
          └───────┬───────┘
                  │ fork()           ← clones a whole process
                  ▼
          ┌───────────────┐
          │ backend proc  │  new PID · own page tables · own file
          │  (~5–10 MB)   │  descriptors · private memory segment
          └───────────────┘  serves ONLY this connection, until it closes

fork() doesn't open a handle — it clones an entire OS process. That's a heavy system call, and it's why a Postgres connection costs orders of magnitude more than a Redis one.

Step 5 — Authentication

The freshly-forked backend consults pg_hba.conf (host-based auth rules) to pick a method for this user + database + client IP, then runs it. The modern default is SCRAM-SHA-256: a challenge/response exchange where the password never crosses the wire.2

backend → client   AuthenticationSASL          "I support SCRAM-SHA-256"
client  → backend  SASLInitialResponse         client-first-message (client nonce)
backend → client   AuthenticationSASLContinue  server-first-message (salt + nonce)
client  → backend  SASLResponse                client-final-message (hashed proof)
backend → client   AuthenticationSASLFinal     verifies, sends OK

Four messages = several more network round trips. And SCRAM runs PBKDF2 — thousands of hash iterations, made deliberately slow to resist brute-force — burning real CPU on both ends.

Older methods trade security for speed: md5 (fewer trips, weaker), trust (no auth — never in production), certificate-based (crypto cost). The point: auth is not a flag-check; it's a repeated CPU + network cost.

Step 6 — Backend setup

With identity confirmed, the backend initializes its session: loads the role and its permissions, applies session settings (the per-connection configuration knobs Postgres calls GUCs), and warms its private catalog caches (the metadata about tables, columns, types it'll need to plan queries). This is also where the ~5–10 MB of per-connection memory gets committed.

Step 7 — ReadyForQuery

The backend sends a ReadyForQuery message — "I'm idle and waiting." Only now, after steps 1–6, can your app actually run SQL. Every one of those steps was latency you paid before any useful work began.

Steps 8–9 — Query, then close

You run your query. Then, in the no-pool world, the connection closes: the backend process exits and its memory is freed. The next request starts again at step 1 — paying the whole bill a second time.

Why this is costly

Two distinct costs fall out of the sequence above.

1. Per request, the setup is expensive. Steps 1–7 — a TCP round trip, possibly TLS, a process fork, a multi-message PBKDF2 auth, and session warm-up — add up to roughly 20–100 ms before your query runs.1 A no-pool app pays that on every request, then discards the warmed-up backend it just built.

2. At scale, many backends fight each other. One connection is one process, so a thousand connections is a thousand processes — and a CPU core runs only one at a time. The OS must rapidly swap which backend is on a core, a context switch, and each swap does real work for zero useful output:

What the switch doesWhy it hurts
Save + restore CPU stateRegisters, stack pointer, program counter dumped to RAM and reloaded — overhead, no query progress
CPU caches go coldThe new backend's data isn't in L1/L2/L3, so it stalls waiting on slow main memory ("cache pollution")
TLB flushThe virtual→physical address cache is invalidated on a process switch → more memory stalls
Branch predictor resetThe CPU's "guess the next instruction" history is useless for the new process

Add one more effect — every transaction start/end grabs the shared ProcArrayLock, so more backends means more contention on a single lock3 — and you have the makings of a hard ceiling. Lesson 2 turns this into the actual numbers and the throughput curve that bends downward.

Interview line "Without pooling you pay a TCP round trip, a process fork, and a multi-message auth handshake on every single request — then discard the warmed backend. A pool keeps backends past step 7 and just hands them out, so that whole 20–100 ms disappears."
The trap "Context switching is slow" is the shallow answer. The deep answer names what gets thrown away: hot CPU caches and the TLB. The switch itself is cheap; the cache misses it causes are what actually cost you. That distinction is the tell of someone who understands the hardware.

Recall — answer out loud before peeking:

① What is a Postgres connection, physically — and why does that make it expensive?

show answer

A dedicated OS backend process, forked by the postmaster, with ~5–10 MB of private memory. So connections are bounded by RAM and CPU, not by how many sockets you can open.

② In the no-pool sequence, which single step is the most expensive, and why?

show answer

The fork (step 4). The OS clones a whole process — new page tables, file descriptors, private memory — not just a socket. It's a heavy syscall.

③ Why does SCRAM-SHA-256 auth take real time, not just a flag check?

show answer

It's a multi-message challenge/response (several network round trips) and runs PBKDF2 — thousands of deliberately-slow hash iterations — costing CPU on both ends.

④ Context switching is slow — but what specifically does the switch destroy that costs you the most?

show answer

The hot CPU caches (L1/L2/L3) and the TLB. The reload itself is cheap; the cache/TLB misses afterward — stalling on slow RAM — are the real cost.
