Connection Pooling · Reference

Decision cheat-sheet

The whole course compressed to what you'd glance at before an interview or a config change: which layer, which mode, which settings. One page. Print it.

1 · Which pooling layer?

Are you on sync WSGI (Gunicorn/uWSGI)?
│
├─ Few workers, few instances, connection count comfortably under max_connections?
│     → CONN_MAX_AGE = N (e.g. 60).  Cuts handshake cost. Done.
│
├─ Many workers/threads in ONE process group, want a real pool but no extra infra?
│     → Django 5.1+  OPTIONS={"pool": True}.  Real pool, but still PER WORKER.
│
└─ Many workers AND/OR many instances → connection count approaching the ceiling?
      → PgBouncer (transaction mode) below the whole fleet.  The structural fix.

On ASGI/async? → disable CONN_MAX_AGE; use a pool (native or PgBouncer).

Rule of thumb: CONN_MAX_AGE fixes latency; PgBouncer fixes count. Reach for PgBouncer when workers × threads × instances threatens max_connections.

3 · PgBouncer mode picker

ModeBackend returned…BreaksPick when
sessionon client disconnectnothing (full compat)you need session features; least multiplexing
transactionat COMMIT/ROLLBACKsession-scoped state ↓default for web apps
statementafter each statementmulti-statement txnsautocommit-only / sharding

Transaction mode breaks (all because consecutive txns may land on different backends): session-level prepared statements · SET/RESET · LISTEN/NOTIFY · session-scoped advisory locks · server-side cursors (WITH HOLD).

4 · Django config recipes

Persistent connections, no pooler

DATABASES["default"]["CONN_MAX_AGE"] = 60        # seconds; None = forever
DATABASES["default"]["CONN_HEALTH_CHECKS"] = True # ping on reuse (survive failovers)

Django 5.1+ native pool (per worker)

DATABASES["default"]["OPTIONS"] = {"pool": True}  # needs psycopg[pool]
DATABASES["default"]["CONN_MAX_AGE"] = 0          # don't double-pool

Behind PgBouncer in transaction mode

DATABASES["default"].update({
    "CONN_MAX_AGE": 0,                    # PgBouncer owns persistence
    "DISABLE_SERVER_SIDE_CURSORS": True,  # .iterator() can't survive txn pooling
    "OPTIONS": {"prepare_threshold": None}, # psycopg3: no server-side prepared stmts
})
Footguns Don't double-pool (PgBouncer + a big CONN_MAX_AGE or native pool fighting). · CONN_MAX_AGE=None + autoscaling = pinned idle backends → ceiling. · Transaction-mode pooler + LISTEN/NOTIFY or session advisory locks = silent breakage. · Oversized pool recreates the Lesson 2 throughput problem.

6 · The arc in one line

Forked backend process (L1) → RAM/CPU ceiling & downward throughput curve (L2) → reuse a small fixed set (L3) → CONN_MAX_AGE is per-worker, not shared (L4) → PgBouncer is the shared pool below the fleet; transaction mode trades session features for multiplexing (L5).
