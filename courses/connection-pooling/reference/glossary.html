Connection Pooling · Reference

Glossary

The shared vocabulary for this course. Every lesson uses these terms exactly as defined here. Print it; keep it next to you.

TermDefinition
Backend processThe dedicated OS process Postgres forks to serve one client connection. Holds ~5–10 MB of private memory. Lives until the connection closes. (Lesson 1)
postmasterThe supervisor Postgres process that listens for new connections and forks a backend process for each one. (Lesson 1)
forkThe OS system call that clones a process. The postmaster forks a new backend per connection — new PID, page tables, file descriptors, private memory. The single most expensive step of connection setup. (Lesson 1)
Startup packetThe first message a client sends after the TCP handshake, declaring protocol version, user, and database. Triggers the backend fork + auth. (Lesson 1)
SCRAM-SHA-256Postgres's modern default auth method. A 4-message challenge/response (password never sent) using PBKDF2 hashing. Costs several network round trips + CPU. (Lesson 1)
pg_hba.confHost-based authentication config. Rules mapping (user, database, client IP) → auth method. The backend consults it during setup. (Lesson 1)
Context switchWhen a CPU core swaps which process it runs. Cheap itself, but trashes hot CPU caches + the TLB, causing memory stalls. Why hundreds of backends drag throughput down. (Lesson 1)
TLBTranslation Lookaside Buffer — the CPU's cache of virtual→physical address mappings. Flushed on a process switch, causing memory-access stalls afterward. (Lesson 1)
Connection setup costThe ~20–100 ms to fork the backend, allocate memory, authenticate, and establish session state. Paid every time a connection is opened. (Lesson 2)
max_connectionsHard cap on simultaneous backend processes. Default 100. Exceeding it returns FATAL: sorry, too many clients already. (Lesson 2)
Connection poolA managed set of pre-opened connections that are handed out, used, and returned — instead of opening a new one per request. (Lesson 3)
Check out / check inBorrowing a connection from the pool (check out / acquire) and returning it still-open when done (check in / release). The core pool operations. (Lesson 3)
Connection resetThe cleanup a pool runs at check-in — roll back any open transaction, clear temporary session settings — so the next borrower gets a clean connection. (Lesson 3)
Checkout timeoutHow long an acquiring request waits in the pool's FIFO queue for a free connection before giving up with an error. (Lesson 3)
Pool exhaustionWhen every pooled connection is checked out and requests pile up in the queue (and may time out). The classic symptom of an undersized pool or leaked connections. (Lesson 3)
Invalidate (vs reset)A pool's two return paths: reset cleans a healthy connection for reuse; invalidate discards a broken one and spawns a replacement. Failed query → reset; failed connection → invalidate. (Lesson 3)
DISCARD ALLPostgres command that fully resets a session — drops temp tables, deallocates prepared statements, resets SET params, unlistens. Not run by psycopg3's default reset (which is ROLLBACK-only); add it via a reset callback for session isolation. (Lesson 3)
Liveness check (check / pre-ping)Pessimistic test that a pooled connection is alive before serving it to a client (psycopg3: check=ConnectionPool.check_connection). Costs a round trip; off by default; worth it across DB failovers. (Lesson 3)
Persistent connectionA connection a process keeps open and reuses across requests rather than closing it each time. Not a pool — no sharing, queueing, or multiplexing. Django's mechanism is CONN_MAX_AGE. (Lesson 4)
CONN_MAX_AGEDjango setting controlling how long a worker reuses a DB connection. 0 = new connection per request (default); N = reuse for N seconds; None = reuse forever. Cuts handshake cost but does not reduce connection count. (Lesson 4)
CONN_HEALTH_CHECKSDjango setting (default False). When True, Django pings a reused persistent connection at the start of a request and reconnects if it's dead — guarding against stale connections after a DB restart/failover. (Lesson 4)
OPTIONS={"pool": True}Django 5.1+ option enabling a real psycopg3 pool inside each worker (needs psycopg[pool]). Genuine pooling, but still per-worker → total = workers × pool_size. (Lesson 4)
WSGI workerOne process of a sync app server (Gunicorn/uWSGI). Each worker is a separate process holding its own Django connections — they are not shared between workers. (Lesson 4)
PgBouncerA lightweight, single-process external pooler between app and Postgres. Accepts thousands of cheap client connections, multiplexes them onto a small fixed set of real backends. Lives below the whole fleet, so backend count is decoupled from worker/instance count. (Lesson 5)
MultiplexingMapping many client connections onto few server connections by lending each backend to whichever client needs it right now. The core of what an external pooler does. (Lesson 5)
Pooling modeWhen a pooler returns a borrowed backend, set by pool_mode. More aggressive return = more multiplexing but breaks more session-scoped features. (Lesson 5)
Session poolingBackend returned when the client disconnects. Full Postgres compatibility, least multiplexing. (Lesson 5)
Transaction poolingBackend returned at each COMMIT/ROLLBACK. Default for web apps — best multiplexing — but breaks session-scoped state (prepared statements, SET, LISTEN, advisory locks, server-side cursors) because consecutive transactions may land on different backends. (Lesson 5)
Statement poolingBackend returned after every statement; multi-statement transactions forbidden. Autocommit-only setups. (Lesson 5)
DISABLE_SERVER_SIDE_CURSORSDjango setting required to run in transaction pooling — QuerySet.iterator() uses server-side cursors that can't survive across transactions on a shared pooler. Set True. (Lesson 5)
Thundering herdWhen many app processes all try to reconnect at once (e.g. after a DB restart), flooding the postmaster with fork requests and slowing recovery.
